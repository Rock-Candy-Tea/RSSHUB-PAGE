data-v-408909ad referrerpolicy="no-referrer"><p><span class="text-remarks" label="备注">本文来自微信公众号：</span><a href="https://mp.weixin.qq.com/s/1JsKfB6Dr3ILpFNolXeTAw" target="_blank" rel="nofollow" style="text-decoration: none;"><span class="text-remarks">果壳（ID：Guokr42）</span></a><span class="text-remarks">，作者：木遥 & 银古桑，题图来自：视觉中国</span></p><p>2021年4月5日，美国最高法院发布了可能是科技业近年来最重要的判决之一。该判决针对“甲骨文诉谷歌”一案，推翻了此前联邦巡回法院的判决，认定谷歌的安卓系统对 Java API 的重用是合理的。 </p><p><strong>这案子缠讼十年，标的88亿美元，历经三次判决反转。</strong></p><p>起因非常简单：Java 是一种在程序员中有非常高人气的编程语言，掌握在 甲骨文手里，谷歌在推出安卓系统时，为了能让更多给安卓写第三方 APP 的程序员尽快上手，直接在安卓 APP 开发工具里复用了大量 Java 的函数接口<span class="text-remarks" label="备注">（API）</span>，但自己重新实现了函数本身。甲骨文据此告谷歌侵权。</p><p>滑动查看谷歌和甲骨文十年诉讼史：</p><ul class=" list-paddingleft-2" style="list-style-type: disc;"><li><p>2010 年，甲骨文起诉谷歌侵犯了 7 件与 Java 相关的版权，要求谷歌赔偿约数十亿美元的损失。</p></li><li><p>2012 年 5 月，美国旧金山联邦法院<span class="text-remarks" label="备注">（或称加州北区法院）</span>的法官裁定，Java API 不受版权保护，任何人都可以免费使用；同年10 月，甲骨文上诉。</p></li><li><p>2014 年，美国联邦巡回上诉法院推翻了一审部分结论，称必须尊重软件的版权保护。</p></li><li><p>谷歌上诉，2015 年 6 月，美国最高法院拒绝就受理谷歌上诉。这起诉讼重返旧金山联邦法院，由该院就谷歌另外提出的“合理使用”的观点进行庭审。</p></li><li><p>2016 年 5 月，旧金山联邦法院复审，判决谷歌公司的行为合理，免付版权赔偿。</p></li><li><p>甲骨文上诉，2018 年 3 月，上诉法院再次判决甲骨文胜诉，并判决谷歌赔偿 88 亿美元</p></li><li><p>2019 年 11 月，在 78 名计算机科学家的陈情下，美国高院受理了谷歌的上诉，将对此前裁决复审。</p></li></ul><h3 label="大标题" class="text-big-title">“编程语言的接口”，是否应当受到版权保护？</h3><p>这里问题的核心是：<strong>一种编程语言的接口是否受到版权保护？对它的复用是否侵权？</strong></p><p>甲骨文的论点非常直接<span class="text-remarks" label="备注">（而且对非业内人士来说其实很有说服力）</span>——</p><blockquote><p>软件是否受到版权保护？当然。</p><p>接口是不是软件的重要组成部分？当然。</p><p>那么接口显然应该受到版权保护。</p></blockquote><p><strong>谷歌的论点就有点复杂，它需要详细辨析“接口”的含义</strong>。对大多数法律界人士和公众来说，API 这个词本身就很陌生。最高法院判决的主笔是82岁的 Breyer 法官，他这辈子很可能一行代码都没写过，但从判决看，他是精确理解 API 的功能和内涵的。</p><p>被判决采纳的论点是：<strong>API是一个发送指令的界面，像是汽车的加油踏板，或者电脑的 QWERTY 键盘。</strong></p><p>这两个例子不是随便举的，因为它们都正好反映出这个案子的实质：谷歌是利用了现成的 Java 接口以吸引程序员能够迅速上手。</p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193603992435.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" data-w="1000" data-h="669" label="图片备注" src="https://img.huxiucdn.com/article/content/202104/06/193603992435.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" referrerpolicy="no-referrer"></p><p label="图片备注" class="text-img-note">汽车的加油踏板 | Pixabay<br></p><p>“利用前人现成的知识节省学习成本”，那么，“前人的知识”是否应该受到保护？</p><p>汽车的加油踏板就是这样一个类似的情况。第一个设计汽车的人已经把加油踏板设计成这样了，如果这种设计本身受到版权保护，那么，每个后来的新造车厂就都会面临一种两难：要么继承这种设计但需要支付高昂的版权费用，要么另起炉灶但不会有用户买它的车，因为没人愿意买辆新车还要形成一套新的肌肉记忆。</p><p>回到API的情况。</p><p>判决指出：它的价值很大程度上体现为程序员群体对这种 API的熟练掌握，以及复用这个API所能导致的学习成本节省。因此，这个判决的核心就是宣布：这种搭便车的做法属于合理应用<span class="text-remarks" label="备注">（fair use）</span>，不应该被惩罚。</p><p>其中最核心的<span class="text-remarks" label="备注">（也是大部分评论最关注的）</span>是这样两段论述：</p><blockquote><p>“我们必须考量的是：对版权的保护是否促进了公众利益，是否促进了创新。”</p><p>“考虑到程序员在学习 Java API 上的投资，如果把这个接口本身保护起来，会有害公众利益，因为这会迫使程序员不得不付出额外的努力去适应新的接口。新的创造就会被锁起来，而钥匙掌握在甲骨文一家手里。这能让甲骨文获得不菲的利润，但这些利润本来可以流向大量掌握了这些接口的人能创造出的新的应用之中。因此这种锁定是和版权的本意相违背的。”</p></blockquote><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193605786524.png?imageView2/2/w/1000/format/png/interlace/1/q/85" data-w="615" data-h="239" src="https://img.huxiucdn.com/article/content/202104/06/193605786524.png?imageView2/2/w/1000/format/png/interlace/1/q/85" referrerpolicy="no-referrer"></p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193607818767.png?imageView2/2/w/1000/format/png/interlace/1/q/85" data-w="623" data-h="482" label="图片备注" src="https://img.huxiucdn.com/article/content/202104/06/193607818767.png?imageView2/2/w/1000/format/png/interlace/1/q/85" referrerpolicy="no-referrer"></p><p label="图片备注" class="text-img-note">GOOGLE LLC v. ORACLE AMERICA, INC<br></p><p>可以想像，这些论述<span class="text-remarks" label="备注">（特别是关于公众利益的部分）</span>的影响会非常深远。</p><h3 label="大标题" class="text-big-title">为了让法官明白何为“API接口”，双方律师掀起了比喻大战</h3><p>在这个具体的例子里，判决是和业界的 common sense 站在一起的。拷贝 Java 的接口<span class="text-remarks" label="备注">（只占 Java 总代码量的极小比例）</span>和拷贝 Java 的具体功能实现是两码事，不可同日而语。</p><p>但这里没有触及的问题是，<strong>对一个系统而言，设计接口并不是一项无足轻重的工作</strong>。从某种意义上来说，一个平台的接口和它背后的实现同样重要。接口有点像程序员世界里的“用户界面”，一个好的接口可以决定性地让一个平台取得优势。在某些极端的情况下，一个平台的价值可以主要就体现在接口上。</p><p>在这场旷日持久的诉讼中，为了让法官搞明白API接口是怎么回事，双方的律师都使出浑身解数，搜肠刮肚地用各种比喻。这在法律上叫做“类推”。当然，什么是合适的类推，只有法官能决定。</p><p>这是一场精彩的比喻大战，尤其是在很多技术专业到普通人无法靠常识理解的时代。开始他们用哈利·波特来类比，然后用电视剧《宋飞正传》，然后是开保险箱，还有打字键盘、足球比赛、餐馆菜单……</p><p>简单举两个例子——</p><p>甲骨文搬出了《哈利·波特》：就拿《哈利·波特》来说，要是有人用这本书的章节名、人物名，还有其他关键部分组合起来，写了一本自己的书然后出售？这是完全抄袭，获取商业利益，影响原作品的市场价值，显然是不公平的。</p><p>谷歌回应，用书本身的结构来比喻才更合适。一本书有书脊，按照页码顺序排列，这是所有书固有的结构和阅读形式，不是《哈利·波特》独有的。</p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193609977661.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" data-w="1000" data-h="1500" label="图片备注" src="https://img.huxiucdn.com/article/content/202104/06/193609977661.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" referrerpolicy="no-referrer"></p><p label="图片备注" class="text-img-note">《哈利波特》 | Pixabay<br></p><p>在这场关于API接口的著作权纠纷中，甲骨文需要找到合适的比喻说明自己的代码多么具有原创性、多么独特，而谷歌则需要说明这个接口多么常规、多么普遍、所有人都在用，如果这都不是合理使用那这个行业就没有合理使用了。</p><p><strong>需要大量使用比喻的另外一个原因是，美国采用普通法<span class="text-remarks" label="备注">（common law）</span>，也经常被称为判例法。</strong>法官们会参考先例，根据案件的具体情况判断，区别于成文法国家按照成文法典、法律法规来判决。在判例法国家，律师们描述案件时要详细说明这个案例的法律事实和法律适用跟先例的相似或者不同，让法官做出对自己有利的判决。在这种情况下，会类比、会讲故事、讲究细节就成了法庭辩论的重要因素，尤其是在法律事实很难在日常生活中找到对应物的时候。不管是用书、保险箱、键盘还是电视剧来打比方，都是为了方便法官从对自己有利的角度去理解。</p><h3 label="大标题" class="text-big-title">比喻有用，也有局限</h3><p><strong>法律依赖于比喻，而比喻永远是不精确的。</strong></p><p>如果把接口和实现的关系，想像成一台巨大的机器外壳上的几个插头和内部丰富的实际功能组件，谷歌的做法就相当于为了兼容性照搬外壳上几个插头的设计而内部完全自己另起炉灶。</p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193611723885.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" data-w="1000" data-h="575" label="图片备注" src="https://img.huxiucdn.com/article/content/202104/06/193611723885.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" referrerpolicy="no-referrer"></p><p label="图片备注" class="text-img-note">接口和实现的关系可以想象为一台巨大的机器外壳上的几个插头和内部丰富的实际功能组件 | www.hull.ac.uk</p><p>但现代软件工程并不是简单的机器，你很难清晰拆分出外壳和内部。大多数情况下，你看到的是一层层功能的封装，大量的智慧都投入在巧妙的封装本身上，而最底层的实现很可能只是继承经典而已。</p><p>问题在于，你如何能把现代软件工程比喻成一个现实生活中的例子呢？</p><p><strong>区块链是一个极端的情形。</strong></p><p>2016年，联合广场投资的分析师 Joel Monegro 写了一篇广为流传的文章：fat-protocols。他指出，和传统网络领域里协议很轻而应用很重的情形相反，在区块链的世界里，协议是“胖的”（fat），而应用无足轻重。投资于应用远不如投资协议回报丰厚。例如，以太坊是一个伟大的协议，它的价值远远胜过在运行在以太坊上的具体应用本身。</p><p><strong>协议<span class="text-remarks" label="备注">（protocol）</span>当然和接口<span class="text-remarks" label="备注">（API）</span>不完全是同一个概念，但它们是强烈相关联的。</strong>复制以太坊的全部接口差不多就相当于复制了以太坊本身。这件事又应该如何被比喻到现实世界之中呢？</p><p>为什么说是“合理使用”？</p><p>当然，这并不意味着今天的判决表示你可以直接复制一整个以太坊。在谷歌的案例里还有许多别的因素需要考量，其中很重要的一点是“转换性使用”<span class="text-remarks" label="备注">（transformative use）</span>，也就是说，谷歌并不是打算创造一个 Java 的等价竞品出来，安卓和 Java 是两个不同领域的东西。</p><p>但这个判决毕竟在比喻的边界处划了一条明确的界限。从今天业界的反应来看，这个界限得到了几乎一面倒<span class="text-remarks" label="备注">（除了甲骨文以外）</span>的业内支持。但其长远影响有待分晓。</p><p>案件中，谷歌对甲骨文API接口的复制使用被确认成了“合理使用”。为什么可以这么判定，美国最高法院从《版权法》中对适用“合理使用”的四个要件来说明：拥有版权作品的性质，使用的目的和特征，使用的数量和实质，对潜在市场的影响。</p><p>关于前三点，法院确认的是谷歌对代码的使用从各个意义上说，都属于“转换性使用”。虽然法条里在说明转换性使用时列举了很多非商业性使用，比如用于教学、学术，但商业性使用是不可避免的，比如新闻报道，必然会带来商业利益。所以商业性使用并不是不可以认定成“合理使用”。</p><p>最后一点“对潜在市场的影响”<span class="text-remarks" label="备注">（对整个互联网行业）</span>的影响，可能也是公众最关心，本案最受关注的点了。法官明确说明：<strong>我们确实考虑了对API接口的使用带来的公共利益，而且这一点在本案中非常关键。</strong></p><p>相比谷歌和安卓的胜利，更重要的是这个判例一举廓清了 API 这个领域的一系列版权纷争。判决里最重要的部分可能是判决中提出的“必须把公众利益<span class="text-remarks" label="备注">（包括是否阻碍后续创新）</span>纳入版权考量”的论述，这一部分的影响估计会超出科技业的范畴，影响到更广泛的领域。</p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193613635150.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" data-w="926" data-h="666" label="图片备注" src="https://img.huxiucdn.com/article/content/202104/06/193613635150.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" referrerpolicy="no-referrer"></p><p label="图片备注" class="text-img-note">比谷歌和安卓的胜利，更重要的是这个判例一举廓清了 API 这个领域的一系列版权纷争 | Christopher Dombres<br></p><p>目前看起来，比较明显可见的短期影响就是谷歌因为不用赔钱了所以股票大涨。但是潜在的影响是另一回事。</p><p>最大的可能性是以后会涌现出大量类似的做法，甚至超出软件领域，比如同人文创作——不，这不是打比方，而是这个判决本身是有可能直接应用在同人文领域的<span class="text-remarks" label="备注">（因为同人文创作和软件接口复用有相似之处）</span>。这是知识产权专家 Charles Duan 的看法。</p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193614269260.png?imageView2/2/w/1000/format/png/interlace/1/q/85" data-w="895" data-h="659" src="https://img.huxiucdn.com/article/content/202104/06/193614269260.png?imageView2/2/w/1000/format/png/interlace/1/q/85" referrerpolicy="no-referrer"></p><p>根据上面“潜在市场的影响”中法院的态度，也有很多读者会考虑到同人文作者是否也会面临同样的处境。美国就有一个同人文案件，派拉蒙诉阿克珊娜<span class="text-remarks" label="备注">（Paramount Pictures v. Axanar）</span>。阿克珊娜是超级《星际迷航》粉丝，她写了很长的同人小说，作品本身非常好，几乎可以作为《星际迷航》的背景故事，甚至也很有可能影响新的电影剧本。但是，派拉蒙起诉她侵犯了版权，她认为人名、道具这些小的元素并不在版权法保护范围内，但最后法院判决侵权成立。另外还有一位J.K.罗琳的粉丝也遇到了类似的情况。</p><p>美国最高法院在判决谷歌甲骨文案件时也非常谨慎，认为这是一个独立的案件，法院并没有改变传统的版权的概念，也并不想推翻或修改原有的“合理使用”的案件，比如说<span class="text-remarks" label="备注">（非正版）</span>仿制品、新闻写作、山寨产品等。看来立法机构，法院和大众对版权和“合理使用”的认知都会逐渐变化，适应新的版权形式，形成新的使用方法。</p><p class="img-center-box"><img class="lazyImg" _src="https://img.huxiucdn.com/article/content/202104/06/193617717974.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" data-w="1000" data-h="1053" label="图片备注" src="https://img.huxiucdn.com/article/content/202104/06/193617717974.jpg?imageView2/2/w/1000/format/jpg/interlace/1/q/85" referrerpolicy="no-referrer"></p><p label="图片备注" class="text-img-note">程序员表示：看着最高法判决中出现 Java 代码还是很好玩的</p><p><span class="text-remarks" label="备注">本文来自微信公众号：</span><a href="https://mp.weixin.qq.com/s/1JsKfB6Dr3ILpFNolXeTAw" target="_blank" rel="nofollow" style="text-decoration: none;"><span class="text-remarks">果壳（ID：Guokr42）</span></a><span class="text-remarks">，作者：木遥 & 银古桑</span></p>
