<p>近日，.NET开源社区<a href="http://mp.weixin.qq.com/s?__biz=MjM5MjAwODM4MA==&mid=2650885959&idx=1&sn=ec09ed00fa2df65096107610ebbf85e5&chksm=bd593e948a2eb7825585eddc72fa70740339af9326df29e7143efc5e17c3ca06bb2aba579562&scene=21#wechat_redirect">“再起波澜”</a>。</p> 
<p>只因<a class="project-link" data-id="3967413" data-name="微软" data-logo="https://img.36krcdn.com/20200916/v2_811751a081924fa9af8741ce120bd7bf_img_png" data-refer-type="2" href="https://36kr.com/projectDetails/3967413" target="_blank">微软</a>在即将发布的 .NET 6 中悄悄删除了 Hot Reload（热重载） 的核心部分，此举彻底激怒了 .NET 开源社区开发者，他们强烈反对微软这种行为，认为微软这样做极不利于 .NET 开源社区健康发展。</p> 
<h2 label="一级标题">1 删除“热重载”功能的意味着什么</h2> 
<p>据了解，所谓 Hot Reload（热重载）功能，是一个可以允许开发者在应用程序运行时更改源代码，并立即看到效果的关键功能。换句话说，就是在开发人员写代码过程中，获得修改代码的即时反馈，从而帮助软件开发人员提高开发效率。</p> 
<p><img src="https://img.36krcdn.com/20211026/v2_34903db784c74258a36402634d89691a_img_000" data-img-size-val="1078,466" referrerpolicy="no-referrer"></p> 
<p label="图片描述" classname="img-desc" class="img-desc">来源于微软官网</p> 
<p>此前，微软曾高调宣布将在 .NET 6 中加入“热重载”功能，但是微软又悄悄地从名为 dotnet watch 的工具中删除 2500 行“热重载”功能的源代码。</p> 
<p>10 月 20 日，微软项目经理 Dmitry Lyalin 发布一篇博文表示：“我们决定从即将发布的 .NET 6 GA 开始，现在只通过 Visual Studio 2022 启用热重新加载功能。”</p> 
<p><img src="https://img.36krcdn.com/20211026/v2_9e359b5168994526957f1d5873843f2b_img_000" data-img-size-val="764,463" referrerpolicy="no-referrer"></p> 
<p label="图片描述" classname="img-desc" class="img-desc">来源于微软官网</p> 
<p>就是这个消息，瞬间引爆了整个 .NET 社区。对于不熟悉开源社区的朋友来说，微软仅仅删除一个“热重载”功能，.NET 社区开发者为什么这么激动呢？</p> 
<p>实际上，微软删除 .NET 6.0 的热重载功能是将其限制在自家产品 Visual Studio 开发平台上，拒绝开放给其它平台使用，微软此举已经违背了开源社区的“发展初心”。</p> 
<p>准确来说，微软已经将开源 .NET SDK 故意削弱了，使得微软自家免费的商业产品 Visual Studio 比竞争对手更具吸引力，包括微软赞助的 Visual Studio Code。最重要的是，此次更改微软是在没有与 .NET 社区协商的情况下完成的。</p> 
<p>为此，人<a href="http://mp.weixin.qq.com/s?__biz=MjM5MjAwODM4MA==&mid=2650885959&idx=1&sn=ec09ed00fa2df65096107610ebbf85e5&chksm=bd593e948a2eb7825585eddc72fa70740339af9326df29e7143efc5e17c3ca06bb2aba579562&scene=21#wechat_redirect">们联想到之前关于 .NET 基金会的“数次危机”</a>。此前一位卸任的董事会成员就对 .NET 基金会的角色提出了质疑，询问它是否仅代表微软的意愿行事、还是致力于帮助培养和促进一个健康的社区？</p> 
<p>据了解，微软此次不仅将 .NET 6.0 的热重载功能删除，目前 macOS 和 Linux 开发人员那里也删除了该选项，内部人士称：“现在这个功能真的成为了 Windows 和 Visual Studio 开发平台独有的工具了”。</p> 
<h2 label="一级标题">2 微软道歉：恢复 .NET SDK 热重载功能</h2> 
<p>由于 .NET 社区强烈谴责微软从 .NET SDK 中删除了热重载（Hot Reload）功能，微软不得不出来道歉，并宣布恢复 .NET 中的热重载功能。</p> 
<p><img src="https://img.36krcdn.com/20211026/v2_2fd875cd99584d4e94f5ddb227cad9b0_img_000" data-img-size-val="982,527" referrerpolicy="no-referrer"></p> 
<p label="图片描述" classname="img-desc" class="img-desc">恢复热重载功能GitHub截图</p> 
<p>10月23日，微软官方发布博文道歉：“首先，也是最重要的，我们要道歉。我们在执行我们的决定时犯了一个错误，花了比预期更长的时间来回复社区。我们已经批准了重新启用这一代码路径的拉取请求，它将在 .NET 6 SDK 的 GA 构建中出现。”</p> 
<p><img src="https://img.36krcdn.com/20211026/v2_72feccf89d1a4107952bafa71d649ac8_img_000" data-img-size-val="679,502" referrerpolicy="no-referrer"></p> 
<p label="图片描述" classname="img-desc" class="img-desc">来源于微软官网</p> 
<p>微软方面解释称，因为于绝大多数 .NET 开发人员都在使用 Visual Studio，他们为了确保 VS 为 .NET 6 提供最佳体验，所以他们选择首先将 Hot Reload 引入 VS 2022。</p> 
<p>同时微软强调是无意中删除了这段源代码，而不是不调用该代码路径。在文章的最后，微软再次进行了道歉，并表示：他们的愿望是为 .NET 创造一个开放和充满活力的生态系统。和许多公司一样，他们也正在学习如何平衡 OSS 社区的需求和作为 .NET 的企业赞助商之间的关系。</p> 
<p><img src="https://img.36krcdn.com/20211026/v2_f05b7de97ba34048bfc1df8c11f36bcc_img_000" data-img-size-val="486,269" referrerpolicy="no-referrer"></p> 
<p label="图片描述" classname="img-desc" class="img-desc">来源于微软官网</p> 
<h2 label="一级标题">3 网友对此看法不一</h2> 
<p>即便微软决定恢复 .NET SDK 中热重载（Hot Reload）功能，还是引起了众多网友的议论。</p> 
<p>@Georg Dangl 网友认为：这是开源理念的一个很好的证明——问题<a class="project-link" data-id="95377" data-name="得到" data-logo="https://img.36krcdn.com/20210807/v2_966db147ab4646ef82349f069ce61219_img_000" data-refer-type="2" href="https://36kr.com/projectDetails/95377" target="_blank">得到</a>了承认，反馈得到了迅速解决。</p> 
<p>@Cory Crooks 网友并不认可微软官方说法。他们认为“微软无意中删除了源代码，而不是不调用该代码路径”。这是否意味着代码将在那里，现在仍然无法从命令行使用？</p> 
<p>@ Allan Lindqvist 网友认为这篇博文听起来是个好消息，但并没有说明如何恢复热重载（Hot Reload）功能，也没有解释微软是否试图以牺牲其他平台为代价，促进 Visual Studio 成为 .net 生态系统中的“一等公民”等问题。</p>
