<div class="articleWidth-content" data-v-6dd0eb6a><div class="update-wrap" data-v-6dd0eb6a></div><div class="content wangEditor-txt minHeight" data-v-6dd0eb6a><blockquote><p style="margin-left:0px;">TL;DR 这是一个关于 NextSSH 的文章。软件的官网：<a href="https://xzhshch.com/">https://xzhshch.com/</a> （前排付费软件提醒）</p></blockquote><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/b865eafc41255e0e2403710b3adb567b.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/b865eafc41255e0e2403710b3adb567b.png" referrerpolicy="no-referrer"><figcaption>NextSSH 在多任务分屏下的截图</figcaption></figure><p style="margin-left:0px;">PS：本文撰写于 NextSSH v1.2.2 发布后，部分截图来自开发中的 NextSSH v1.3.0-beta，可能存在部分细节上的差异。</p><h2 style="margin-left:0px;"><strong>前因</strong></h2><p style="margin-left:0px;">各位好，我是 NextSSH 的开发者。</p><p style="margin-left:0px;">我最早开始对计算机编程感兴趣时，接触的是 C51 单片机、无线电入门和网站开发（PHP）。（请不要吐槽这个跨度 😆）</p><p style="margin-left:0px;">随着网站开发出现在我的技能树里，远程连接服务器就变成了基本操作。在早期的互联网资料中，连接远程服务器的主要工具是 Putty，因为他绿色、开源、<s>无公害</s>。</p><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/3ee97b19f8f47ddc0198950b689b13fa.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/3ee97b19f8f47ddc0198950b689b13fa.png" referrerpolicy="no-referrer"><figcaption>Putty（图片来自网络）</figcaption></figure><p style="margin-left:0px;">我看到这个界面后，坚定的选择了 Windows Server 和 IIS 作为服务器。😅</p><p style="margin-left:0px;">看上去 Windows Server 和 Linux 之间的选择和 Putty 没什么关系。但是回想一下，一个孩子，第一次拥有一台服务器（啊当然是 VPS），手足无措又想按耐不住好奇心的时候，一个黑框框和一个熟悉的桌面，哪一个更有安全感呢。</p><p style="margin-left:0px;">Linux 当时给我一种捉摸不透的感觉，时常让我觉得不知道正在发生什么。当然，这个感受随着年龄的增长和之后的学习消散了。</p><p style="margin-left:0px;">但是，有一个“执念”留了下来。直观、清晰、设计良好的的软件界面能特别吸引我的注意力。</p><h2 style="margin-left:0px;"><strong>盒子里都有什么？</strong></h2><p style="margin-left:0px;">我觉得我还算是用过不少 SSH 客户端软件。终端+ssh 命令、Putty、Finalshell、Xshell、Termius、Tabby、Windterm、Electerm。</p><p style="margin-left:0px;">暂时放下远程终端这个本身功能不谈。有一些功能可以称得上“左膀右臂”：</p><ul><li>系统数据的图形化显示（CPU、内存用量，网络状态……）</li><li>SFTP</li><li>端口转发</li><li>保存常用的命令片段</li></ul><p style="margin-left:0px;">除了这些功能外，有一些是软件设计上十分实用的特点：</p><ul><li>打包传输文件</li><li>在线的文件编辑器</li><li>分屏、多窗口管理</li></ul><p style="margin-left:0px;">当思考这些功能时，其实就能回答一个问题「为什么不用 ssh 命令？」</p><p style="margin-left:0px;">「因为我在将简单的问题复杂化。」</p><p style="margin-left:0px;">我并非想要找到一个朴素的工具，而是想寻找一个趁手的工具箱。当然，工具箱里的东西越简洁越好。</p><h3 style="margin-left:0px;"><strong>数据监看</strong></h3><p style="margin-left:0px;">说起来，这个功能直接来自于 Finalshell 的影响。</p><p style="margin-left:0px;">从一个界面上一下子了解系统的诸多数据，曾经帮助我很快的定位了内存泄漏、文件膨胀之类的问题。</p><p style="margin-left:0px;">于是我为 NextSSH 设计了一个起始页。就像是浏览器的起始页，当新打开一个服务器的连接时，先看到的就是这个页面。</p><p style="margin-left:0px;">诚然，NextSSH 的数据汇总页面比不上专门监控服务器的工具。而且，NextSSH 本身也不会记录历史数据或绘制图表。但这都不是问题，因为这个功能的名字叫做“监看”。想象一下导演坐在监视器前，确认各个摄影机位的画面是否满意的感觉，就是这个功能的意图。</p><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/331ebe8a746c62dfe4bf74a855408c86.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/331ebe8a746c62dfe4bf74a855408c86.png" referrerpolicy="no-referrer"></figure><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/f9310e03e25a2ac4b8c363d3056bb7e5.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/f9310e03e25a2ac4b8c363d3056bb7e5.png" referrerpolicy="no-referrer"></figure><h3 style="margin-left:0px;"><strong>终端</strong></h3><p style="margin-left:0px;">终端就是 SSH 软件的本体功能了。</p><p style="margin-left:0px;">NextSSH 在除了终端本身的基础上，有两个我得意的设计。其一是下面的编辑栏。其实很多软件也有类似的功能。</p><p style="margin-left:0px;">一般情况下命令都比较短小精干，毕竟命令行交互是很简洁的。</p><p style="margin-left:0px;">但，总有些那么意外的情况，比如 Docker 的启动命令。每当从网上复制一个启动命令过来，总是不得不有一些参数需要修改，而在 Shell 里反复移动光标总是没有鼠标痛快。</p><p style="margin-left:0px;">于是终端下面就增加了一个可以展开的编辑器。可以临时停靠复制来的长命令。</p><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/c819f9047b4201379c2b952b33757201.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/c819f9047b4201379c2b952b33757201.png" referrerpolicy="no-referrer"></figure><p style="margin-left:0px;">另一个功能则来自用户之前的反馈。</p><p style="margin-left:0px;">在终端界面下按 <code>Ctrl/Cmd+Shift+K</code> 即可呼出一个搜索栏。</p><p style="margin-left:0px;">这个搜索栏可以查找所有曾经输入过的命令，也包括其他主机上的输入，以及保存过的命令片段。理论上可以把常用的配置、重启服务、安装软件之类的命令保存下来，在这就可以快速使用。</p><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/931dcd5c86a379310ede7155063be8f3.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/931dcd5c86a379310ede7155063be8f3.png" referrerpolicy="no-referrer"></figure><h3 style="margin-left:0px;"><strong>SFTP 与编辑器</strong></h3><p style="margin-left:0px;">「如果一个 VS Code 嵌在 SSH 里岂不是很有趣……」</p><p style="margin-left:0px;">我当时就是这么想的。</p><p style="margin-left:0px;">最早接触网站开发的我，将常被反反复复修改文件折磨。因为调试或者部署网站就有 <code>PHP配置文件NGINX配置文件APACHE配置文件数据库配置文件网站本身的CONFIGINC文件……</code> 一大堆文件需要反复调整和编辑。</p><p style="margin-left:0px;">一般情况下可以使用 vim 等工具在 Shell 里进行编辑。<s>然后就卡在 vim 里不知道怎么保存退出。</s></p><p style="margin-left:0px;">有一些工具之后，可以把文件下载到本地，使用常用的编辑器修改后重新上传覆盖。除了步骤多一些，也很好用。</p><p style="margin-left:0px;">小孩子才做选择，我选择自己做一个选项。一个嵌入到软件里的 Monaco 编辑器，带有语法高亮、多编码支持。双击文件就能打开，<code>Ctrl/Cmd+S</code> 就能保存。</p><p style="margin-left:0px;">更有趣的是，NextSSH 可以在 Shell 环境里注入一个 <code>nsopen</code> 命令，使用命令可以在远程终端里打开本地 NextSSH 的编辑器。（注入命令会自动执行脚本，具体内容和功能禁用方法请参考<a href="https://xzhshch.com/nextssh/Sensitive-or-automatic-operation.html#nsopen%E5%91%BD%E4%BB%A4-%E5%9C%A8%E8%BF%9E%E6%8E%A5%E7%BB%88%E7%AB%AF%E6%97%B6%E8%87%AA%E5%8A%A8%E6%89%A7%E8%A1%8C%E8%BE%85%E5%8A%A9%E8%84%9A%E6%9C%AC">文档</a>）</p><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/66ba353d5f5dd673ceea93923e9414af.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/66ba353d5f5dd673ceea93923e9414af.png" referrerpolicy="no-referrer"></figure><p style="margin-left:0px;">另外值得一提的是文件打包传输功能。比如我需要将网站的源码上传到服务器上，直接将文件夹拖到 NextSSH，之后就能自动完成压缩、上传、解压缩的步骤。</p><p style="margin-left:0px;">打包传输文件其实和正常传输的效果没有区别。只是网站的源码文件可能有成百上千个小文件，反复传输小文件将把时间都浪费在 SFTP 的协议指令和网络延时上。</p><figure class="image ss-img-wrapper"><img src="https://cdn.sspai.com/2022/05/13/3b5966de1568f06ad75ad49081558e51.png?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" data-original="https://cdn.sspai.com/2022/05/13/3b5966de1568f06ad75ad49081558e51.png" referrerpolicy="no-referrer"></figure><h2 style="margin-left:0px;"><strong>结语</strong></h2><p style="margin-left:0px;">经过来来回回的修改和推敲，才有了 NextSSH 现在的样子。如果你去搜索引擎里翻找，还能看见它之前的样子。在 v1 之前的界面几乎和现在是两个东西。</p><p style="margin-left:0px;">如果你经常需要和服务器打交道的话，如果你喜欢 NextSSH 的设计的话，都十分欢迎你下载软件来体验一下 <a href="https://xzhshch.com/">https://xzhshch.com/</a>。</p><p style="margin-left:0px;">如果遇到什么问题，在网站上与软件里都有反馈的渠道。</p><p style="margin-left:0px;">以上，祝愉快。</p></div><div class="update-details-wrap"
