<div class="articleWidth-content" data-v-6a669db8><div class="content wangEditor-txt minHeight" data-v-6a669db8><p style="margin-left:0;">本文将介绍一台有些年代感，低功耗、相对廉价的 MiniPC： NUC7CJYH。在使用了半年之后，我觉得是一个合适的时间点来“盘一盘”它了。</p><h2><strong>写在前面</strong></h2><p style="margin-left:0;">今年3月初购置了一台小巧的 NUC7，之所以购置这台落后的设备，有三个原因：</p><ul><li>之前使用<a href="https://soulteary.com/2020/08/12/synology-data-migration-at-ds-920.html"><strong>群晖</strong></a>跑一些简单服务，因为存在磁盘读写，导致硬盘无法休眠。费电不说，还会有隐隐约约的磁盘运行声音，不能忍 。 <strong>我希望换个设备来跑这些服务，可以让耳根子更清净一些。</strong></li><li>之前的 Confluence 和一些需要远程访问的服务运行在 EliteDesk 家用服务器上，随着设备升级更新成“笔记本”后，我经常重启折腾新的设备环境。 <strong>我希望有一个稳定的新环境运行它们。</strong></li><li><strong>我希望这个设备能够足够静音，并且因为是 7x24 小时运行，我希望它能够相对省电。</strong></li></ul><figure class="image ss-img-wrapper image_resized" style="width:100%;"><img src="https://cdn.sspai.com/2021/10/19/article/b6512e6ef66798655f133386a3001849?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" alt="之前购买的设备们，最上面那台薄机就是主角" data-original="https://cdn.sspai.com/2021/10/19/article/b6512e6ef66798655f133386a3001849" referrerpolicy="no-referrer"><figcaption>之前购买的设备们，最上面那台薄机就是主角</figcaption></figure><p> </p><p style="margin-left:0;">早些时候买过一台 <a href="https://soulteary.com/2021/01/31/nuc-notes-linux-system.html"><strong>NUC8</strong></a>，对 NUC 这种 Mini PC 持续跑服务，还是比较放心的。美中不足的是，NUC8 的散热不佳和风扇噪音过高让我比较无奈。（解决方案见<a href="https://soulteary.com/2021/10/14/recent-heat-dissipation-upgrade-record-of-homelab.html"><strong>《近期家用设备（NUC、猫盘、路由器）散热升级记录》</strong></a>）。产生上面这个问题的原因，主要还是 CPU 功耗太高，机箱自身散热能力有限，机器内置小风扇搞不定热量排出的原因。那么，如果我**<strong>使用功耗低不少的赛扬系列的 CPU ，能不能解决这个问题呢？</strong> **</p><p style="margin-left:0;">由于之前购置过 j3455（ 群晖 DS720+ ） 和 j4125 （ 群晖 DS 920+ ）的设备，并且用它们跑过一些容器服务，所以对类似赛扬系列的低功耗 CPU 还是比较有信心的。NUC 迄今为止推出过两款性能尚可、功耗较低的设备，分别是 <a href="https://ark.intel.com/content/www/tw/zh/ark/products/126137/intel-nuc-kit-nuc7pjyh.html"><strong>NUC7PJYH</strong></a>（j5005） 和 <a href="https://ark.intel.com/content/www/us/en/ark/products/126135/intel-nuc-kit-nuc7cjyh.html"><strong>NUC7CJYH</strong></a>（j4005）。虽然 CPU 命名都是 j 开头，但是两者分别归属“奔腾”和“赛扬”两个系列，虽然两者是连续的“几千”命名，看似跨了一代，但其实两者发布时间几乎一致，两者主要差异在于前者默频比较低，但是核心数量是后者的一倍，当然，价格上同为裸机，前者稍贵一些。由于之前使用的多是“赛扬”系列的低功耗设备，加之我购买的店铺恰好只到货了“j4005”版本，于是我也就入手了这个“低配”版本。</p><h2><strong>硬件规格及配置清单</strong></h2><p style="margin-left:0;">再次回顾一下这台机器的配置，你会发现是一台宝藏主机（让你被动省钱）：</p><ul><li><strong>只配备了一块 SATA 硬盘位</strong>。<ul><li>相比较 NVMe 便宜一些，而且因为 CPU 相对孱弱，也不需要使用它跑啥比较重的软件，最终存储空间也可以选择小一些。而且因为是 SATA 接口，随便一块 SSD 就能到达带宽天花板。</li></ul></li><li><strong>只支持 DDR4-2400 规格的内存</strong>，并且上限限制在了 8GB（实际系统显示主板支持 32G内存）。<ul><li>相比较 3200/3600/4000+ 频率的内存，2400 的内存价格也是便宜不少，甚至不需要追求多通道，随便插一根就行。</li></ul></li></ul><p style="margin-left:0;">由于 NUC 众所周知的“购买文化”，全新的准系统实际到手价格在 799。（年初价格，目前应该更便宜才是）考虑到我只是运行前文中提到的软件，以及希望机器尽可能在运行时“保持安静”，硬盘随便买了一块 200 块出头的浦科特，内存的话，则是用了之前升级替换下来的国货之光：光威弈Pro的内存条，综合成本在千元左右。平时运行功耗在 1w～4w 之间，一年电费成本在 17 块钱左右，按照机器能够持续运行 3 到 5 年来看（前3年是官方质保），年均成本不到 200 ，折合每天的成本在 0.5 元左右，属于真香系列。</p><figure class="image ss-img-wrapper image_resized" style="width:100%;"><img src="https://cdn.sspai.com/2021/10/19/article/b3886636bb611f465850e3d9e7f6e497?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" alt="机柜内叠罗汉的 NUC" data-original="https://cdn.sspai.com/2021/10/19/article/b3886636bb611f465850e3d9e7f6e497" referrerpolicy="no-referrer"><figcaption>机柜内叠罗汉的 NUC</figcaption></figure><p style="margin-left:0;">收起账本，扔掉规格表，来看看我用它跑了哪些服务。</p><h2><strong>运行服务</strong></h2><p style="margin-left:0;">如前文提到的，这台设备主要用于提供以下几个方面的功能：文档知识库、页面收藏夹、RSS、远程访问、网络唤醒、计划任务。</p><h3 style="margin-left:0;"><strong>文档知识库</strong></h3><p style="margin-left:0;">我最初使用它运行 <a href="https://soulteary.com/subject/tech/#confluence"><strong>Confluence</strong></a> 这个众所周知的资源大户，在这台设备上实际表现状态还是比较赞的，几乎所有页面都是秒开，配合 <a href="https://soulteary.io/2020/10/04/frp-in-docker.html"><strong>FRP</strong></a> ，和一些前端优化，可以做到不在家的时候，在远程时依旧是秒开的丝滑体验。</p><p style="margin-left:0;">但是随后因为官方公布不再出售授权，仅提供海外的在线 SaaS 服务，虽然我的授权会在 2022 年到期，但是考虑到长久使用，免不了要未雨绸缪。兜兜转转，最终将方案切换到了资源需求更低的，并且开源的方案：<a href="https://soulteary.com/2021/09/05/opensource-documentation-wiki-software-outline-part-1.html"><strong>Outline</strong></a> 。</p><figure class="image ss-img-wrapper image_resized" style="width:100%;"><img src="https://cdn.sspai.com/2021/10/19/article/edacfeb72492d08fc8a1ae7b19bebfb8?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" alt="Outline 日常使用界面" data-original="https://cdn.sspai.com/2021/10/19/article/edacfeb72492d08fc8a1ae7b19bebfb8" referrerpolicy="no-referrer"><figcaption>Outline 日常使用界面</figcaption></figure><h3 style="margin-left:0;"><strong>页面收藏夹</strong></h3><p style="margin-left:0;">考虑到当今网页资料“失效速度”越来越快，对于一些写的不错的参考资料，使用超链接书签的形式进行资料保存，最终得到的势必是一个 “404 Not Found”、或者“另作它用的地址”。</p><p style="margin-left:0;">而如果未经整理，就堆在笔记系统里，那么笔记系统也会因为这些没有结构化整理的内容，而变得臃肿，散发着糟糕的味道。</p><p style="margin-left:0;">我在 <a href="https://github.com/HypothesisWorks/hypothesis"><strong>hypothesis</strong></a> 和 <a href="https://github.com/ArchiveBox/ArchiveBox"><strong>ArchiveBox</strong></a> 中徘徊了一阵，最终决定暂时先使用后者进行网页内存的持久化保存，后者配合 <a href="https://github.com/valeriansaliou/sonic"><strong>sonic</strong></a> （类似 ES 的轻量实现）能够方便的实现高性价比的全文检索。</p><figure class="image ss-img-wrapper image_resized" style="width:100%;"><img src="https://cdn.sspai.com/2021/10/19/article/7f6cb55b3f367e4588d1a20002f0700a?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" alt="ArchiveBox  界面" data-original="https://cdn.sspai.com/2021/10/19/article/7f6cb55b3f367e4588d1a20002f0700a" referrerpolicy="no-referrer"><figcaption>ArchiveBox 界面</figcaption></figure><p> </p><p style="margin-left:0;">如果你只是想完全自我管理收藏夹，也可以考虑使用 <a href="https://github.com/go-shiori/shiori"><strong>shiori</strong></a>。不过五月份的时候，给官方提供两个 <a href="https://github.com/go-shiori/shiori/pulls/soulteary"><strong>PR</strong></a>，至今没有维护者反馈和合并。</p><h3 style="margin-left:0;"><strong>RSS</strong></h3><p style="margin-left:0;"><a href="https://soulteary.com/subject/tech/#rss"><strong>Miniflux</strong></a> 作为“高性价比”的自部署软件，拥有极低的资源占用，以及简洁的界面，你既可以使用它自带的 Web 界面进行信息浏览，也可以使用支持兼容 Fever 协议的阅读器来进行阅读，比如：Reeder。</p><figure class="image ss-img-wrapper image_resized" style="width:100%;"><img src="https://cdn.sspai.com/2021/10/19/article/11a067c23c76d9cb7208872e88c83f94?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" alt="MiniFlux 简洁的界面" data-original="https://cdn.sspai.com/2021/10/19/article/11a067c23c76d9cb7208872e88c83f94" referrerpolicy="no-referrer"><figcaption>MiniFlux 简洁的界面</figcaption></figure><p> </p><h3 style="margin-left:0;"><strong>远程访问</strong></h3><p style="margin-left:0;">如果你的网络环境能够提供直接链接的话，使用 DDNS 的方式直接访问能够获得非常好的访问质量，搭配自己的 DDNS 服务（比如<a href="https://soulteary.com/2021/07/31/use-nginx-to-provide-ddns-service-part-1.html"><strong>《使用 Nginx 提供 DDNS 服务》</strong></a>），使用起来安全可靠。</p><p style="margin-left:0;">但是，如果你有国内的云服务器（哪怕是1c1g的共享带宽小主机），用来作为备份线路都是不错的方案。（比如<a href="https://soulteary.com/2020/10/04/frp-in-docker.html"><strong>《容器化 FRP 使用方案》</strong></a>）相比较第一个方案，完全不用担心宽带运营商突然改变策略造成不可用。</p><h3 style="margin-left:0;"><strong>网络唤醒</strong></h3><p style="margin-left:0;">相对于这台设备而言，有的设备我并不希望它同样以 7x24 的方式运行，而是按需启动，提供服务。</p><p style="margin-left:0;">虽然路由器后台支持网络唤醒，但哪怕进行了加密处理、直接将掌握所有设备联网管理权限的主路由对外提供访问，也是不安全的。所以，我用 Node 写了一个简单的小服务，用于快速进行指定设备的关闭和快速的网络唤醒。</p><h3 style="margin-left:0;"><strong>计划任务</strong></h3><p style="margin-left:0;">虽然我使用的服务，都能使用 Docker 运行起来，并且都能通过容器 “Health Check” 的方式来做计划任务和健康检查，确保服务稳定。但是如果一律使用 Health Check 的方式来刷新计划任务，未免有些“不环保”。况且，偶尔也需要执行临时的计划任务。我使用了一款叫做 <a href="https://github.com/jhuckaby/Cronicle"><strong>Cronicle</strong></a> 的软件来进行任务的执行和监控。</p><figure class="image ss-img-wrapper image_resized" style="width:100%;"><img src="https://cdn.sspai.com/2021/10/19/article/a102bdc7c404c7f8e1fec1b6a358aa22?imageView2/2/w/1120/q/40/interlace/1/ignore-error/1" alt="Cronicle 计划任务界面" data-original="https://cdn.sspai.com/2021/10/19/article/a102bdc7c404c7f8e1fec1b6a358aa22" referrerpolicy="no-referrer"><figcaption>Cronicle 计划任务界面</figcaption></figure><h2><strong>其他</strong></h2><p style="margin-left:0;">这台设备从购置至今，几乎没有运行过图形化界面的程序（BIOS设置还是得在图形化界面下），所以如果你想了解，拿它来外接一块两块显示器性能如何，我应该是回答不了你。</p><p style="margin-left:0;">但是，从使用它用容器的方式稳定的运行上述服务，整个机器外壳几乎没有温度而言，我觉得这台设备买的挺值的。</p><p style="margin-left:0;">关于上述软件的搭建的使用，先挖个坑，稍后慢慢整理吧。</p><h2><strong>最后</strong></h2><p style="margin-left:0;">许多时候，我们遇到的问题都可以通过“小马过河”的方式来解决，水深水浅，自己一试便知。</p><p style="margin-left:0;">--EOF</p><hr><p
