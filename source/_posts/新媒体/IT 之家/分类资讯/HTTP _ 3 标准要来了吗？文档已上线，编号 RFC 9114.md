<p data-vmark="1b51"><a class="s_tag" href="https://www.ithome.com/" target="_blank">IT之家</a> 6 月 8 日消息，IETF（互联网工程任务组）宣布了 <span class="accentTextColor">HTTP / 3 标准</span>，在 RFC Editor 上已经出现了<a href="https://www.rfc-editor.org/info/rfc9114" target="_blank">文档</a>，编号为 <span class="accentTextColor">RFC 9114</span>，目前状态为“PROPOSED STANDARD”，还未成为正式标准。</p><p data-vmark="e676" style="text-align: center;"><img src="https://img.ithome.com/newsuploadfiles/2022/6/00c79cfb-3aa4-4e9b-bbdc-00967b5f3945.png?x-bce-process=image/watermark,image_aW1nL3dhdGVybWFyay9xdy9xdzE3MC5wbmc=,g_3,x_17,y_17,a_0,t_100" w="1106" h="766" title="HTTP / 3 标准要来了吗？文档已上线，编号 RFC 9114" width="1106" height="568" referrerpolicy="no-referrer"></p><p data-vmark="5e79">互联网工程任务组（IETF）最早于 2015 年启动了 HTTP / 2，它带来的主要改进之一是对重用的支持。但是它使用 TCP 作为传输协议和 TCP 中的丢失恢复机制，<span class="accentTextColor">因此丢失的数据包仍然会导致所有活动事务的延迟</span>。</p><p data-vmark="134a">HTTP / 3 则采用了谷歌多年探索的基于 UDP 的 QUIC 协议，原名叫 HTTP-over-QUIC，在 2018 年被 IETF 批准更名为 HTTP / 3。</p><p data-vmark="4ef8">HTTP /3 可以进一步改善传输过程，<span class="accentTextColor">因为丢失的数据包只会影响直接受其影响的活动</span>。在相同丢包率的情况下，HTTP / 3 相比 HTTP / 2 的请求时间大幅降低。</p><p data-vmark="54e6" style="text-align: center;"><img src="https://img.ithome.com/newsuploadfiles/2022/6/33f330af-a3a7-456e-9e91-d3dbddb214e6.png" w="1440" h="810" title="HTTP / 3 标准要来了吗？文档已上线，编号 RFC 9114" width="1440" height="461" referrerpolicy="no-referrer"></p><p data-vmark="47e4">文档页面显示：</p><blockquote><p data-vmark="9713">QUIC 传输协议具有 HTTP 传输所需的几个特性，例如流多路复用、每个流的流控制和低延迟连接建立。本文档描述了 HTTP 语义在 QUIC 上的映射。本文档还确定了 QUIC 包含的 HTTP / 2 功能，并描述了如何将 HTTP / 2 扩展移植到 HTTP / 3。</p></blockquote><p data-vmark="e2ce">IT之家了解到，虽然还未成为正式标准，但 HTTP / 3 已经被多数主流浏览器支持，包括谷歌 Chrome、Firefox 火狐等。</p>
