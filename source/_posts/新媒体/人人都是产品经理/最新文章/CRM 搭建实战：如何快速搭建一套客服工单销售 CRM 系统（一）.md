<div>   
<blockquote><p>编辑导读：对于B端行业的互联网人来说，搭建CRM系统是一个常见的业务。本文从收到需求开始，逐步分析、拆解、梳理需求，由客服工作流入手，设计出一套提高客服效率的 CRM 工具，希望对你有帮助。</p></blockquote>
<p><img data-action="zoom" class="size-full wp-image-4572827 aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/05/Edrxn02TUtCet5NjO0Y3.jpg" alt width="900" height="420" referrerpolicy="no-referrer"></p>
<p>今天接到一家游戏工作室搭建「客服工单销售」 CRM 的需求。看过对方的需求，发现是一套该有的功能都有，但整体逻辑并不复杂的 CRM 。这个案例非常适合刚刚开始做 to b 业务或正准备给自己公司搭建一套 CRM 的小伙伴参考。</p>
<p>本文从收到需求开始，逐步分析、拆解、梳理需求，由客服工作流入手，设计出一套提高客服效率的 CRM 工具。我们来一起思考整套系统搭建中的关键。</p>
<h2 id="toc-1">一、收到客户需求并分析</h2>
<h3>1. 客户需求</h3>
<p><img data-action="zoom" class="aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/05/Khph6hBneq1RCnFFPMeR.png" alt width="773" height="559" referrerpolicy="no-referrer"></p>
<p style="text-align: center;">客户微信发来一段需求</p>
<p>作为 to b 的产品或后端，一般收到客户（甲方）或业务部门（公司内部）提的需求时，都会接到类似上面这类根据实际业务描述的工作流需求。</p>
<h3>2. 分析需求：梳理产品底层逻辑</h3>
<p>我们的产品设计应该站在使用者的视角，帮助使用者提高工作效率。我们要先回答这三个基本问题，帮助梳理产品设计思路。</p>
<ul>
<li>这套 CRM 系统给谁用？</li>
<li>操作者的工作流是什么？</li>
<li>如何帮助操作者提高效率？</li>
</ul>
<p>在回答这些问题之前，我们先要搞清楚，CRM 的本质是什么？在没有 CRM 前，上面这些工作要怎么完成呢？</p>
<p>其实用一张大白纸即可。</p>
<p><img data-action="zoom" class="aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/05/XOdJXHwGxv4L8MP3IkMe.png" width="513" height="177" referrerpolicy="no-referrer"></p>
<p><strong>大白纸：</strong>我们可以用一张大白纸手写用户信息（用户信息系统），记销售情况（销售系统）、用户问题（工单系统），但每一个变动，修改，查找都非常麻烦，纸质记录很容易到达数据上线，比如管理3万名用户的用户信息，纯纸质记录的劣势就显现出来了。</p>
<p><strong>Excel：</strong>大白纸的升级版是 Excel。有了 Excel 后，极大的提高了信息记录和信息流动的效率，我们可以很方便的以表单形式记录信息。使用用户 ID 为 key 来做关联的锚点，扩展出无限多的信息记录，他可以是销售记录，也可以是工单记录。但问题又来了，就算 Excel 再简单，一但开始构建稍微复杂的 CRM，情况就不一样了。上手学习成本，记录数据多样性（对长文本和富媒体不友好），多员工协同，都是 Excel 的劣势。</p>
<p><strong>可视化UI+数据库：</strong>Excel 的升级版是「可视化UI+数据库」，它可以是 ERP 、 CRM、OA、CMS、HRM、财务还是进销存。不论他叫什么，底层都是由「可视化UI+数据库」构成。它们区别在于对流入系统的数据，做了对应的产品逻辑优化，让操作系统的人，能够最高效便捷的使用这些数据。</p>
<h2 id="toc-2">二、搭建 CRM 思考</h2>
<p>具体到我们即将搭建的这套 CRM（客户关系管理），它的本质是以客户信息为圆心，分功能记录客户对应的需求，并跟踪这些需求至结束。</p>
<p><img data-action="zoom" class="aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/05/iPmMcEb1w63fq4w7cgtg.png" width="535" height="352" referrerpolicy="no-referrer"></p>
<p>搞明白 CRM 的本质后，我们来回答一下前文所提出的问题。</p>
<p><strong>1）这套 CRM 系统给谁用？</strong></p>
<p>工作室的客服。我们要站在他的角度观察他们的工作流程。</p>
<p><strong>2）操作者的工作流是什么？</strong></p>
<p>客服在 CRM 中创建、查找玩家信息（玩家信息系统），在「玩家信息」的基础上记录对应玩家提出的问题（工单系统）和与玩家之间的销售情况（销售系统）。</p>
<p><strong>3）如何帮助操作者提高工作效率？</strong></p>
<ul>
<li>「玩家信息管理」的关键是帮助客服快速找到对应的玩家并全面展示该玩家信息。</li>
<li>「工单系统」帮助客服快速查找每位玩家对应的工单，跟踪工单状态直至工单处理结束。</li>
<li>「销售系统」的关键是记录客服每次与玩家接触时的情况，方便后续跟进，以及成交后交易信息的记录。</li>
</ul>
<p>以上三个问题的答案是，我们在设计整套系统的过程中，应该时刻牢记在心的设计原则和目的。</p>
<p style="text-align: center;"><strong>玩家 ←信息输入 → CRM（信息承载） ← 信息输出 → 客服</strong></p>
<p>通过客户提交的运营工作流方案，我们可以看到，这套 CRM 由三个小系统构成。</p>
<p><img data-action="zoom" class="aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/05/judoh6cw621Uvdv2dS8J.png" alt="卡拉云 - CRM搭建" width="497" height="395" referrerpolicy="no-referrer"></p>
<ul>
<li>玩家信息管理系统</li>
<li>玩家工单系统</li>
<li>玩家销售系统</li>
</ul>
<p>「工单系统」和「销售系统」，都是建立在「玩家信息管理系统」基础上扩展出来的系统。</p>
<p>接下来，我们来逐一分析每套系统的逻辑，各个击破。</p>
<h2 id="toc-3">三、玩家信息管理系统 – 整套系统的信息起点</h2>
<p>「玩家信息」是整套 CRM 系统的信息流起点，系统要先录入玩家信息，才能开启后续几套系统。</p>
<p><strong>1）客服记录玩家信息工作流</strong></p>
<p>游戏客服的录入玩家信息的工作流是，玩家通过游戏中的流程引导添加客服微信。与客服初次建立连接后，客服要将玩家的信息录入「玩家管理系统」中，以方便接下来与玩家的互动。</p>
<p><strong>2）设计「玩家信息管理」产品逻辑</strong></p>
<p>客服输入信息及自动抓取信息：</p>
<ul>
<li>玩家手机号：玩家手机号是玩家信息与玩家游戏中的身份关系的对应key，系统需要通过玩家手机号到游戏服务器中抓取玩家游戏信息。</li>
<li>玩家微信名称及微信ID：客服与玩家交流的ID是微信，微信ID是客服用于识别玩家的识别码</li>
<li>玩家对应分配的客服微信及客服微信ID</li>
</ul>
<p>「信息管理系统」根据「玩家手机号」，通过 API 自动在游戏服务器中抓取玩家在游戏中的信息。</p>
<ul>
<li>玩家注册的游戏</li>
<li>玩家游戏中的昵称</li>
<li>玩家游戏所属服务器</li>
</ul>
<p>「玩家信息管理」页面看起来非常简单，但它是整套 CRM 的基石，搭建好这个页面，我们就可以在这个信息的基础上，构建「工单系统」和「销售系统」了。</p>
<h2 id="toc-4">四、工单系统 – 帮助客服跟踪玩家问题</h2>
<p>工单系统其实可以理解为 「玩家基础信息+玩家问答记录」，再添加上查找工单（搜索）和进度 tag 就可以了。</p>
<h3>1. 客服Q&A工作流</h3>
<ul>
<li>玩家在微信中询问客服问题。（玩家与客服接触点，输入问题）</li>
<li>客服拿到玩家微信ID，在工单系统中找到对应的玩家。（客服与 CRM 接触点）</li>
<li>将玩家询问问题记录在工单中。（记录问题）</li>
<li>寻中解决方案，通常需要与技术、产品、财务、文案等部门同事沟通。（解决问题）</li>
<li>找到解决方案，微信回复玩家并在系统中记录解决方案。（输出答案）</li>
<li>结束工单。</li>
</ul>
<h3>2. 设计「工单系统」产品逻辑</h3>
<ul>
<li>抓取「玩家信息管理」中的玩家基础信息。</li>
<li>创建「搜索框」，搜索玩家手机号、微信ID，方便客服找到对应的玩家。</li>
<li>创建「问题详情」与「解决方案」，客服可记录对应的信息。</li>
<li>工单并非实时完成，需要给工单三个状态，即「待处理」「处理中」「完成」。</li>
</ul>
<p>至此，客服在工单中需要处理的动作都完成了。</p>
<h2 id="toc-5">五、销售系统 – 帮助客服跟踪每一笔销售交易</h2>
<p>客服在服务游戏大R玩家时，会根据玩家当下情况，推销打包好的「游戏物品」合集礼包给玩家。客服需要一套系统记录在销售过程中与玩家沟通的情况，记录玩家状态，销售时为什么卖出去了，为什么没有销售成功。方便下一次继续有针对性的销售。</p>
<h3>1. 客服销售工作流</h3>
<ul>
<li>根据当下促销活动，客服联系自己手中管理的大R玩家，并向他们销售促销活动中的礼包。</li>
<li>如销售成功，记录销售礼包ID，玩家付款时间、付款方式、付款金额。</li>
<li>如销售未成功，记录销售情况，记录未成功原因，方便下次销售时，接续上次销售情况继续推销</li>
</ul>
<h3>2. 设计「销售系统」产品逻辑</h3>
<ul>
<li>抓取「玩家信息管理」中的玩家基础信息，便于客服找到对应的玩家。</li>
<li>玩家基本信息展示，让客服看一眼了解全部玩家信息，以便有针对性的进行销售。</li>
<li>销售记录：记录每一次向玩家销售时的情况，帮助之后更有针对性的推销礼包。</li>
<li>成单信息记录：详细记录玩家购买礼包ID，付款金额、方式、时间等成交记录。</li>
</ul>
<h2 id="toc-6">六、系统设置 – 系统初始启动信息</h2>
<p>系统的起始信息配置页面，比如客服信息，游戏物品信息，游戏礼包信息等。这些配置信息是启动这套 CRM 的基本信息。</p>
<h3>1. 客服信息设置</h3>
<p>游戏公司的客服团队由多人组成，这套 CRM 也 是提供多名客服使用。</p>
<p>一个客服 user 信息由3个部分构成：</p>
<ul>
<li>客服 ID。（系统中客服身份唯一标示）</li>
<li>客服的微信号。（客服与玩家之间由微信连接，微信ID与客服ID唯一对应）</li>
<li>客服的微信名称。（方便客服直观的找到登录这个微信对应的手机）</li>
</ul>
<p>客服本人只需要记住自己的ID，即可在系统中通过自己的ID来关联其他信息。客服ID是系统中客服身份的唯一标示。</p>
<h3>2. 礼包管理页</h3>
<p>礼包是游戏物品的合集，礼包的销售价格是礼包中包含物品价格总和的一个折扣价。客服可根据自己手中维护的大R玩家，自己组合一套礼包进行销售。</p>
<p><strong>1）礼包管理工作流</strong></p>
<ul>
<li>系统通过游戏 API 自动抓取物品价目表，客服可在物品管理中进行微调。</li>
<li>客服根据自己手中玩家情况，在礼包页面根据公司内部优惠规则，创建礼包。</li>
<li>在礼包列表页，查看礼包对应的礼包ID，在销售页面填入对应ID，系统即可自动抓取对应信息。</li>
</ul>
<p><strong>2）设计「礼包管理」产品逻辑</strong></p>
<ul>
<li>礼包管理页面最小单位是物品，首先我们要通过 api 连接到游戏服务器把游戏中物品列表抓下来。</li>
<li>创建礼包页，客服可根据近期促销活动主题，给自己手中管理的大R玩家定制礼包。</li>
<li>礼包列表：可查看已经生成的礼包的ID、名称、价格，以及礼包中包含的物品信息。</li>
</ul>
<h2 id="toc-7">七、CRM 首页信息汇总 – 让客服工作更轻松</h2>
<p>系统首页应该汇集客服工作中较常用的功能，在不增加系统复杂性的前提下，能尽量在首页操作的，就让客服在首页完成。降低客服劳动强度，帮助客服提升自己的KPI（在本案，客服的KPI是 快速应答玩家需求（工单），尽量多的卖给玩家礼包（销售））</p>
<p><strong>1）首页工作流</strong></p>
<ul>
<li>客服在微信中收到玩家信息，在玩家信息列表中通过微信ID，找到该玩家</li>
<li>选中该玩家条目，工单和销售表单自动抓取该玩家对应的工单和销售记录</li>
<li>根据实际情况，客服可在首页直观的看到玩家信息，及直接操作「工单」、「销售记录」</li>
</ul>
<p><strong>2）设计首页产品逻辑</strong></p>
<ul>
<li>微信 ID 搜索框+玩家信息列表，方便客服快速定位玩家。选中玩家后，右侧显示全部玩家信息。</li>
<li>通过客服选取的玩家，系统自动调出此玩家对应的「工单」和「销售记录」。</li>
<li>在本案中，工单更常用，放在上面，客服不仅可在首页浏览工单，还可以直接新建/更新工单。</li>
<li>销售记录，抽取销售 touch point 关键部分，方便客服快速回顾此玩家的销售情况。</li>
</ul>
<h2 id="toc-8">八、写在最后</h2>
<p>万事万物皆有一而生，从最简单的 to B 系统开始入手，不断叠加更复杂的产品需求，逐渐构成一套有自己思考的大型系统。也许是比较好的学习 to B 产品的路径。</p>
