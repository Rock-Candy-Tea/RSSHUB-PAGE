<p><img data-action="zoom" class="aligncenter size-full wp-image-4516300" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/05/IVl9bbdmsu2j2ZGqW7As.jpg" alt width="900" height="420" referrerpolicy="no-referrer"></p>
<p>今天想跟大家讨论一下关于设计团队工作效率的问题。</p>
<p>不知道大家有没有在工作中有过类似的感受：</p>
<ul>
<li>项目周期过长，推进十分缓慢？</li>
<li>每天都在忙东忙西却没有什么实质进展？</li>
<li>跟组员的沟通不畅，无法做到有效对接？</li>
<li>……</li>
</ul>
<p>笔者根据自己以往的工作经验对一些设计师在工作中可能面对的问题进行了举例，并且总结了一些应对方法，希望能对大家有一点帮助。</p>
<p>如果大家觉得有所共鸣，希望我们可以逐一击破，进行改正；如果大家暂时还没有碰到过类似情况，那么恭喜你，以后继续注意规避问题，保持好的工作状态吧~</p>
<ul>
<li>工作流程颠倒，进度停滞不前</li>
<li>产品定位和目标用户的临时变动</li>
<li>组员们各自忙碌，无法有效沟通</li>
<li>个人能力有限，需要历练提升</li>
</ul>
<h2 id="toc-1">一、工作流程颠倒，进度停滞不前</h2>
<p>在开启一个新的设计项目时，如果一开始没有做好每一步的规划，确定最终的设计目标，就很容易导致我们在日后的设计过程中不停返工，频繁地去“查漏补缺”。</p>
<p>笔者个人认为，导致返工的主要原因有两点：</p>
<ol>
<li>与其他team的合作计划有问题</li>
<li>设计师层面的必做工作没有计划好</li>
</ol>
<h3>1. 与其他team的合作工作</h3>
<p>当我们拿到一个新的设计项目时，首先应该明确我们需要哪些团队的协助来帮助我们顺利地推进接下来每一步的工作。</p>
<p>设计师是无法单打独斗的，我们不可避免地需要其他team的帮助，例如：</p>
<ul>
<li>规划团队——定义用户和产品定位，阐明期望的最终产品效果，明确项目节点</li>
<li>用研团队——针对目标用户进行痛点收集，分析真实诉求</li>
<li>人因团队——筛选研究资料，从同类产品中获得使用规范和可发展需求</li>
<li>开发团队——软件能力是否有限制，协助确定我们的设计目标需要达到的级别</li>
<li>……</li>
</ul>
<p>一般情况下，在项目初期我们就会跟规划团队进行一次沟通，由他们告诉我们：我们的用户是谁，我们需要向他们提供一个什么样的产品，我们有多长的时间来完成它。</p>
<p>这一点毋庸置疑是我们设计的“立足之本”。</p>
<p>规划团队所制定的初步计划有助于我们制定后续的设计计划，让我们对需要的外力协助有一个初步决定，根据项目节点进行安排。比如：</p>
<p><strong>做一个市面上已有的产品：</strong></p>
<ul>
<li>需要用研团队协助我们进行用户招募，搜集目标用户的痛点和需求；</li>
<li>咨询人因团队是否储备了相关研究资料，给设计师一些启示和灵感。</li>
</ul>
<p><strong>做一个从无到有的新产品：</strong></p>
<ul>
<li>需要规划团队首先阐明此产品的出发点是什么，它的潜在市场在哪里；</li>
<li>需要开发团队与设计师拉通能力储备，了解能做到什么程度的产品。</li>
</ul>
<p>当我们明确了需要和哪些团队建立合作关系时，需要尽早和他们进行一次有效沟通，明确各自的能力和需求，计划后续各自的任务和大家的沟通节点，并且保持联系，及时交换工作流程和偶发的变动。</p>
<h3>2. 设计师层面的必做工作</h3>
<p>相信大家已经对我们常用的设计方法（们）都十分熟悉了，其实各类设计工具书教给我们的设计方法都如出一辙。</p>
<p>但设计师在工作中一定要根据实际情况调整，对每一步工作进行合理分配，以免“顾此失彼”。</p>
<p><strong>Point 1</strong></p>
<p>如果一开始就知道有一些工作是其他team可以帮助你的，设计师大可以安排好合作计划以后去同步做一些其他工作，提升设计效率。</p>
<p>但注意了，前提一定是安排好。</p>
<p>比如用户调研，设计师可以提前跟用研team沟通好合作计划，规定目标用户的个数和用研时长，并在此过程中注意跟进进展。</p>
<p>在等待用研的工作结果的同时，设计师可以进行竞品分析和市场调研之类的工作，了解同类竞品，分析敌我优劣。</p>
<p>但是，若设计师觉得用研team的产出不足以支撑后续的设计工作，那么一定要及时自己出马！而且一定要做好计划，明确设计师可以有多少时间去做调研，到底想要了解一些什么内容，以免拖自己后腿，更不要拖团队后腿。</p>
<p>笔者实在是吃了很多次这样的亏，用研团队迟迟招募不到用户，又或者一个调研花费周期太长，结果都已经到了该进行v2的设计时才发现，用户数据不够啊，设计工作无法有效开展。</p>
<p>结果设计师又全员返工去做调研，觉得无奈却又不得不做。最后组员们的完成时间都不一样，又花了很多时间去等待，讨论，复盘…</p>
<p><strong>Point 2</strong></p>
<p>若时间紧张，无法让设计师顾全方方面面“大展拳脚”，则需要对手头的工作进行优先级排序。</p>
<p>例如，基于项目的全集推进计划，此时已经到了搭建设计框架这一步了，但实际上我们才刚刚完成用研，该怎么办？</p>
<p>笔者个人认为，此时我们重点需要且仅仅需要的东西，是“用户痛点”。只要我们将手中的用户痛点提炼出来，整理好并在团队内共享即可。</p>
<p>至于那些写在漂亮模版里的persona和user journey map，并不是说它们不重要，而是从大局观来看，我们“暂时不需要它们”。</p>
<p>大家都知道这两样设计工具的内容在制作和阅读方面都是相对比较复杂的，且每位设计师都会依据各自的特色进行调整，这对于后续的团队资源共享和复盘其实都无意中造成了一定的困难。在如此紧张的时刻我们需要抓住重点，追求效率，而不是“好看”。</p>
<p>笔者的团队曾经在一个项目中花了一个星期的时间等所有人做完persona和user journey map，然后又花了三天时间协调大家的时间安排会议，复盘每个人的persona和user journey map，然后又花了一两天时间让大家各自去调整……</p>
<p>一个本该在月底就完成的项目，还停留在用研总结阶段，整体计划又硬生生地往后推了两个月。</p>
<p>实话实说，我们在学校里以及在书籍上学到的设计工具的产出物，在实际工作中最终起到的作用，可能只是在项目完结汇报上给PPT增光添色。</p>
<p>以上就是笔者针对设计工作中的返工问题的一些经验和想法。希望对大家能有所帮助，下篇再来和大家聊聊剩下的几点。</p>
