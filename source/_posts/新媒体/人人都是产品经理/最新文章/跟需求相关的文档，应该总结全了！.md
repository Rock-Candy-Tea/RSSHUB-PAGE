<p><img data-action="zoom" class="size-full wp-image-5393805 aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2022/04/xkhC0zUXHjnqq003XBbv.jpg" alt width="900" height="420" referrerpolicy="no-referrer"></p>
<p>需求可以说是一个产品的重中之重，那么作为产品经理，你都产出过哪些跟需求相关的文档呢？今天我们就来总结一下！</p>
<h2 id="toc-1">一、经典三件套</h2>
<p>首先呢，跟需求相关的文档，那必然是经典的三件套了，也就是MRD/BRD/PRD，不管这三种文档你有没有写过，但是作为产品经理，你肯定听过！</p>
<p>我们再来依次介绍一下~</p>
<h3>1. MRD</h3>
<p>M是market的首字母，MRD可以说是需求产出的第一份文档。</p>
<p>也就是你想打造一款产品的话，那你得先研究一家伙整个市场环境吧，是红海还是蓝海，市场饱不饱和，有没有进入的机会，都有哪些客户，客户是否愿意买单，等等。</p>
<p>把这些落实到文档当中，那么一个MRD的目录大概会是这个样子的：</p>
<p>《MRD》</p>
<p>1. 市场的问题和机会</p>
<ul>
<li>（1）市场问题；</li>
<li>（2）市场机会；</li>
<li>（3）产品问题和机会；</li>
<li>（4）技术问题和机会；</li>
</ul>
<p>2. 市场概述</p>
<ul>
<li>（1）目标市场特征；</li>
<li>（2）目标市场趋势；</li>
<li>（3）目标市场细分；</li>
<li>（4）目标市场时间约束；</li>
</ul>
<p>3. 客户和购买者</p>
<ul>
<li>（1）业务决策购买者动机和目标；</li>
<li>（2）技术决策购买者动机和目标；</li>
</ul>
<p>4. 市场需求</p>
<ul>
<li>（1）市场需求概要表；</li>
<li>（2）方案概述；</li>
<li>（3）技术概述；</li>
</ul>
<h3>2. BRD</h3>
<p>MRD搞完以后呢，就得接着研究BRD了，B是business的首字母。</p>
<p>MRD是对整个市场的研究，比较宏观，而BRD则相对微观一些，就更加聚焦于产品未来的商业模式了，可以认为是一份商业计划书。</p>
<p>BRD的文档目录，大概是这个样子的：</p>
<p>《BRD》</p>
<p>1. 市场分析</p>
<ul>
<li>（1）产品的背景；</li>
<li>（2）产品的市场概述；</li>
<li>（3）产品的市场需求；</li>
<li>（4）产品的市场定位；</li>
</ul>
<p>2. 产品发展规划</p>
<ul>
<li>（1）产品的发展蓝图；</li>
<li>（2）产品的版本规划；</li>
<li>（3）里程碑规划；</li>
</ul>
<p>3. 产品研发规划</p>
<ul>
<li>（1）需求分析；</li>
<li>（2）产品设计；</li>
<li>（3）研发计划；</li>
</ul>
<p>4. 市场营销规划</p>
<ul>
<li>（1）产品上市计划；</li>
<li>（2）产品手册；</li>
<li>（3）市场推广；</li>
</ul>
<p>5. 可行性分析</p>
<ul>
<li>（1）产品竞争力分析；</li>
<li>（2）资源要求及保障能力；</li>
<li>（3）风险分析及对策；</li>
<li>（4）经济效益分析；</li>
</ul>
<p>大家可以从目录当中看得出来，BRD当中有很多内容，是可以借鉴MRD的~</p>
<h3>3. PRD</h3>
<p>PRD相信大家作为产品经理都不陌生，这是我们在搞产品设计时，交付给开发的，可以说是必备的内容。</p>
<p>而关于这块，我们公众号之前也发过很多，而且网上也有很多相关的资料，这个就不赘述了，有想了解的同学，可以查看我们的往期文章。</p>
<p>《PRD文档范例，产品经理值得收藏的写作手册！》</p>
<p>《永远经典的PRD规范！》</p>
<h2 id="toc-2">二、需求前期</h2>
<p>需求的前期，跟需求相关的模板，可以分为两种情况：</p>
<h3>1. 主动的需求收集</h3>
<p>这种情况大多是B端项目，你需要去向用户了解需求了，该怎么问呢？</p>
<p>问了以后，又该整理出一些什么东西呢？</p>
<p>这个时候，就得有一个需求收集的模板了，可以参考如下内容：</p>
<p>《需求收集文档》</p>
<p>1. 需求编号</p>
<p>2. 需求类型</p>
<p>（1）功能性需求；</p>
<p>（2）非功能性需求；</p>
<p>3. 需求来源</p>
<p>（1）用户角色；</p>
<p>（2）用户资料；</p>
<p>4. 需求场景</p>
<p>5. 需求描述</p>
<p>6. 需求原因</p>
<p>7. 验收标准</p>
<p>8. 需求重要性</p>
<h3>2. 被动的需求接收</h3>
<p>这种情况大多是C端产品，由运营等团队向产品提需求。</p>
<p>但不能让他们乱提吧，不能是口头说两句话，或者是钉钉上随便发个消息，这需求就提过来了？</p>
<p>那后期出了问题，锅是谁的？</p>
<p>所以说，接收需求的话，我们产品经理也是有原则的，不能啥客都接是吧！</p>
<p>需求接收的原则，可以参考如下内容：</p>
<p>《需求接收文档》</p>
<p>1. 基本信息</p>
<ul>
<li>（1）提出部门；</li>
<li>（2）提出人员；</li>
<li>（3）提出时间；</li>
<li>（4）期望完成时间；</li>
</ul>
<p>2. 需求信息</p>
<ul>
<li>（1）需求背景：要解决什么问题；</li>
<li>（2）需求描述：要什么功能；</li>
<li>（3）建议方案：建议怎么做；</li>
<li>（4）参考竞品：觉得谁做的好；</li>
<li>（5）优先级；</li>
</ul>
<h2 id="toc-3">三、需求中期</h2>
<p>前期是把各种需求收集过来，而中期就需要对需求进行管理了，这个时候，就需要一个需求池的模板啦。</p>
<p>其实需求池不需要什么高端的工具，大多数情况下，一个Excel表格足以。</p>
<p>《需求池》</p>
<ul>
<li>1. 需求名称</li>
<li>2. 需求描述</li>
<li>3. 所属模块</li>
<li>4. 子模块</li>
<li>5. 需求状态</li>
<li>6. 优先级</li>
<li>7. 提出人</li>
<li>8. 需求分类</li>
<li>9. 产品负责人</li>
<li>10. 提交时间</li>
<li>11. UI负责人</li>
<li>12. 预计完成时间</li>
<li>13. 技术负责人</li>
<li>14. 预计完成时间</li>
<li>15. 测试负责人</li>
<li>16. 预计完成时间</li>
<li>17. 预计上线时间</li>
<li>18. 实际上线时间</li>
</ul>
<h2 id="toc-4">四、需求后期</h2>
<p>需求后期，主要是需求变更。</p>
<p>当然，前期的需求调研与产品设计阶段如果搞得足够细致，那就能够最大程度地避免需求变更。</p>
<p>但需求变更，是不可避免的，因为前期的规划或者设计出来的东西，跟后期实际的东西，存在差异是很正常的。</p>
<p>我们不反对需求变更，我们需要避免的，是无意义的和反复的需求变更，这个时候，一个流程和需求变更的模板，就能够派上用场了。</p>
<p>《需求变更申请表》</p>
<p>1. 基本信息</p>
<ul>
<li>（1）项目名称；</li>
<li>（2）项目编号；</li>
<li>（3）申请人；</li>
<li>（4）申请日期；</li>
</ul>
<p>2. 变更需求</p>
<ul>
<li>（1）变更原因；</li>
<li>（2）变更内容；</li>
</ul>
<p>3. 变更影响</p>
<ul>
<li>（1）工作量；</li>
<li>（2）时间评估；</li>
<li>（3）其他影响；</li>
</ul>
<p>4. 审批意见</p>
<h2 id="toc-5">五、结语</h2>
