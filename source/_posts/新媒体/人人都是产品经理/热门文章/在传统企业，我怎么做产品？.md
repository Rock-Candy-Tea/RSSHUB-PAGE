<blockquote><p>编辑导语：不论是传统企业产品经理还是互联网产品经理，在做产品方面都有着共通之处，可以相互借鉴；产品经理要确认业务的方向，进行产品的设计；本文作者分享了关于传统企业做产品的思考和实例介绍，我们一起来了解一下。</p></blockquote>
<p><img data-action="zoom" class="aligncenter size-full wp-image-4461392" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/04/Ousm4hi6leGShgekpIYb.jpg" alt width="900" height="420" referrerpolicy="no-referrer"></p>
<p>去年中，我根据在公司出现并解决的一些实例写了一篇<a href="http://www.woshipm.com/pmd/4054767.html">在传统企业做互联网产品经理，我有几点思考 </a>；时隔近一年，我有收到一些类似背景的同学提出疑问，经过交流，回顾起过去的一些经历，不难发现尽管大家来自不同行业在做不同的产品，在解决一些具体事项的时候，有些思路方法是可以相同的。</p>
<p>灵感一来，我想带上3个新的实例以此展开来说说：在传统企业，我是如何做产品？同时，针对“做产品”，我带来的3个可探讨的实例是：</p>
<ul>
<li>产品方案：在传统企业，产品系统更倾向于解决哪方面的问题？</li>
<li>产品规划：公司想要尝试新业务方向，该如何设计方案正确落地？</li>
<li>需求博弈：如何有效拒绝业务部门的不合理需求？</li>
</ul>
<h2 id="toc-1">一、产品系统更倾向于解决哪方面的问题？</h2>
<p>我温馨奉劝各位：千万不要完美解决问题，尤其是发展阶段的企业，业务模式不稳定，面面俱到的系统容易埋雷。</p>
<p><strong>举个例子：商机（线索）分配机制、提高转化率。</strong></p>
<p>常见线索前后两端的部门领导唉声叹气：广告部门头痛获客成本高企，期望选择优质渠道提高回报；业务部门头痛手下员工转化差，家长式管理又担心人员流失。</p>
<p>于是，他们找到产品经理，期望系统能够提供一些数据或者一些功能能够帮助他们解决这些问题。</p>
<p>我们跟广告部门说：解铃还需系铃人，如果仅从头尾两侧提炼数据，远不足以支撑判断渠道是否优质，前期还需要更灵活可控：</p>
<ul>
<li>相同分类内的渠道之间做比较，将渠道分类：搜索类、信息流、电商类、自然流量、其他包括线下、公众号；</li>
<li>掌握不同分类的属性，拉取一定周期内的数据，观察时间段，会得出属性不同对应的峰值不同，决定投放时间段及阈值；</li>
<li>不同属性投放的广告关键词及宣传内容也需要灵活配置。</li>
</ul>
<p>前期基本上可以通过人工定量投放渠道和内容测试，但数据可以在系统做个简单的表格，方便拉取筛选得出结论，后续再考虑将人工部分实现系统配置。</p>
<p>转头，我们也跟业务部门了解到线索进来后是如何运转，线索进入系统后有通知及时提醒，所以重点不是跟进不及时，发现到问题：</p>
<ul>
<li>业务能了解到线索的信息太少了，以致于每个线索没有差异化的跟进，久而久之精力消耗过多，不愿接触新线索；</li>
<li>存量线索没有合适的方法跟进，仅仅依靠主观标签很难识别客户画像、潜在意向。</li>
</ul>
<p>除此，还有内部也存在矛盾的地方，业务管理者希望线索是被有效跟进的：谁能干就多干，线索不被浪费；业务人员却希望自己的线索越多越好：干不完没关系，放着就有机会被转化。</p>
<p>以此，系统前期不介入矛盾，基础建设先补充线索字段完善线索画像，在广告渠道端增加广告浏览时长用来评估客户意向；不同留存的跳转，比如：加社群、不感兴趣、直接拨打电话提供用户选择。</p>
<p>通过业务调研提炼出更客观标准的标签项，让业务经过前期交流即可操作勾选，后续再结合权重公式，算出客户等级，如：</p>
<ul>
<li>客户价格接受区间（免费至X万区间）；</li>
<li>客户期望达到的效果（无要求至有全方位的要求区间）；</li>
<li>客户对公司产品的了解程度（不了解至很熟悉程度）。</li>
</ul>
<p>客户等级的客观性在一定程度上解决了原有业务主观标签的问题，方便后续在公海被其他业务能手去“攻克”，另外也方便后续用户画像数据清洗。</p>
<p>综上，不难发现系统更显理智，不全将问题痛点部署到线上去解决，更倾向于解决可被标准化可被长期流转的问题。</p>
<h2 id="toc-2">二、公司尝试新业务方向，该如何设计方案正确落地？</h2>
<p>我的答案是：详细规划，内部达成共识一致的目标后再开工</p>
<p><strong>举个例子：</strong><strong>新业务是我产品生涯的盲区。</strong></p>
<p>我一直负责CRM系统产品设计，近来公司有新业务想要尝试，安排我负责初期规划，同时并行招聘新同事后续转职负责这条业务线。</p>
<p>我心里非常没底，因为我几乎从未接触过这条新业务——学龄在线教育，临时抱佛脚做了一番竞品分析、市场调研、行业了解，大致有了知识储备，我梳理了一套落地模式。</p>
<ul>
<li>点：场景中的重要角色有哪些？</li>
<li>A线：他们分别遇到什么问题？</li>
<li>B线：他们希望有什么解决方案？</li>
<li>面：技术可行性或运营分别可以怎么解决？</li>
</ul>
<p>目前新业务现处于敲定一期目标阶段，最后，我先献上梳理出的规划思路，如下：</p>
<p><img data-action="zoom" class="aligncenter" src="https://cors.zfour.workers.dev/?http://image.woshipm.com/wp-files/2021/04/yTPRfyGnLgARZ3TpcpkS.png" alt width="1364" height="544" referrerpolicy="no-referrer"></p>
<p>这仅是落地第一阶段，重点是如何说服领导及团队，难免遇到三连问：</p>
<ul>
<li>尝新的第一步要做什么？</li>
<li>目标是什么？如何衡量？</li>
<li>为什么是这样？</li>
</ul>
<p><strong>最后，我们拟定规划：</strong></p>
<ul>
<li>搭建免费知识平台，通过完整的知识体系解决了该行业人员服务水平参差不齐、用户信息壁垒的问题；</li>
<li>免费服务：我们分析这个行业多个竞品及案例提炼出了用户痛点设计成了一条标准化的流程，相对付费服务来说，颗粒较粗，但不影响用户服务的完整性；</li>
<li>付费增值服务：颗粒拆细到任何一个角度或阶段的用户都能找适合自己的服务，并且在某种程度上，我们并没有将付费入口直接开放，而是在免费体验过程中介入到用户需要引导或帮助的节点上。</li>
</ul>
<h2 id="toc-3">三、如何礼貌地拒绝业务部门的不合理需求？</h2>
<p>我在一年前一篇<a href="http://www.woshipm.com/pmd/4054767.html" target="_blank" rel="noopener">在传统企业做互联网产品经理，我有几点思考 </a>提到过引导需求的案例，至于直接拒绝需求，方法当然还是理智派的数据大法。</p>
<p><strong>举个例子：短信于精细化运营的关系。</strong></p>
<p>运营部门KPI是提高公海客户转化率，前期应运营方案针对这部分客户群体搭建了积分商城，长期实行推荐有礼效果平平，这回运营部门希望再拿这批公海客户出来发短信，我对发短信这项运营举措颇为反感，主要是体验极差：</p>
<ul>
<li>对客户无差异化发送相同内容的短信；</li>
<li>发送频率较高，试行每周每人1条。</li>
</ul>
<p>前期我没有拒绝发送短信“骚扰”客户这事，也想着让他们先试试看会有什么更好的效果；但后续我暗地里筛选数据发现并无效果，且同期运营同步的数据结论是：所有收到短信的客户已有业务跟进，后续转化签单即定义短信运营有效，明显结论是不合理的。</p>
<p>为了拒绝新的一批短信需求，我拉了以下数据：</p>
<ul>
<li>收到短信的客户分类：客户等级、客户已有积分；</li>
<li>客户收到短信后是否有登录积分商城；</li>
<li>客户签单的主要原因是什么。</li>
</ul>
<p>数据结果也与我预想一致：</p>
<ul>
<li>已签单的客户从未登陆过短信中积分商城的链接；</li>
<li>签单主要是依靠线下业务部门的推荐活动而非线上；</li>
<li>至于为什么将客户分类，而是告知运营目前积分存量对应能够对应什么礼品，将短信内容精细化，面对什么类型的客户传达不同的福利激励信号。</li>
</ul>
<p>最终，需求被有效拒绝，但事情变得有些令人不快，我被运营同事”反目成仇“了：因为某些被清理过的数据维度在系统中找不到，他们无法像我一样知情，当然对我来说也是站不住脚，毕竟我使用原始数据也只花了一个上午。</p>
<h2 id="toc-4">四、结语</h2>
<p>传统企业产品经理也好，互联网产品经理也好，两者在对应企业中发挥的价值有很多可以借鉴参考的经验，没有谁的方法一定就是对的。</p>
<p>我常常喜欢参加一些业务部门的会议，听他们讨论表达观点，业务流程中一些的方法如何融合到自己的产品方案中，可谓是一门手艺。</p>
<p>希望本篇文章可以给各位的工作带来一些帮助。</p>
