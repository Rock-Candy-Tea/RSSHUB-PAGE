<h2 data-id="heading-0">一、背景</h2>
<p>当前环境下，微信小程序成为了重要的活动运营渠道。为了满足页面高效率在小程序端、h5、app 端上线和迭代的需求，大量运营和活动页面在小程序端选择了 webview 中内嵌h5的形式。因此，腾讯课堂小程序实现了 common-act 页面作为通用的小程序 webview 承载页。</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/bc5b30dcc4ec4df9923a85b1ead96952~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>此页面先后承载了赠一得一、堂豆福利社（后迭代升级为 kbone 架构）、拉新红包、春节活动、三月大促等20+系列活动页面在微信小程序端的运营。</p>
<p>由于大部分活动都是 <strong>线上运营</strong> ，用户从主站跳转打开活动页时页面加载时长问题不明显。但在薪选活动的<strong>线下运营</strong>中，用户主要通过扫描小程序码启动小程序并打开活动页，整个启动较慢的问题十分凸显，迫切需要对通用活动页面的启动速度进行优化。</p>
<p>经过对小程序启动过程的分析，以及此 webview 页面较少依赖小程序公共方法和公共组件的特点，选择了对页面拆分独立分包的方式进行优化。活动页的独立分包相比于主包体积减小了93%。经过小程序性能测速监控对比发现，优化后用户启动页面的速度<strong>总体提升了 30%</strong>，高端机平均可以在 <strong>1s 内</strong>启动 common-act 页面。</p>
<h2 data-id="heading-1">二、小程序活动页启动过程分析</h2>
<h3 data-id="heading-2">2.1 common-act 页面是如何打开h5的</h3>
<p>作为承载 h5 的通用小程序页，commmon-act 负责处理不同形式的入参，得到业务需要的完整 url，并以 query 的形式向 web 页面注入用户票据。</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/76ddce89040f4497824916524ad92154~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>可以看到打开页面的整个过程是串行的，用户需要先从 common-act 页面启动小程序，页面 onLoad 阶段会通过参数生成 h5 链接并通过 setData 传输给视图层的 <code><webview /></code> 组件。在页面参数中已包含页面地址（不用从配置平台换取 url）的情况下，整个打开的过程约为 <strong>小程序启动时长</strong> + <strong>h5 加载时长。</strong></p>
<h3 data-id="heading-3">2.2  common-act 活动页启动的各阶段</h3>
<p>小程序启动指：用户点击小程序卡片/扫码/点击其他跳转链接呼起小程序到首屏 onReady 生命周期。</p>
<p>h5 加载指：webview 中开始准备打开新的 web 页面（navigationStart）到页面首屏的时间。
完整过程的各个阶段及对应页面表现如下。</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/e232f7f40b8248d48c0abf78632e2bd0~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<h3 data-id="heading-4">2.3 微信小程序启动和加载过程</h3>
<p>根据<a href="https://link.juejin.cn/?target=https%3A%2F%2Fdevelopers.weixin.qq.com%2Fminiprogram%2Fdev%2Fframework%2Fperformance%2Ftips%2Fstart.html" target="_blank" rel="nofollow noopener noreferrer" title="https://developers.weixin.qq.com/miniprogram/dev/framework/performance/tips/start.html" ref="nofollow noopener noreferrer">微信官方文档</a>，微信小程序启动过程包括以下几个部分：</p>
<ul>
<li>小程序资源准备</li>
<li>代码注入</li>
<li>首页渲染</li>
</ul>
<p>其中资源准备可以划分为<strong>小程序环境准备</strong>和<strong>代码包准备</strong>两个阶段。</p>
<p>在启动流程中，<strong>小程序代码包准备、开发者代码注入和首页渲染</strong>是与开发者相关的，开发者可以进行一定的优化工作。其他部分目前开发者暂时无法干预。</p>

<h1 data-id="heading-5">三、小程序活动页启动时长优化方案</h1>
<h3 data-id="heading-6">3.1 小程序活动页启动现状</h3>
<p>通过小程序管理后台启动数据可知，5月首周，小程序整体在<strong>全量更新</strong>情况下平均启动数据如图所示</p>






























<table><thead><tr><th align="left"></th><th>全部机型</th><th>低端机型</th></tr></thead><tbody><tr><td align="left">下载代码包</td><td>600ms</td><td>1000ms</td></tr><tr><td align="left">代码注入</td><td>270ms</td><td>700ms</td></tr><tr><td align="left">初次渲染</td><td>170ms</td><td>400ms</td></tr><tr><td align="left">总耗时</td><td>3900ms</td><td>7800ms</td></tr></tbody></table>
<p>但现阶段小程序管理后台缺少<strong>从不同页面启动</strong>的具体数据。</p>
<p>common-act 页面是普通分包，代码包准备和代码注入阶段都执行了完整的主包。
在加载和处理主包的过程中由于<strong>同步</strong>准备了很多核心页面依赖的基础信息和公共方法，耗时较长。但大部分的主包内容在活动页逻辑中是不依赖的。</p>
<p>因此可以从此角度进行优化。如果只下载、注入活动页需要的代码包将<strong>极大降低代码包准备和代码注入的时长</strong>。对于启动速度提升效果将比较明显。</p>
<h3 data-id="heading-7">3.2 独立分包方案</h3>
<p>独立分包是小程序中的特殊分包，可独立于主包和其他分包运行。从独立分包启动小程序时，不需要同步准备主包（从普通分包页面启动小程序需完整加载主包）。等用户进入普通分包或主包内页面时，主包才会被下载和解析。因此可以提高小程序的加载速度。（文档可见：<a href="https://link.juejin.cn/?target=https%3A%2F%2Fdevelopers.weixin.qq.com%2Fminiprogram%2Fdev%2Fframework%2Fsubpackages%2Findependent.html" target="_blank" rel="nofollow noopener noreferrer" title="https://developers.weixin.qq.com/miniprogram/dev/framework/subpackages/independent.html" ref="nofollow noopener noreferrer">独立分包</a>）</p>
<p>由于独立分包中不能依赖主包和其他分包中的内容，相关依赖需要全部位于页面文件夹中，分析页面涉及的公共依赖包括：
基础npm包依赖： @tencent/imwxutils（封装了一些小程序基础方法）、@tencent/imlog-wx（用来做日志上报）、simple-base-converter（用于压缩参数长度）</p>
<p>登陆相关： 判断登陆态、获取小程序登陆票据</p>
<p>路由相关： 页面跳转</p>
<p>请求相关：keFetch (业务封装的请求方法)</p>
<p>以上涉及容器页基础能力的依赖需要整理出来放到页面文件夹下，页面文件目录如下：</p>
<pre><code class="copyable">common-act/

├── common-act.wxml
├── node_modules/...

└── utils
    ├── const.ts
    ├── fn.ts
    ├── index.ts
    ├── performance.js
    └── type.ts
<span class="copy-code-btn">复制代码</span></code></pre>
<p>经过独立分包处理后，构建产物中 common-act 页面独立分包大小为<strong>134.92KB</strong>，相比于主包体积<strong>减少了93%</strong>。</p>
<h3 data-id="heading-8">3.3 小程序性能测速上报方案</h3>
<p>页面启动速度的优化需要有比较详细的启动数据作为依据，当前管理后台的性能数据还无法满足页面启动速度监控的需要。
小程序 <a href="https://link.juejin.cn/?target=https%3A%2F%2Fdevelopers.weixin.qq.com%2Fminiprogram%2Fdev%2Fapi%2Fopen-api%2Fperformance%2Fwx.getPerformance.html" target="_blank" rel="nofollow noopener noreferrer" title="https://developers.weixin.qq.com/miniprogram/dev/api/open-api/performance/wx.getPerformance.html" ref="nofollow noopener noreferrer">performance API</a> 可以返回以下性能指标：</p>
<ul>
<li>启动时长：小程序启动的时间，仅在启动页面抛出</li>
<li>代码注入时长：主包、分包注入时抛出</li>
<li>页面首次渲染时长：页面首屏渲染时长，页面打开时抛出</li>
<li>路由时长：页面跳转时抛出</li>
</ul>
<p>其余可能影响启动速度的因素和对应需要上报的数据包括：</p>
<ul>
<li>通过 getSystemInfo 返回机型、基础库信息</li>
<li>通过 getNetworkType 返回网络类型、信号强度</li>
<li>通过 getAccountInfoSync 返回小程序版本号</li>
<li>同时上报页面路径和页面参数作为日志分析的参考信息</li>
</ul>
<p>基于这些数据，imlog 日志上报字段如下</p>

























<table><thead><tr><th align="left">project</th><th>message</th><th>cmd</th><th>pathname</th><th>cost_time</th><th>addition</th><th>platform</th><th>query</th></tr></thead><tbody><tr><td align="left">weapp-ke</td><td>performance</td><td></td><td></td><td></td><td></td><td></td><td></td></tr></tbody></table>
<ul>
<li>通过 project 和 message 字段能定位到小程序性能上报数据</li>
<li>cmd 的值可能为 appLaunch、firstRender、route、evaluateSript</li>
<li>pathname 标识性能记录产生的页面，如 pages/common-act/common-act</li>
<li>cost_time 表示此次 cmd 表示行为的耗时</li>
<li>addtion 记录了此次性能记录的其他相关信息，包括此次行为开始时间 startTime、机型、微信版本、基础库版本、网络类型、信号强度。（由于 firstRender、evaluateSript 记录在启动和页面跳转时都会产生，addtion 中用 isLaunch 标识此条上报是启动时的行为记录。）</li>
<li>platform 包含小程序正式版版本信息。用于比较不同发布版本的表现。</li>
<li>query 为页面参数，用于附加判断页面信息。</li>
</ul>
<h2 data-id="heading-9">四、独立分包优化效果分析</h2>

</ul>

<p>从优化效果上看，低端机的启动时长在<strong>绝对数值上减少的更多</strong>。但由于高端机在小程序环境准备、代码注入、页面渲染的时间更短，独立分包提升<strong>比例更大</strong>。低端机相对而言独立分包提升比例更有限一些。</p>
<p>#五、优化总结</p>
<ol>
<li>common-act 独立分包的优化方案对小程序 common-act 页启动的提升效果较明显，总体上看，启动时长的中位数由 3.2s 减少至 2.2s（高端机总体启动速度 1s 内），从活动页启动速度提升 30%。</li>
<li>采用独立分包后，common-act 页面启动长时间等待的比例明显下降，3s 内完成启动的用户比例由 44% 提升至 70%。对用户体验提升明显。</li>
<li>小程序启动速度的主要影响因素为用户的移动设备性能，性能好的机型平均启动时长小于1s。</li>
<li>启动过程中，开发者仅能够从代码层面影响代码包准备和代码注入的时长。其余<strong>微信底层的环境准备和处理时长对整体启动时长的影响更大</strong>，且直接依赖于用户的设备性能和网络情况，开发者可以优化的空间有限。</li>
</ol>
