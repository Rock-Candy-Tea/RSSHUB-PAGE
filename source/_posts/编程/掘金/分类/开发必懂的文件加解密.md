<h2 data-id="heading-0">背景</h2>
<p>最近团队遇到一个小需求，存在两个系统 A、B，系统 A 支持用户在线制作皮肤包，制作后的皮肤包用户可以下载后，导入到另外的系统 B 上。皮肤包本身的其实就是一个 zip 压缩包，系统 B 接收到压缩包后，<strong>解压</strong>并做一些常规的校验，比如版本、内容合法性校验等，整体功能也比较简单。</p>
<p>但没想到啊，一帮测试人员对我们开发人员一顿输出，首先绕过系统 A 搞了几个视频文件，把后缀改成 <code>zip</code> 就直接想上传，系统 B 每次都是等到上传完后才发现文件不合法，系统 B 在文件没上传完前又无法解压，也不知道文件内容是不是合法的，就这么消耗了大量带宽、大量时间后才提示用户皮肤包有问题。</p>
<p>这里涉及了两个问题，我们来捋一捋：</p>
<ol>
<li>文件如何做加密，这样用户便无法去逆向，压缩包内部的敏感信息不会泄露出去。</li>
<li>服务端在接收到信息流时，在未传输完时如何去判断压缩包的合法性，提前告知用户。</li>
</ol>
<h2 data-id="heading-1">AES VS RSA</h2>
<p>说到加密，自己很多人会想到<a href="https://link.juejin.cn/?target=https%3A%2F%2Fen.wikipedia.org%2Fwiki%2FAdvanced_Encryption_Standard" target="_blank" rel="nofollow noopener noreferrer" title="https://en.wikipedia.org/wiki/Advanced_Encryption_Standard" ref="nofollow noopener noreferrer">对称算法 AES</a> 以及<a href="https://link.juejin.cn/?target=https%3A%2F%2Fen.wikipedia.org%2Fwiki%2FRSA_(cryptosystem)" target="_blank" rel="nofollow noopener noreferrer" title="https://en.wikipedia.org/wiki/RSA_(cryptosystem)" ref="nofollow noopener noreferrer">非对称算法 RSA</a>。这两种算法按字面意思也较好理解，对称加密技术说白一点就是加密跟解密使用的是同一个密钥，这种加密算法速度极快，安全级别高，加密前后的大小一致；非对称加密技术则有<code>公钥PK</code>、<code>私钥SK</code>，算法的原理在于寻找两个素数，让他们的乘积刚好等于一个约定的数字，非对称算法的安全性是依赖于大数的分解，这个目前没有理论支持可以快速破解，它的安全性完全依赖于这个密钥的长度，一般用 1024 位已经足够使用。但是它的速度相比对称算法慢得多，一般仅用于少量数据的加密，<strong>待加密的数据长度不能超过密钥的长度</strong>。</p>
<h2 data-id="heading-2">使用 AES 对文件加密</h2>
<p>结合这两种加密方式的优缺点，我们采用 AES 对文件本身做加解密，使用 AES 的原因主要考虑如下：</p>
<ol>
<li>加解密性能问题，AES 的速度极快，相比 RSA 有 1000 倍以上提升。</li>
<li>RSA 对源文有长度的要求，最大长度仅有密钥长度。</li>
</ol>
<p>AES 的加密算法 Node.js 的<a href="https://link.juejin.cn/?target=http%3A%2F%2Fnodejs.cn%2Fapi%2Fcrypto.html" target="_blank" rel="nofollow noopener noreferrer" title="http://nodejs.cn/api/crypto.html" ref="nofollow noopener noreferrer">crypto</a>模块中已经有内置，具体的使用可以参考官方文档。</p>

<h3 data-id="heading-5">AES 具体使用</h3>
<p>有了上述两个接口后，我们便可实现一个简单的对称加密了：</p>
<pre><code class="hljs language-javascript copyable" lang="javascript"><span class="hljs-keyword">const</span> key = <span class="hljs-string">'abcdefghijklmnopqrstuvwxyz123456'</span>; <span class="hljs-comment">// 32 共享密钥，长度跟算法需要匹配上</span>
<span class="hljs-keyword">const</span> iv = <span class="hljs-string">'abcdefghijklmnop'</span>;  <span class="hljs-comment">// 16 初始向量，长度跟算法需要匹配上</span>
<span class="hljs-keyword">let</span> fileBuffer = Buffer.from(<span class="hljs-string">'abc'</span>);

<span class="hljs-comment">// 加密</span>
<span class="hljs-keyword">let</span> encrypted = aesEncrypt(fileBuffer, key, iv);

<span class="hljs-comment">// 解密</span>
<span class="hljs-keyword">let</span> context = aesDecrypt(encrypted);
<span class="hljs-built_in">console</span>.log(context.toString());

<p>一般情况下，这个密钥较为重要，如果发生泄露则加密失去意义，所以<code>key</code>、<code>iv</code>会使用随机数动态生成，比如：</p>
<pre><code class="hljs language-javascript copyable" lang="javascript"><span class="hljs-keyword">const</span> key = crypto.randomBytes(<span class="hljs-number">32</span>);
<span class="hljs-keyword">const</span> iv = crypto.randomBytes(<span class="hljs-number">16</span>);
<span class="copy-code-btn">复制代码</span></code></pre>
<p>通过上述的调整后，加解密文件是比较容易的，回到我们的业务系统上面，系统 A 生成的压缩包，最终是需要给系统 B 使用，两个系统是隔离的，那这样 <code>key</code>、<code>iv</code> 如何传输到系统 B 上面呢，况且还是动态生成的，生成出来 <code>key</code> 系统 B 是不知道的。</p>
<p>读到这聪明的你可能会想到，在把<strong>压缩包给到 B 的时候，顺便把 <code>key</code>、<code>iv</code> 一同提交过去不就可以了</strong>，但细想了下，这个肯定不能明文把这个密钥发送过去，要不这个加密意义何在。</p>
<p>这时便需要用上 <strong>RSA 非对称加密技术</strong>了。</p>
<h2 data-id="heading-6">使用 RSA 算法对密钥再次进行非对称加密</h2>
<p>RSA 的加密算法 Node.js 的 <a href="https://link.juejin.cn/?target=http%3A%2F%2Fnodejs.cn%2Fapi%2Fcrypto.html" target="_blank" rel="nofollow noopener noreferrer" title="http://nodejs.cn/api/crypto.html" ref="nofollow noopener noreferrer">crypto 模块</a> 中已经有内置，具体的使用可以参考官方文档。</p>
<h3 data-id="heading-7">生成 RSA 的公钥与私钥</h3>
<p>使用 <a href="https://link.juejin.cn/?target=https%3A%2F%2Fwww.openssl.org%2F" target="_blank" rel="nofollow noopener noreferrer" title="https://www.openssl.org/" ref="nofollow noopener noreferrer">openssl</a> 组件可以直接生成 <code>RSA</code> 的公钥私钥对，具体的命令可以参考：<a href="https://link.juejin.cn/?target=https%3A%2F%2Fwww.scottbrady91.com%2FOpenSSL%2FCreating-RSA-Keys-using-OpenSSL" target="_blank" rel="nofollow noopener noreferrer" title="https://www.scottbrady91.com/OpenSSL/Creating-RSA-Keys-using-OpenSSL" ref="nofollow noopener noreferrer">www.scottbrady91.com/OpenSSL/Cre…</a>。</p>
<pre><code class="hljs language-bash copyable" lang="bash"><span class="hljs-comment"># 生成私钥</span>

<span class="hljs-comment"># 提取公钥</span>

<h3 data-id="heading-10">RSA 具体使用</h3>
<p>有了上述接口后，便可对 AES 的密钥进行加密后再传输，服务器 B 保存好 <code>RSA 私钥</code> ，服务器 A 则可以直接用 <code>RSA 公钥</code> 对数据加密后再发送，结合刚 <code>AES</code> 的逻辑后，如下：</p>

<p>这样结合在一起后，服务器 A 生成的压缩包，只要包含好 <code>&#123;file, key&#125;</code> 这两块内容，服务器 B 便可把文件解密出来了，这样基本上实现了我们第一点的目标：<strong>1. 文件如何做加密，这样用户便无法去逆向，压缩包内部的敏感信息不会泄露出去</strong></p>
<p>但还遗留了另外一个问题需要解决：<strong>2. 服务端在接收到信息流时，在未传输完时如何去判断压缩包的合法性，提前告知用户</strong></p>
<h2 data-id="heading-11">优化加密文件</h2>
<p>按上面的加密方式，输出的结果是一个 <code>buffer文件</code> 内容，以及一个 <code>加密过的key</code>，除了这些信息外，一般这个 <code>buffer文件</code> 压缩包还会有一些额外的信息，比如：版本号、压缩包生成时间，描述信息等。这些信息按常规的方式，可能是分成几个文件，然后再打一个压缩包把文件放在一起，比如：</p>
<pre><code class="hljs language-text copyable" lang="text">// zip file
- pkg
    manifest.json       // 额外的信息
    key.json            // 保存了加密过的密钥
    file.json           // 加密过的文件
<span class="copy-code-btn">复制代码</span></code></pre>
<p>但如果用这种方式保存，一般情况下还要对这个 <code>zip文件</code> 做下加密，然后改下后缀名，但是服务器 B 在读取这个文件后仍然是需要全部接收，再解压到临时目录，读取内容后才可以做校验，这样问题仍然解决不了。</p>
<p>除此之外，还有另外一个常见的需求，产品一般希望在浏览器侧在文件上传时就先做初步的解析，把明显不合法的文件提示到用户，这样用户体验更好。</p>
<p>这个问题的解决方案也不难，这些所有额外的信息都是可以把它当成二进制插入到文件的头部上的，比如：</p>
<pre><code class="hljs language-text copyable" lang="text">包字段描述：|----插入的额外信息----|----后面才是真正的文件内容----|  
二进制文件：010101010101010101010xxxxxxxxxxxxxxxxxxxxxxxxxxxx

<h3 data-id="heading-12">文件头字段设计</h3>
<p>我们把这些所有信息，按一定的格式，使用二进制的方式全部串连在一起，最终交付的只有一个组合过的文件，比如：</p>
<pre><code class="hljs language-text copyable" lang="text">// theme pkg.

0                8                16                 
|------flag------|--extra length--|
|----------extra data...----------|
|-------------data...-------------|
<span class="copy-code-btn">复制代码</span></code></pre>
<ul>
<li><code>flag</code>：</li>
</ul>
<p>固定标识 <code>THEME</code>，长度：<code>8 byte</code>，说明该压缩包为一个皮肤包，这样可以快速对压缩包进行识别</p>
<ul>
<li><code>extra length</code>：</li>
</ul>
<p><code>extra data</code> 的真实长度，这是一个 16 进制的数据，长度：<code>8 byte</code>，说明插入的数据长度。比如：长度 <code>35</code> 的数据，转化为 16 进制后为 <code>0x23</code>，那这字段为 <code>00000023</code></p>
<ul>
<li><code>extra data</code>：</li>
</ul>
<p>使用 <code>RSA</code> 加密过的数据，我们可以把上述需要用 <code>RSA</code> 加密的信息全部放在这里，比如 <code>key</code> 字段、版本号、描述信息等</p>
<ul>
<li><code>data</code>：</li>
</ul>
<p>使用 <code>AES</code> 加密过的数据，可以通过 <code>extra data</code> 里面保存的 <code>key</code> 把真实的数据全部解密出来</p>

<h2 data-id="heading-16">结语</h2>
<p>文件的加密、解密在后端其实是一个很常规的操作，除了上面聊到的 <code>AES</code>、<code>RSA</code>，其实还有其它很多加密方案，具体可以看看 <a href="https://link.juejin.cn/?target=http%3A%2F%2Fnodejs.cn%2Fapi%2Fcrypto.html" target="_blank" rel="nofollow noopener noreferrer" title="http://nodejs.cn/api/crypto.html" ref="nofollow noopener noreferrer">Node.js crypto 模块</a>，已经有内置比较多的方案可以直接使用。</p>
<p>当然文件的加解密，也可以直接用 <code>zip</code>、<code>7z</code> 等这些压缩工具，再配合密码的方案，一般情况也是够用的，但是免不了有定制化的需求，一般也都是结合使用，比如上面的 <code>fileBuffer</code> 实际内部就是先用这些工具对文件进行了压缩并加密。还是以场景为重，多种方案结合效果更好。</p>
