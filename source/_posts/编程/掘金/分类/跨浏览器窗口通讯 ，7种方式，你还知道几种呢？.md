<h2 data-id="heading-0">前言</h2>
<p>为什么会扯到这个话题，最初是源于听 <a href="https://link.juejin.cn/?target=https%3A%2F%2Fy.qq.com%2F" target="_blank" rel="nofollow noopener noreferrer" title="https://y.qq.com/" ref="nofollow noopener noreferrer">y.qq.com/</a> QQ音乐，</p>
<ul>
<li>播放器处于单独的一个页面</li>
<li>当你在另外的一个页面搜索到你满意的歌曲的时候，点击播放或添加到播放队列</li>
<li>你会发现，播放器页面做出了响应的响应</li>
</ul>
<p>这里我又联想到了商城的购物车的场景，体验确实有提升。<br>
刚开始，我怀疑的是Web Socket作妖，结果通过分析网络请求和看源码，并没有。 最后发现是localStore的storage事件作妖，哈哈。</p>
<hr>
<p>回归正题，其实在一般正常的知识储备的情况下，我们会想到哪些方案呢？</p>
<p><strong>先抛开如下方式：</strong></p>
<ol>
<li>各自对服务器进行轮询或者长轮询</li>
<li>同源策略下，一方是另一方的 <code>opener</code></li>
</ol>

<h2 data-id="heading-2">两个浏览器窗口间通信</h2>
<h3 data-id="heading-3">WebSocket</h3>
<p>这个没有太多解释，WebSocket 是 HTML5 开始提供的一种在单个 TCP 连接上进行全双工通讯的协议。当然是有代价的，需要服务器来支持。<br>
js语言，现在比较成熟稳定当然是 <a href="https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Fsocketio%2Fsocket.io" target="_blank" rel="nofollow noopener noreferrer" title="https://github.com/socketio/socket.io" ref="nofollow noopener noreferrer">socket.io</a>和<a href="https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Fwebsockets%2Fws" target="_blank" rel="nofollow noopener noreferrer" title="https://github.com/websockets/ws" ref="nofollow noopener noreferrer">ws</a>. 也还有轻量级的<a href="https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2FClusterWS%2FClusterWS" target="_blank" rel="nofollow noopener noreferrer" title="https://github.com/ClusterWS/ClusterWS" ref="nofollow noopener noreferrer">ClusterWS</a>。</p>
<p>你可以在<a href="https://link.juejin.cn/?target=https%3A%2F%2Fdeveloper.mozilla.org%2Fen-US%2Fdocs%2FWeb%2FAPI%2FWebSockets_API" target="_blank" rel="nofollow noopener noreferrer" title="https://developer.mozilla.org/en-US/docs/Web/API/WebSockets_API" ref="nofollow noopener noreferrer">The WebSocket API (WebSockets)
</a>看到更多的关于Web Socket的信息。</p>
<h3 data-id="heading-4">定时器 + 客户端存储</h3>
<p>定时器：setTimeout/setInterval/requestAnimationFrame<br>
客户端存储： cookie/localStorage/sessionStorage/indexDB/chrome的FileSystem</p>
<p>定时器没啥好说的，关于客户端存储。</p>
<ul>
<li>cookie: 每次会带到服务端，并且能存的并不大，4kb?，记得不太清楚</li>
<li>localStorage/sessionStorage 应该是5MB, sessionStorage关闭浏览器就和你说拜拜。</li>
<li>indexDB 这玩意就强大了，不过读取都是异步的，还能存 Blob文件，真的是很high。</li>
<li>chrome的FileSystem ,<a href="https://link.juejin.cn/?target=https%3A%2F%2Fcaniuse.com%2F%23search%3DfileSystem" target="_blank" rel="nofollow noopener noreferrer" title="https://caniuse.com/#search=fileSystem" ref="nofollow noopener noreferrer">Filesystem & FileWriter API</a>,主要是chrome和opera支持。这玩意就是文件系统。</li>
</ul>
<h3 data-id="heading-5">postMessage</h3>
<p><a href="https://link.juejin.cn/?target=https%3A%2F%2Fcaniuse.com%2F%23search%3DpostMessage" target="_blank" rel="nofollow noopener noreferrer" title="https://caniuse.com/#search=postMessage" ref="nofollow noopener noreferrer">Cross-document messaging</a> 这玩意的支持率98.9%。 好像还能发送文件，哈哈，强大。<br>
不过仔细一看 window.postMessage()，就注定了你首先得拿到window这个对象。 也注定他使用的限制， 两个窗体必须建立起联系。 常见建立联系的方式：</p>
<ul>
<li>window.open</li>
<li>window.opener</li>
<li>iframe</li>
</ul>
<p><strong>提到上面的window.open,  open后你能获得被打开窗体的句柄，当然也可以直接操作窗体了。</strong></p>
<hr>
<p>到这里，我觉得一般的前端人员能想到的比较正经的方案应该是上面三种啦。<br>
当然，我们接下来说说可能不是那么常见的另外三种方式。</p>

<ul>
<li>key ：代表属性名发生变化.当被clear()方法清除之后所有属性名变为null</li>

<li>storageArea：被操作的storage对象</li>
<li>url：key发生改变的对象所在文档的URL地址</li>
</ul>
