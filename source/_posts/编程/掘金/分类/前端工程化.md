<h1 data-id="heading-0">前端工程化</h1>
<ul>
<li>工程化解决的主要问题是什么？ 技术是为了解决问题而存在的
<ul>
<li>想要使用es+新特性，提高编码效率， 但是兼容有问题</li>
<li>想要使用less/sass/postcss 增强css的编程性，但是运行环境不能直接支持。</li>
</ul>
</li>
<li>想要使用模块化的方式提高项目的可维护性，但运行环境并不能直接支持。</li>
<li>部署上线前需要手动压缩代码及资源文件</li>
<li>部署过程需要手动上传代码传到服务器</li>
<li>多人协同开发，无法硬性统一大家的代码风格，从仓库中pull回来的代码质量也没有办法保证。</li>
<li>部分功能接口开发时需要等待后端服务接口提前完成。</li>
</ul>
<p>总结如下：</p>
<pre><code class="hljs language-js copyable" lang="js"><span class="hljs-number">1.</span> 传统语言的或语法的弊端
<span class="hljs-number">2.</span> 无法使用模块化/组件化
<span class="hljs-number">3.</span> 重复的机械式工作
<span class="hljs-number">4.</span> 代码风格统一。质量保障
<span class="hljs-number">5.</span> 依赖后端服务接口支持
<span class="hljs-number">6.</span> 整体依赖后端项目
<span class="copy-code-btn">复制代码</span></code></pre>
<p>一句话总结： 解决问题，提高效率</p>
<h4 data-id="heading-1">工程化</h4>
<blockquote>
<p>一切重复的工作都应该被自动化</p>
<p>或者说： 对项目的规划和架构</p>
</blockquote>
<pre><code class="hljs language-js copyable" lang="js">创建项目 --> 编码 --> 预览/测试 --> 提交 --> 部署 
创建项目： 创建项目结构  创建特定类型文件
编码： 格式化代码。校验代码风格。使用新特性。编译/构建/打包
预览/测试： web Server/Mock live_reloading/HRM  SourceMap(开发代码和运行代码的连接工具)
提交： git hooks(质量和风格和提交日志) / lint-staged / 持续集成
部署： CI/CD 自动发布
<span class="copy-code-btn">复制代码</span></code></pre>

<h5 data-id="heading-13">gulp的组合任务</h5>
<h5 data-id="heading-14">gulp的异步任务的三种方式</h5>
<h5 data-id="heading-15">构建过程的的核心原理</h5>
<ul>
<li>
<p>将文件读出来进行一些加工转换，最后写入到另外一个位置</p>
</li>

<p>没有构建系统的情况下， 我们都是人工那么做的。</p>

</li>
</ul>
<p>Gulp —————— The streaming build system</p>
<p>官方定义： 基于流的构建系统， gulp希望实现构建管道的概念。能实现更多的扩展。</p>
<h5 data-id="heading-16">gulp的文件操作api 和 插件的使用</h5>
