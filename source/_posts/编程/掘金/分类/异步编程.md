h2&#123;font-size:20px&#125;.markdown-body h3&#123;font-size:18px&#125;&#125;</style><h3 data-id="heading-0">异步编程的由来</h3>
<p>JavaScript语言的执行环境是的是单线程的，一次只能执行一个任务，多个任务需要排队等候，这种模式会出现阻塞代码，代码的执行顺序比较混乱，导致代码执行效率低下，，会造成页面卡顿，影响用户的体验度；为了避免这个问题，提出了异步编程；异步编程的方式有：<strong>callback回调函数</strong>、<strong>事件监听</strong>、<strong>promise</strong> 等；</p>
<h3 data-id="heading-1">JavaScript执行模式</h3>
<ul>
<li>同步模式
<ul>
<li>javascript代码是 <strong>依次执行</strong> ，后面代码要等待前面代码执行完毕后，才能执行；javascript代码大多数是以 <strong>同步模式</strong> 进行执行的</li>
</ul>
</li>
<li>异步模式
<ul>
<li>javascript代码 <strong>不用等待</strong> 前面的代码执行完毕才开始执行，</li>
<li>我们将执行的代码放入调用栈中执行，如果是同步的直接执行，如果是异步代码放入Queue中等后同步代码执行完毕；</li>
<li>EventLoop它会 <strong>监听调用栈</strong> 和 <strong>Queue</strong> 中的任务</li>
<li>当调用栈中所有的任务结束以后，它会从Queue中取出一个任务放进调用栈中，开始执行，等待所有循环结束</li>
</ul>
</li>
</ul>
<h3 data-id="heading-2">异步环境组成：</h3>
<p>web APIs环境， Queue消息队列，Call stack调用栈， Console打印 ,EventLoop：负责监听call stack 和 Queue</p>
<ul>
<li>消息队列（Queue）: Queue是只暂时存储异步任务的地方</li>
<li>宏任务
<ul>
<li>当前调用栈中执行的代码成为宏任务；浏览器为了让JS内部宏任务与DOM操作能有有序的执行，会在一个宏任务执行后，在下一个宏任务执行开始前，对页面进行重新渲染；</li>
<li>宏任务包含：setTimeout、setInterval、I/O、UI交互事件</li>
</ul>
</li>
<li>微任务
<ul>
<li>当前任务执行结束后需要立即执行的任务，不用进入Queue等候执行，它是在页面渲染之前执行，速度相比宏任务更快；</li>
<li>微任务包含：Promise、MutaionObserver、process.nextTick(‘Node.js’）环境)等</li>
</ul>
</li>
</ul>
<blockquote>
<p>浏览器执行引擎先去执行Call stack的任务，通过Event Loop从queue中去一个任务出来去执行。整个任务过程中，我们随时可以忘消息队列中放入一些任务，这些任务会Queue中排队等候循环。</p>
</blockquote>
<p><img src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/81c7290eb3f04f9eba0f0051c5de9c11~tplv-k3u1fbpfcp-watermark.image" alt="image.png" loading="lazy" referrerpolicy="no-referrer"></p>
<ul>
<li>回调函数：所有异步编程方案的根基（可以理解为一件你想要做的事情），有调用者定义，交给执行者执行的函数，称之为回调函数；</li>
</ul>
<h3 data-id="heading-3"><strong>Promise</strong>：</h3>
<p>CommonJS社区提出了Promise规范。解决回调 “<strong>地狱</strong>”，尽可能将原来嵌套的回调变为扁平化处理。</p>
<p><img src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/09a0aa5e0b6b47299a9fc5c1c2888d58~tplv-k3u1fbpfcp-watermark.image" alt="image.png" loading="lazy" referrerpolicy="no-referrer">
Promsie常见误区：</p>
<blockquote>
<p>嵌套使用的方式是promise的错误使用</p>
</blockquote>
<p><img src="https://p1-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/1e72f9c809d342c8b7a4e888471caa29~tplv-k3u1fbpfcp-watermark.image" alt="image.png" loading="lazy" referrerpolicy="no-referrer"></p>
<blockquote>
<p>借助于promise then方法链式调用的特点，尽量去保证异步任务的扁平化。</p>
</blockquote>
<p>每一个then方法实际上是为上一个then返回的promise对象添加状态明确过后的回调，那这些promise会依次执行。而且也可能在promise的对象中手动回调。</p>
<ul>
<li>promise对象的then方法会返回一个全新的promise对象</li>
<li>后面的then方法就是为上一个then方法返回的promise注册回调</li>
<li>前面then方法中回调函数的返回值会作为后面then方法回调的参数</li>
<li>如果回调中返回的是promise，那后面then方法的回调会等待它的结束</li>
</ul>
<h3 data-id="heading-4">promise异常处理</h3>
<p>catch方法注册失败回调
<img src="https://p9-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/f9b2bb316d674c129745947f4a0f1de9~tplv-k3u1fbpfcp-watermark.image" alt="image.png" loading="lazy" referrerpolicy="no-referrer"></p>
<ul>
<li>promise失败或出现异常会执行onRejected注册的回调</li>
<li>then方法的第二个参数，缺点：无法捕获到本轮then方法以及下层的异常</li>
<li>catch方法：推荐使用</li>
</ul>
<h3 data-id="heading-5">promise静态方法</h3>
<p><img src="https://p1-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/90903fb6d7524597bdc222b20a16d618~tplv-k3u1fbpfcp-watermark.image" alt="image.png" loading="lazy" referrerpolicy="no-referrer">
有时需要将现有对象转为 Promise 对象，Promise.resolve()方法就起到这个作用 Promise.resolve()</p>
<pre><code class="copyable">Promise.resolve('foo')
// 等价于
new Promise(resolve => resolve('foo'))
<span class="copy-code-btn">复制代码</span></code></pre>
<ul>
<li>可以将普通值快速转换为promise对象</li>
<li>如果传递的为promise，会将这个promise原样返回</li>
<li>如果传递的是一个对象，并且带有then方法，第一个参数为onFulfilled,第二个参数为onRejected，也可以被当做promise继续向下执行。这种对象实现了thenable的接口。</li>
</ul>
<h3 data-id="heading-6">promise并行执行</h3>
<ul>
<li>promise.all() 等待所有任务结束才结束</li>
<li>promise.race()只会等待第一个结束的任务</li>
</ul>
<h3 data-id="heading-7">promise执行时序</h3>
<ul>
<li>微任务
提高整体的响应能力，不用等候</li>
<li>宏任务：回调对了（Queue）中的任务称之为【宏任务】，在宏任务执行过程中可以临时加上一些额外需求，额外需求可以选择作为一个新的宏任务进到队列中排队。也可以作为当前任务的【微任务】，直接在当前任务结束过后立即执行不同排队。</li>
</ul>
<blockquote>
<p>特殊：promise的回调会作为微任务执行，MutationObserver和process.nextTick作为微任务</p>
</blockquote>
