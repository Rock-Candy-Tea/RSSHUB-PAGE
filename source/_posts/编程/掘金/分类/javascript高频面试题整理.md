<h2 data-id="heading-8">作用域&作用域链</h2>
<ul>
<li>简单的说，作用域就是变量与函数的可访问范围，即作用域控制着变量与函数的可见性和生命周期。作用域是一套规则，在当前作用域以及嵌套的子作用域中根据标识符名称进行变量查找。</li>
<li>作用域链的作用是保证执行环境里有权访问的变量和函数是有序的，作用域链的变量只能向上访问，变量访问到window对象即被终止，作用域链向下访问变量是不被允许的。</li>
</ul>
<h2 data-id="heading-9">new操作符具体干了什么</h2>
<ul>
<li>创建一个空对象</li>
<li>继承了该函数的原型</li>
<li>属性和方法被加入到 this 引用的对象中</li>
<li>新创建的对象由 this 所引用，并且最后隐式的返回 this</li>
</ul>
<h2 data-id="heading-10">严格模式的限制</h2>
<ol>
<li>变量必须声明后再使用</li>
<li>函数的参数不能有同名属性，否则报错</li>
<li>不能使用with语句</li>
<li>不能对只读属性赋值，否则报错</li>
<li>不能使用前缀0表示八进制数，否则报错</li>
<li>不能删除不可删除的属性，否则报错</li>
<li>不能删除变量delete prop，会报错，只能删除属性delete global[prop]</li>
<li>eval不会在它的外层作用域引入变量</li>
<li>eval和arguments不能被重新赋值</li>
<li>arguments不会自动反映函数参数的变化</li>
<li>不能使用arguments.callee</li>
<li>不能使用arguments.caller</li>
<li>禁止this指向全局对象</li>
<li>不能使用fn.caller和fn.arguments获取函数调用的堆栈</li>
<li>增加了保留字（比如protected、static和interface）</li>
</ol>

<h2 data-id="heading-12">面向对象编程 & 面向过程编程</h2>
<p><strong>概念</strong>：</p>
<ul>
<li>面向过程就是分析出解决问题所需要的<strong>步骤</strong>，然后用函数把这些步骤一步一步实现，使用的时候一个一个依次调用就可以了；</li>
<li>面向对象是把构成问题事务分解成各个对象，建立对象的目的不是为了完成一个步骤，而是为了描叙某个事物在整个解决问题的步骤中的行为。面向对象是以<strong>功能</strong>来划分问题，而不是<strong>步骤</strong>。面向对象使用对象，类，继承，封装等基本概念来进行程序设计。</li>
</ul>
<p><strong>优缺点</strong>：</p>
<ol>
<li>面向过程：</li>
</ol>
<ul>
<li><strong>优点</strong>：性能比面向对象高，因为类调用时需要实例化，开销比较大，比较消耗资源;比如嵌入式开发、Linux/Unix等一般采用面向过程开发，性能是重要的因素。</li>
<li><strong>缺点</strong>：没有面向对象易维护、易复用、易扩展</li>
</ul>
<ol start="2">
<li>面向对象</li>
</ol>
<ul>
<li><strong>优点</strong>：可读性高，易维护、易复用、易扩展，由于面向对象有封装、继承、多态性的特性，可以设计出低耦合的系统，使系统更加灵活、更加易于维护。
<ul>
<li>易维护：每个开发人员只需要维护自己所负则的那个类的功能字段和方法的定义和扩展就OK了。</li>
<li>易复用：对象与对象之间相互独立，功能与对象之前的耦合性小。</li>
<li>易扩展：对象的属于与方法扩展性强。</li>
</ul>
</li>
<li><strong>缺点</strong>：类调用时需要实例化，开销比较大，性能比面向过程低</li>
</ul>
<h2 data-id="heading-13">箭头函数与普通函数的区别</h2>
<ol>
<li>函数体内的this对象，就是定义时所在的对象，而不是使用时所在的对象</li>
<li>不可以当作构造函数，也就是说，不可以使用new命令，否则会抛出一个错误</li>
<li>不可以使用arguments对象，该对象在函数体内不存在。如果要用，可以用Rest参数代替</li>
<li>不可以使用yield命令，因此箭头函数不能用作Generator函数</li>
</ol>
<h2 data-id="heading-14">异步编程的实现方式</h2>
<ol>
<li>回调函数</li>
</ol>
<p>优点：简单、容易理解
缺点：不利于维护，代码耦合高</p>
<ol start="2">
<li>事件监听(采用时间驱动模式，取决于某个事件是否发生)</li>
</ol>
<p>优点：容易理解，可以绑定多个事件，每个事件可以指定多个回调函数
缺点：事件驱动型，流程不够清晰</p>
<ol start="3">
<li>发布/订阅(观察者模式)</li>
</ol>
<p>类似于事件监听，但是可以通过‘消息中心’，了解现在有多少发布者，多少订阅者
Promise对象
优点：可以利用then方法，进行链式写法；可以书写错误时的回调函数；
缺点：编写和理解，相对比较难</p>
<ol start="4">
<li>Generator函数</li>
</ol>
<p>优点：函数体内外的数据交换、错误处理机制
缺点：流程管理不方便</p>
<ol start="5">
<li>async函数</li>
</ol>
<p>优点：内置执行器、更好的语义、更广的适用性、返回的是Promise、结构清晰。
缺点：错误处理机制</p>
<h2 data-id="heading-15">事件循环 & 宏任务、微任务</h2>
<p>事件循环：event loop
宏任务：macrotask 也称为 task
微任务：microtask 也称为 jobs</p>
<p><strong>微任务</strong></p>
<ul>
<li>process.nextTick (Node)</li>
<li>promise</li>
<li>Object.observe</li>
<li>MutationObserver (浏览器)</li>
</ul>
<p><strong>宏任务</strong></p>
<ul>
<li>script</li>
<li>setTimeout</li>
<li>setInterval</li>
<li>setImmediate</li>
<li>I/O</li>
<li>UI rendering</li>
<li>回调函数（事件、ajax等）</li>
</ul>
<p>在一次事件循环中，会先执行js线程的主任务，然后会去查找是否有微任务microtask（promise），如果有那就优先执行微任务，如果没有，在去查找宏任务macrotask（setTimeout、setInterval）进行执行。</p>
<p>一个事件循环的执行步骤：</p>
<ul>
<li>执行同步代码，这属于宏任务</li>
<li>执行所有微任务</li>
<li>必要的话渲染 UI</li>
<li>然后开始下一轮 Event loop，执行宏任务中的异步代码</li>
</ul>
<p>在一个事件循环中，微任务会比宏任务先执行，因此即使<code>new promise()</code>里立即执行的代码比<code>setTimeout(() => &#123;&#125;, 0)</code>的代码后面，也会提前执行。</p>

<h2 data-id="heading-16">Javascript垃圾回收</h2>
<ol>
<li>标记清除（mark and sweep）</li>
</ol>
<ul>
<li>这是JavaScript最常见的垃圾回收方式，当变量进入执行环境的时候，比如函数中声明一个变量，垃圾回收器将其标记为“进入环境”，当变量离开环境的时候（函数执行结束）将其标记为“离开环境”</li>
<li>垃圾回收器会在运行的时候给存储在内存中的所有变量加上标记，然后去掉环境中的变量以及被环境中变量所引用的变量（闭包），在这些完成之后仍存在标记的就是要删除的变量了</li>
</ul>
<ol start="2">
<li>引用计数(reference counting)</li>
</ol>
<ul>
<li>在低版本IE中经常会出现内存泄露，很多时候就是因为其采用引用计数方式进行垃圾回收。</li>
<li>引用计数的策略是跟踪记录每个值被使用的次数。</li>
<li>当声明了一个 变量并将一个引用类型赋值给该变量的时候这个值的引用次数就加1</li>
<li>如果该变量的值变成了另外一个，则这个值得引用次数减1，</li>
<li>当这个值的引用次数变为0的时 候，说明没有变量在使用，这个值没法被访问了，因此可以将其占用的空间回收，这样垃圾回收器会在运行的时候清理掉引用次数为0的值占用的空间</li>
</ul>

<h2 data-id="heading-18">进程与线程</h2>
<p>本质上来说，进程与线程都是 CPU 工作时间片的一个描述。</p>
<ul>
<li>进程描述了 CPU 在运行指令及加载和保存上下文所需的时间，放在应用上来说就代表了一个程序。</li>
<li>线程是进程中的更小单位，描述了执行一段指令所需的时间。</li>
</ul>
<p><strong>举例</strong>：
把这些概念拿到浏览器中来说，当你打开一个 <code>Tab</code> 页时，其实就是创建了一个<strong>进程</strong>，一个进程中可以有多个线程，比如<strong>渲染</strong>线程、<strong>JS 引擎</strong>线程、<strong>HTTP 请求</strong>线程等等。当你发起一个请求时，其实就是创建了一个线程，当请求结束后，该线程可能就会被销毁。</p>
<p><strong>单线程的好处</strong>：
上文说到了 JS 引擎线程和渲染线程，大家应该都知道，在 JS 运行的时候可能会阻止 UI 渲染，这说明了两个线程是互斥的。这其中的原因是因为 JS 可以修改 DOM，如果在 JS 执行的时候 UI 线程还在工作，就可能导致不能安全的渲染 UI。这其实也是一个单线程的好处，得益于 JS 是单线程运行的，可以达到节省内存，节约上下文切换时间，没有锁的问题的好处</p>

<p>为何<code>currentTarget</code>是<code>null</code>：</p>
<ul>

<h2 data-id="heading-23">§ typescript</h2>
<p>TypeScript是一种由微软开发和维护的免费开源编程语言。它是一个强类型的JavaScript超集，目前不能直接在浏览器环境运行，可编译为纯JavaScript后执行。</p>
<p>TypeScript可以被认为是Javascript的优化版本，弃其糟粕，取其精华，在保持其动态语言的相对自由度前提下，修饰了没有编译类型检查，不适合超大型项目的缺陷。</p>
<p><strong>使用ts的场景</strong>：TypeScript拥有更好的类型检查，但在某种程度上也让js的灵活性收到了限制。因此我们应该在日常开发相对独立的业务需求使用js，在开发多人协作、逻辑复杂的大型项目时使用ts。</p>

<h3 data-id="heading-26">interface 和 type 的区别</h3>
<p><strong>1. interface接口</strong>
<strong>定义</strong>：对值所具有的结构进行类型检查。
<strong>作用</strong>：为类型命名和第三方代码定义数据类型规范。</p>
<p><strong>2. type类型别名</strong></p>
<p><strong>作用</strong>：给类型起个新名字。</p>
<p><strong>3. type 可以而 interface 不行</strong></p>
<ul>
<li>type 可以声明基本类型别名，联合类型，元组等类型，类和interface不能(只能描述对象和函数)。</li>
<li>type 语句中还可以使用 typeof 获取实例的类型进行赋值</li>
</ul>
<p><strong>4. interface 可以而 type 不行</strong></p>
<ul>
<li>interface 能够声明合并（当接口名字相同，取并集）</li>
</ul>
<p><strong>5. interface 和 type 都可以</strong></p>
<ul>
<li>两者都可以用来描述对象或函数的类型，但是语法不同。</li>
</ul>
