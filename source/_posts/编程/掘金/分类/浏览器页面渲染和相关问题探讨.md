<h2 data-id="heading-0">简单说说浏览器的渲染过程</h2>
<ul>
<li><strong>解析HTML，生成 DOM 树。</strong></li>
<li><strong>解析 CSS，生成  CSSOM（CSS 对象模型）。</strong> 大概是根据后代选择器之类的构建出类似 “字典树” 一样的结构。另外需要注意的是，浏览器会提供默认的样式（User Agent）作为兜底。</li>
<li><strong>DOM 树和 CSSOM 树进行合成，创建渲染树（render tree）。</strong> 具体做法是从 DOM 树根节点遍历可见节点，即排除设置了<code>display: none;</code> 和不进行渲染（如 <code><head></code>）的节点。并对每个可见节点找到适配的 CSSOM 规则，需要考虑 “样式继承”（比如 <code>font-size</code> 属性会影响子元素） 和 “样式层叠” （就是优先级，比如 ID 选择器比 类选择器优先级高）的情况。</li>
<li><strong>【重排】进入 “布局”（Layout） 阶段，计算可见节点的几何信息。</strong> 计算元素的几何信息（如位置和尺寸）和样式信息（背景色之类）。比如 <code>width: 50%;</code> 这种相对值会根据父元素宽度得到绝对的像素值。</li>
<li><strong>分层</strong>。文档中的层叠上下文（比如绝对定位的元素）和需要裁切的节点（比如在容器里放不下的文本）都会创建新的独立图层。</li>
<li><strong>【重绘】绘制</strong>。或者叫栅格化，将渲染树得到的信息绘制到屏幕上。</li>
</ul>
<p>PS：重排也叫回流。</p>
<h2 data-id="heading-1">重排</h2>
<h3 data-id="heading-2">重排发生的条件</h3>
<ul>
<li>页面首次渲染</li>
<li>增删或替换可见的 DOM 元素</li>
<li>浏览器窗口发生变化</li>
<li>元素位置或尺寸发生变化</li>
</ul>
<p>除了首次渲染，其他情况都对元素的的几何属性进行了修改，可能会对布局产生巨大的影响，所以需要对节点的几何信息等做重新的计算，也就是重排，然后绘制出来。</p>
<p>重排之后，都是要重绘的，流程就是这样。</p>
<h3 data-id="heading-3">不会触发重排的重绘的操作</h3>
<ul>
<li>对元素进行不会导致元素几何信息变化的操作。比如修改背景色、字体颜色。</li>
</ul>
<h3 data-id="heading-4">如何减少重排重绘</h3>
<p>因为重排很耗费性能，所以浏览器自己做了优化，会将多个修改操作缓存到队列里，在合适的时候再重排，也就说是异步的。但如果执行到 <strong>需要获取布局信息的代码</strong>（如 getBoundingClientRect，scrollHeight 之类的） 时，机会立即清空队列，进行重排。因为重新渲染的话，是拿不到最新的布局信息的。</p>
<p>减少重排的方法有：</p>
<ul>
<li><strong>减少 DOM 操作，使用合适的算法。</strong> 比如更新新增列表的一个项时，不要销毁掉所有的元素，再根据数组创建全新的列表项。React 和 Vue 的虚拟 DOM 的 Diff 算法优化就是减少了 DOM 的修改操作。</li>
<li><strong>添加一个元素，先把它的子元素都创建好，再添加到 DOM 中。</strong></li>
<li><strong>合适地缓存布局信息</strong>。避免在循环中频繁获取布局信息，导致不断发生重排。</li>
<li><strong>将经常发生变化的元素放在独立的一层</strong>。比如设置了 <code>transform</code> 的元素会使用独立的一个图层，会使用 <code>GPU</code> 渲染，进行硬件加速，都不会引起重绘。我想可能是独立出来的一套动画系统。绝对定位这些脱离文档的方式，也会创建图层，只会在自身局部会发生回流重绘，能减少工作量。</li>
</ul>
<h2 data-id="heading-5">JS 阻塞</h2>
<p>首先了解下 script 标签的两个属性的功能：</p>
<ul>
<li><code>defer</code>：在 DOM 树完成解析后再执行脚本；</li>
<li><code>async</code>：脚本下载完后立即执行。无法保证 JS 脚本按顺序执行</li>
</ul>
<p>阻塞的情况：</p>
<ul>
<li><strong>JS 会阻塞 DOM 解析。</strong> 因为 JS 可能会操作 DOM，为了不做无用的工作，浏览器决定暂且搁置 DOM 的解析，先让脚本下载然后执行完再继续。思考一下，为什么我们建议把业务代码的 script 标签放在末尾？这是因为放在末尾的脚本能够等到 DOM 树解析完，业务代码往往需要访问节点元素，这样才不会拿到 null。至于一些第三方库不需要访问节点，所以放在开头并没有问题。</li>
</ul>
<h2 data-id="heading-6">CSS 阻塞</h2>
<p>要想构建 CSSOM 树，首先需要拿到 CSS 内容，而通常我们使用的都是外链的 CSS 文件。这样就会有一个请求的过程，于是阻塞发生了。</p>
<ul>
<li><strong>CSS 阻塞不会影响 DOM 树的解析生成。</strong>（可通过 <code><script defer></code> 和一个延时返回的 CSS 资源来测试，<code>defer</code> 能够在 DOM 树完成解析后再执行脚本，结果会是先执行脚本，页面要晚一点才渲染出来 ）</li>
<li>**CSS 会阻塞页面渲染。**指的是阻塞 CSSOM 树的生成，因为信息不够完全，</li>
<li><strong>CSS 阻塞可能会阻塞后面的 JS 脚本的运行</strong>（其后存在没有设置 <code>async</code> 或 <code>defer</code> 的  script 标签时）。效果就是 <code>CSS 阻塞 -> JS 阻塞 -> DOM 解析阻塞</code>，于是我们看到了 <strong>CSS 阻塞 DOM 解析</strong> 现象的发生。个人猜测原因是：浏览器认为脚本里可能有需要获取容器高度之类的内容，而这些必须要等页面渲染出来次能拿到，而标记了 <code>async</code> 或 <code>defer</code> 的脚本则不考虑这些。</li>
</ul>
<h2 data-id="heading-7">Vue 首次加载白屏的问题</h2>
<p>Vue 要构建 SPA（单页面应用），需要将所有的组件都提前放到 JS 脚本里，一次性下载。如果组件很多， JS 太大了，就会导致 DOM 渲染变慢，从而导致白屏加载速度较慢。</p>
<p>优化方案有：</p>
<h3 data-id="heading-8">合理地使用路由懒加载</h3>
<p>下面是 vue-cli 脚手架启用路由选项后，给我们创建的路由配置。</p>
<pre><code class="hljs language-js copyable" lang="js">&#123;
  <span class="hljs-attr">path</span>: <span class="hljs-string">'/about'</span>,
  <span class="hljs-attr">name</span>: <span class="hljs-string">'About'</span>,
  <span class="hljs-comment">// route level code-splitting</span>
  <span class="hljs-comment">// this generates a separate chunk (about.[hash].js) for this route</span>
  <span class="hljs-comment">// which is lazy-loaded when the route is visited.</span>

  <span class="hljs-comment">// 看这里</span>
  <span class="hljs-attr">component</span>: <span class="hljs-function">() =></span> <span class="hljs-keyword">import</span>(<span class="hljs-comment">/* webpackChunkName: "about" */</span> <span class="hljs-string">'../views/About.vue'</span>)
&#125;,
<span class="copy-code-btn">复制代码</span></code></pre>
<p>About 组件只有在路由被访问时才会加载。也就是说，第一次访问首页或其他路由时，About 组件不会被加载。从而减少了首次加载 js 的大小，减少白屏时间。另外，注释里的 <code>webpackChunkName</code> 制定了加载的脚本的名称，如果多个懒加载组件都是用同一个 <code>webpackChunkName</code> ，那么它们会被放在同一个脚本里。也就是说，对于放在同一个脚本里的多个组件，当访问其中一个路由组件匹配的路由时，其他的组件也会提前下载好。</p>
<h3 data-id="heading-9">静态资源都使用 CDN 加速</h3>
<p>CDN 通过缓存代理，将源站的资源分发在各个网络节点上，构造专用网络。这个专用网络是跨运营商、跨地域的，是真正的高速网络。通过负载均衡（利用 DNS），用户能够访问离它最近的资源，从而提高资源加载速度。相比跋山涉水地访问遥远的源节点，就近取水无疑速度更快。</p>
<p>这样 JS 脚本就能被快速加载，从而减少白屏时间。</p>
<p>这个算是提高加载速度的优化，我们还可以使用 http2、对脚本进行压缩、http 开启 gzip 压缩等老生常谈的减少资源大小和提高资源请求速度的方式。</p>
