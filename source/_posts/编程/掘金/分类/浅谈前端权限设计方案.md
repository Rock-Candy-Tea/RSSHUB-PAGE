h2&#123;font-size:20px&#125;.markdown-body h3&#123;font-size:18px&#125;&#125;</style><h1 data-id="heading-0">前言</h1>
<p>前端权限架构的设计一直都是备受关注的技术点.通过给项目引入了权限控制方案,可以满足我们灵活的调整用户访问相关页面的许可.</p>
<p>比如哪些页面向游客开放,哪些页面必须要登录后才能访问,哪些页面只能被某些角色访问(比如超级管理员).有些页面即使用户登录了但受到角色的限制,他也只被允许看到页面的部分内容.</p>
<p>出于实际工作的需要,很多项目(尤其类后台管理系统)需要引入权限控制.倘若权限整体的架构设计的不好或者没有设计,会导致项目中各种权限代码混入业务代码造成结构混乱,其次想给新模块引入权限控制或者功能扩展都十分棘手.</p>
<p>虽然前端在权限层面能做一些事情,但很遗憾真正对权限进行把关的是后端.例如一个软件系统,前端在不写一行权限代码的情况下,当用户进入某个他无权访问的页面时,后端是可以判断他越权访问并拒绝返回数据的.由此可见前端即使不做什么整个系统也是可以正常运行的,但这样应用的体验很不好.另外一个很重要的原因就是前端做的权限校验都是可以被本地数据造假越权通过.</p>
<p>前端如果能判断某用户越权访问页面时,就不要让他进入那张页面后再弹出无权访问的信息提示,因为这样体验很差.最优方案是直接关闭那些页面的入口,只让他看到他能访问的页面.即使他通过输入路径恶意访问,导航最后只会将它带到默认页面或<code>404</code>页面.</p>
<p>前端做的权限控制大抵是先接受后台发送的权限数据,然后将数据注入到应用中.整个应用于是开始对页面的展现内容以及导航逻辑进行控制,从而达到权限控制的目的.前端做的权限控制虽然能提供一层防护,但根本目的还是为了优化体验.</p>
<p>本文接下来将从下面三个层面,从易到难步步推进,讲述目前前端主流的权限控制方案的实现.(下面代码将会以<code>vue3</code>和<code>vue-router 4</code>演示)</p>
<ul>
<li>登录权限控制</li>
<li>页面权限控制</li>
<li>内容权限控制</li>
</ul>
<h1 data-id="heading-1">登录权限控制</h1>
<p>登录权限控制要做的事情,是实现哪些页面能被游客访问,哪些页面只有登录后才能被访问.在一些没有引入角色的软件系统中,通过是否登录来评定页面能否被访问在实际工作中非常常见.</p>
<p>实现这个功能也非常简单,首先按照惯例定义一份路由.</p>
<pre><code class="copyable">    export const routes = [
      &#123;
         path: '/login', //登录页面
         name: 'Login',
         component: Login,
      &#125;,
      &#123;
         path:"/list", // 列表页
         name:"List",
         component: List, 
      &#125;,
      &#123;
         path:"/myCenter", // 个人中心
         name:"MyCenter",
         component: MyCenter, 
         meta:&#123;
            need_login:true //需要登录
         &#125;
      &#125;
    ]
<span class="copy-code-btn">复制代码</span></code></pre>
<p>假定存在三个页面:登录页、列表页和个人中心页.登录页和列表页所有人都可以访问,但个人中心页面需要登录后才能看到,给该路由添加一个<code>meta</code>对象,并将<code>need_login</code>置为<code>true</code>;</p>
<p>另外对于那些需要登录后才能看到的页面,用户如果没有登录就访问,就将页面跳转到登录页.等到他填写完用户名和密码点击登录后直接跳转到原来他想访问的页面.</p>
<p>在代码层面,通过<code>router.beforeEach</code>可以轻松实现上述目标,每次页面跳转时都会调用<code>router.beforeEach</code>包裹的函数,代码如下.</p>
<p><code>to</code>是要即将访问的路由信息,从其中拿到<code>need_login</code>的值可以判断是否需要登录.再从<code>vuex</code>中拿到用户的登录信息.</p>
<p>如果用户没有登录并且要访问的页面又需要登录时就使用<code>next</code>跳转到登录页面,并将需要访问的页面路由名称通过<code>redirect_page</code>传递过去,在登录页面就可以拿到<code>redirect_page</code>等登录成功后直接跳转.</p>
<pre><code class="copyable">//vue-router4 创建路由实例
const router = createRouter(&#123;  
  history: createWebHashHistory(),
  routes,
&#125;);

router.beforeEach((to, from, next) => &#123;
  const &#123; need_login = false &#125; = to.meta;
  const &#123; user_info &#125; = store.state; //从vuex中获取用户的登录信息
  if (need_login && !user_info) &#123;
    // 如果页面需要登录但用户没有登录跳到登录页面
    const next_page = to.name; // 配置路由时,每一条路由都要给name赋值
    next(&#123;
      name: 'Login',
      params: &#123;
        redirect_page: next_page,
        ...from.params, //如果跳转需要携带参数就把参数也传递过去
      &#125;,
    &#125;);
  &#125; else &#123;
    //不需要登录直接放行
    next();
  &#125;
&#125;);

<h1 data-id="heading-2">页面权限控制</h1>
<p>页面权限控制要探讨的问题是如何给不同角色赋予不同的页面访问权限,接下来先了解一下角色的概念.</p>
<p>在一些权限设置比较简单的系统里,使用上面第一种方法就足够了,但如果系统引入了角色,那么就要在上面基础上,再进一步改造增强权限控制的能力.</p>
<p>角色的出现是为了更加个性化配置权限列表.比如当前系统设置三个角色:普通会员,管理员以及超级管理员.普通会员能够浏览软件系统的所有内容,但是它不能编辑和删除内容.管理员拥有普通会员的所有能力,另外它还能删除和编辑内容.超级管理员拥有软件系统所有权限,他单独拥有赋予某个账号为管理员或移除其身份的能力.</p>
<p>一旦软件系统引入了角色的概念,那么每个账户在注册之后就会被赋予相应的角色,从而拥有相应的权限.我们前端要做的事情就是依据不同角色给与它相应页面访问和操作的权限.这里要注意,前端依据的客体是角色,不是某个账户,因为账户是依托于角色的.</p>
<p>普通会员,管理员以及超级管理员这样角色的安排还是一种非常简单的划分方式,在实际项目中,角色的划份要更加细致的多.比如一些常见的后台业务系统,软件系统会按照公司的各个部门来建立角色,诸如市场部,销售部,研发部之类.公司的每个成员就会被划分到相应角色中,从而只具备该角色所拥有的权限.</p>
<p>公司另外一些高层领导他们的账户则会被划分到普通管理员或高级管理员中,那么他们相较于其他角色也会拥有更多的权限.</p>
<p>上面介绍那么多角色的概念其实是为了从全栈的维度去理解权限的设计,但真正落地到前端项目中是不需要去处理角色逻辑的,那部分功能主要由后端完成.</p>
<p>现在假定后端不处理角色完全交给前端来做会出现什么问题.首先前端新建一个配置文件,假定当前系统设定三种角色:普通会员,管理员以及超级管理员以及每个角色能访问的页面列表(伪代码如下).</p>
<pre><code class="copyable"> export const permission_list = &#123;
   member:["List","Detail"], //普通会员
   admin:["List","Detail","Manage"],  // 管理员
   super_admin:["List","Detail","Manage","Admin"]  // 超级管理员

<p>数组里每个值对应着前端路由配置的<code>name</code>值.普通会员能访问<code>列表页</code>和<code>详情页</code>,管理员能额外访问<code>内容管理页面</code>,超级管理员能额外访问<code>人员管理页面</code>.</p>
<p>整个运作流程简述如下.当用户登录成功之后,通过接口返回值得知用户数据和所属角色.拿到角色值后就去配置文件里取出该角色能访问的页面列表数组,随后将这部分权限数据加载到应用中从而达到权限控制的目的.</p>
<p>从上面流程看,角色放在前端配置也是可以的.但假如项目已经上线,产品经理要求项目急需增加一个新角色<code>合作伙伴</code>,并把原来已经存在的用户<code>张三</code>移动到<code>合作伙伴</code>角色下面.那这样的变动会导致前端需要修改代码文件,在原来的配置文件上再新建角色来满足这一需求.</p>
<p>由此可见由前端来配置角色列表是非常不灵活且容易出错的,那么最优方案是交给后端去配置.用户一旦登录后,后端接口直接返回该账号拥有的权限列表就行了,至于该账户属于什么角色以及角色拥有的页面权限全部丢给后端去处理.</p>
<p>用户登录成功后,后端接口数据返回如下.</p>
<pre><code class="copyable">&#123;

<p>通过接口的返回值<code>permission_list</code>可知,张三能访问<code>列表页</code>、<code>详情页</code>以及<code>内容管理页</code>.我们先回到路由配置页面,看看如何配置.</p>

<p>现在将所有路由分成两部分,静态路由<code>routes</code>和动态路由<code>dynamic_routes</code>.静态路由里面的页面是所有角色都能访问的,它里面主要区分登录访问和非登录访问,处理的逻辑与上面介绍的<code>登录权限控制</code>一致.</p>
<p>动态路由<code>dynamic_routes</code>里面存放的是与角色定制化相关的的页面.现在继续看下面张三的接口数据,该如何给他设置权限.</p>
<pre><code class="copyable">&#123;

<p>用户登录成功后,一般会将上述接口信息存到<code>vuex</code>和<code>localStorage</code>里面.假如此时刷新浏览器,我们就要动态添加路由信息.</p>
<pre><code class="copyable">import store from "@/store";

    &#125;)

<p>核心代码在动态添加路由里面,主要利用了<code>vue-router 4</code>提供的<code>API</code>即<code>router.addRoute</code>,它能够给已经创建的路由实例继续添加路由信息.</p>
<p>我们先从<code>vuex</code>里面拿到当前用户的权限列表,然后遍历动态路由数组<code>dynamic_routes</code>,从里面过滤出允许访问的路由,最后将这些路由动态添加到路由实例里.</p>
<p>这样就实现了用户只能按照他对应的权限列表里的规则访问到相应的页面,至于那些他无权访问的页面,路由实例根本没有添加相应的路由信息,因此即使用户在浏览器强行输入路径越权访问也是访问不到的.</p>
<p>由于<code>vue-router 4</code>废除了之前的<code>router.addRoutes</code>,换成了<code>router.addRoute</code>.每一次只能一个个添加路由信息,所以要将<code>allow_routes</code>遍历循环添加.</p>
<p>动态添加路由这部分代码最好单独封装起来,因为用户第一次使用还没登录时,<code>store.state.user</code>是为空的,上面动态添加路由的逻辑会被跳过.那么在用户登录成功获取到权限列表的信息后,需要再把上面动态添加路由的逻辑执行一遍.</p>

<h2 data-id="heading-4">切换用户</h2>
<p>切换用户信息是非常常见的功能,但是应用在切换成不同账号后可能会引发一些问题.例如用户先使用超级管理员登录,由于超级管理员能访问所有页面,因此所有页面路由信息都会被添加到路由实例里.</p>
<p>此时该用户退出账号,使用一个普通会员的账号登录.在不刷新浏览器的情况下,路由实例里面仍然存放了所有页面的路由信息,即使当前账号只是一个普通会员,如果他越权访问相关页面,路由还是会跳转的,这样的结果并不是我们想要的.</p>
<p>解决方案有两个.第一是用户每次切换账户后刷新浏览器重新加载,刷新后的路由实例是重新配置的所以可以避免这个问题,但是刷新页面会带来不好的体验.</p>
<p>第二个方案是当用户选择登出后,清除掉路由实例里面处存放的路由栈信息(代码如下).</p>
<pre><code class="copyable">  const router = useRouter(); // 获取路由实例
  const logOut = () => &#123; //登出函数
      //将整个路由栈清空
      const old_routes = router.getRoutes();//获取所有路由信息
      old_routes.forEach((item) => &#123;
        const name = item.name;//获取路由名词
        router.removeRoute(name); //移除路由
      &#125;);
      //生成新的路由栈
      routes.forEach((route) => &#123;
        router.addRoute(route);
      &#125;);
      router.push(&#123; name: "Login" &#125;); //跳转到登录页面
    &#125;;
<span class="copy-code-btn">复制代码</span></code></pre>
<p>移除单个路由主要利用了官方提供的<code>API</code>,即<code>router.removeRoute</code>.</p>
<p>路由栈清空后什么页面都不能访问了,甚至登录页面都访问不了.所以需要再把静态的路由列表<code>routes</code>引入进来,使用<code>router.addRoute</code>再添加进入.这样就能让路由栈恢复到最初的状态.</p>
<h1 data-id="heading-5">内容权限控制</h1>
<p><code>页面权限控制</code>它能做到让不同角色访问不同的页面,但对于一些颗粒度更小的项目,比如希望不同的角色都能进入页面,但要求看到的页面内容不一样,这就需要对内容进行权限控制了.</p>
<p>假设某个后台业务系统的界面如下图所示.表格里面存放的是列表数据,当点击发布需求时跳转到新增页面.当勾选列表中的某一条数据后,点击修改按钮显示修改该条数据的弹出框.同理点击删除按钮显示删除该条数据的弹出框</p>
<p><img alt="table.png" class="lazyload" src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/f30bc879f62a477eae902062e2ad3d18~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>假设项目需求该系统存在三个角色:职员、领导和高层领导.职员不具备<code>修改</code>、<code>删除</code>以及<code>发布需求</code>的功能,他只能查看<code>列表</code>.当职员进入该页面时,页面上只显示<code>列表</code>内容,其他三个按钮移除.</p>
<p>领导角色保留<code>列表</code>和<code>发布需求</code>按钮.高级领导角色保留页面上所有内容.</p>
<p>我们拿到图片后要先要对页面内容整体分析一遍,按照<code>增删查改</code>四个维度对页面内容进行归类.使用简称<code>CURD</code>来标识(<code>CURD</code>分别代表创建（<code>Create</code>）、更新（<code>Update</code>）、读取（<code>Retrieve</code>）和删除（<code>Delete</code>）).</p>
<p>上图中<code>列表</code>内容属于查询操作,因此设定为<code>R</code>.凡是具备<code>R</code>权限的用户就显示该<code>列表</code>内容.</p>
<p><code>发布需求</code>属于新增操作,设定凡是具备<code>C</code>权限的用户就显示该按钮.</p>
<p>同理<code>修改</code>按钮对应着<code>U</code>权限,<code>删除</code>按钮对应着<code>D</code>权限.</p>
<p>由此可以推断出职员角色在该页面的权限编码为<code>R</code>,它只能查看<code>列表</code>内容无法操作.</p>
<p>领导角色对应的权限编码为<code>CR</code>.高级领导对应的权限编码为<code>CURD</code>.</p>
<p>现在用户登录完成后,假设后端接口返回的数据如下(将这条数据存到<code>vuex</code>):</p>
<pre><code class="copyable">&#123;
  user_id:1,
  user_name:"张三",
  permission_list:&#123;
    "List":"CR", //权限编码
    "Detail":"CURD"  //权限编码
  &#125;
&#125;
<span class="copy-code-btn">复制代码</span></code></pre>
<p>张三除了静态路由设置的页面外,他只能额外访问<code>List</code>列表页以及<code>Detail</code>详情页.其中列表页他只具备创建和新增权限,详情页他具备增删查改所有权限.那么当张三访问上图中的页面时,页面中应该只显示<code>列表</code>和<code>发布需求</code>按钮.</p>
<p>我们现在要做的就是设计一个方案尽可能让页面内容方便被权限编码控制.首先创建一个全局的自定义指令<code>permission</code>,代码如下:</p>
<pre><code class="copyable">import router from './router';
import store from './store';

const app = createApp(App); //创建vue的根实例

app.directive('permission', &#123;
  mounted(el, binding, vnode) &#123;
    const permission = binding.value; // 获取权限值
    const page_name = router.currentRoute.value.name; // 获取当前路由名称
    const have_permissions = store.state.permission_list[page_name] || ''; // 当前用户拥有的权限
    if (!have_permissions.includes(permission)) &#123;
      el.parentElement.removeChild(el); //不拥有该权限移除dom元素
    &#125;
  &#125;,
&#125;);
<span class="copy-code-btn">复制代码</span></code></pre>
<p>当元素挂载完毕后,通过<code>binding.value</code>获取该元素要求的权限编码.然后拿到当前路由名称,通过路由名称可以在<code>vuex</code>中获取到该用户在该页面所拥有的权限编码.如果该用户不具备访问该元素的权限,就把元素<code>dom</code>移除.</p>
<p>对应到上面的案例,在页面里按照如下方式使用<code>v-permission</code>指令.</p>
<pre><code class="copyable"><template>
    <div>
      <button v-permission="'U'">修改</button>  <button v-permission="'D'">删除</button>
    </div>
    <p>
      <button v-permission="'C'">发布需求</button>
    </p>

    <!--列表页-->
    <div v-permission="'R'">
     ...
    </div>
</template>
<span class="copy-code-btn">复制代码</span></code></pre>
<p>将上面模板代码和自定义指令结合理解一下就很容易明白整个内容权限控制的逻辑.首先前端开发页面时要将页面分析一遍,把每一块内容按照权限编码分类.比如修改按钮就属于<code>U</code>,删除按钮属于<code>D</code>.并用<code>v-permission</code>将分析结果填写上去.</p>
<p>当页面加载后,页面上定义的所有<code>v-permission</code>指令就会运行起来.在自定义指令内部,它会从<code>vuex</code>中取出该用户所拥有的权限编码,再与该元素所设定的编码结合起来判端是否拥有显示权限,权限不具备就移除元素.</p>
<p>虽然分析过程有点复杂,但是以后每个新页面想接入权限控制非常方便.只需要将新页面的各个<code>dom</code>元素添加一个<code>v-permission</code>和权限编码就完成了,剩下的工作都交给自定义指令内部去做.</p>

<h2 data-id="heading-7">动态导航</h2>
<p>下图中的动态导航也是实际工作中非常常见的需求,比如销售部所有成员只能看到销售模块下的两个页面,同理采购部成员只能看到采购模块下的页面.</p>
<p>下面侧边栏导航组件需要根据不同权限显示不同的页面结构,以满足不同角色群体的要求.</p>
<p><img alt="nav.png" class="lazyload" src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/7b5c75bd3d7b43719deb645fcf07a35a~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>我们要把这种需要个性化设置的组件与上面使用<code>v-permission</code>控制的模式区分开.上面那些页面之所以能使用<code>v-permission</code>来控制,主要原因是因为产品经理在设计整个软件系统的页面时是按照<code>增删查改</code>的规则进行的.因此我们就能抽象出其中存在的共性与规律,再借助自定义指令来简化权限系统的开发.</p>
<p>但是侧边栏组件一般全局只有一个,没有什么特别的规律而言,那就只需要在组件内部使用<code>v-if</code>依据权限值动态渲染就行了.</p>
<p>比如后台接口如下:</p>
<pre><code class="copyable"> &#123;
