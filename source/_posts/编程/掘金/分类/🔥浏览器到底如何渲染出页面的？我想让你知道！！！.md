<h3 data-id="heading-0">浏览器解析渲染页面</h3>
<h4 data-id="heading-1"><strong>流程简述</strong></h4>
<p>浏览器内核拿到内容后，<code>渲染步骤</code>大致可以分为以下几步：</p>
<ol>
<li>
<p><code>解析HTML，构建DOM树</code></p>
</li>
<li>
<p><code>解析CSS，生成CSS规则树</code></p>
</li>
<li>
<p><code>合并DOM树和CSS规则，生成render树</code></p>
</li>
<li>
<p><code>布局render树（Layout/reflow），负责各元素尺寸、位置的计算</code></p>
</li>
<li>
<p>绘制render(渲染)树（paint），绘制页面像素信息</p>
</li>
<li>
<p>浏览器会将各层的信息发送给GPU，GPU会将各层合成（composite），显示在屏幕上</p>
</li>
</ol>
<p><strong>OS:--</strong>
<code>reflow：也称作layout，中文叫回流/重排</code>，<code>一般意味着元素的内容、结构、位置或尺寸发生了变化，需要重新计算样式和渲染树，这个过程称为reflow。</code></p>
<p><code>repaint：中文重绘，</code>意味着元素发生的改变<code>只是影响了元素的一些外观</code>之类的时候(例如：背景色，边框颜色，文字颜色等)，此时<code>只需要应用新样式绘制这个元素</code>就可以了。</p>
<hr>
<h4 data-id="heading-2">流程详解</h4>
<h5 data-id="heading-3">1.根据 HTML 解析 DOM 树</h5>
<ul>
<li>根据 HTML 的内容，将标签按照结构解析成为 DOM 树，DOM 树解析的过程是一个<code>深度优先遍历</code>。即先构建当前节点的所有子节点，再构建下一个兄弟节点。</li>
<li>在读取 HTML 文档，构建 DOM 树的过程中，<code>若遇到 script 标签，则 DOM 树的构建会暂停，直至脚本执行完毕。</code></li>
</ul>
<h5 data-id="heading-4">2.根据 CSS 解析生成 CSS 规则树</h5>
<ul>
<li><code>解析 CSS 规则树时 js 执行将暂停，直至 CSS 规则树就绪</code>。</li>
<li><code>浏览器在 CSS 规则树生成之前不会进行渲染</code>。</li>
</ul>
<h5 data-id="heading-5">3.结合 DOM 树和 CSS 规则树，生成渲染树</h5>
<ul>
<li>DOM 树和 CSS 规则树全部准备好了以后，浏览器才会开始<code>构建渲染树</code>。</li>
<li>精简 CSS 并可以加快 CSS 规则树的构建，从而加快页面相应速度。</li>
</ul>
<h5 data-id="heading-6">4.根据渲染树计算每一个节点的信息（布局）</h5>
<ul>
<li>布局：通过渲染树中渲染对象的信息，计算出每一个渲染对象的<code>位置和尺寸</code></li>
<li>回流：在布局完成后，发现了某个部分发生了变化影响了布局，那就需要倒回去<code>重新渲染。</code></li>
</ul>
<h5 data-id="heading-7">5.根据计算好的信息绘制页面</h5>
<ul>
<li>绘制阶段，系统会遍历呈现树，并调用呈现器的<code>“paint”方法</code>，将呈现器的内容显示在屏幕上。</li>
<li>重绘：某个元素的背景颜色，文字颜色等，不影响元素周围或内部布局的属性，将只会引起浏览器的重绘。</li>
<li>回流：某个元素的尺寸发生了变化，则需重新计算渲染树，重新渲染。</li>
</ul>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/0efc188a5dbd4647bbc16e6357f0b02a~tplv-k3u1fbpfcp-watermark.image" alt="img" loading="lazy" referrerpolicy="no-referrer"></p>
<hr>
<h4 data-id="heading-8">思考：如何减少回流和重绘？</h4>
<p>我们都知道：<code> Reflow要比Repaint更花费时间，也就更影响性能。要尽量避免过多的Reflow。</code>那该怎么来解决呢？</p>
<p>我们首先来回顾一下<code>渲染流水线</code>的流程:</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/eb9965fb65ad4ec2960dfba32e170269~tplv-k3u1fbpfcp-watermark.image" alt="img" loading="lazy" referrerpolicy="no-referrer"></p>
<p>想要尽量减少回流和重绘的次数，提高性能，首先我们得先知道<code>触发条件</code>。</p>
<p>常见的<code>触发回流的情况：</code></p>
<ol>
<li>一个 DOM 元素的几何属性变化，常见的几何属性有<code>width</code>、<code>height</code>、<code>padding</code>、<code>margin</code>、<code>left</code>、<code>top</code>、<code>border</code> 等等, 这个很好理解。</li>
<li>使 DOM 节点发生<code>增减</code>或者<code>移动</code>。</li>
<li>读写 <code>offset</code>族、<code>scroll</code>族和<code>client</code>族属性的时候，浏览器为了获取这些值，需要进行回流操作。</li>
<li>调用 <code>window.getComputedStyle</code> 方法。</li>
</ol>
<p>现在我们来看看<code>回流过程。</code></p>
<p>依照上面的渲染流水线，触发回流的时候，如果 DOM 结构发生改变，则重新渲染 DOM 树，然后将后面的流程(包括主线程之外的任务)全部走一遍。</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/3ef77d4bfc294eddbe434e40befef81d~tplv-k3u1fbpfcp-watermark.image" alt="img" loading="lazy" referrerpolicy="no-referrer"></p>
<p>相当于将<code>解析和合成的过程重新又走了一遍</code>，开销是非常大的。</p>
<hr>
<p>常见的<code>触发重绘的情况：</code></p>
<p>当 DOM 的修改导致了<code>样式的变化</code>，并且<code>没有影响几何属性的时候</code>，会导致<code>重绘</code>(<code>repaint</code>)。</p>
<p><code>重绘过程</code>:由于没有导致 DOM 几何属性的变化，因此元素的位置信息不需要更新，从而省去布局的过程。流程如下：</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/35f3484f7ca64da7b773c583ec1a1ef1~tplv-k3u1fbpfcp-watermark.image" alt="img" loading="lazy" referrerpolicy="no-referrer"></p>
<p>跳过了<code>生成布局树</code>和<code>建图层树</code>的阶段，直接生成绘制列表，然后继续进行分块、生成位图等后面一系列操作。</p>
<p>所以说：<code>重绘不一定导致回流，但回流一定发生了重绘</code>。</p>
<hr>
<p>知道上面的原理之后，对于开发过程有什么指导意义呢？</p>
<ol>
<li>避免频繁使用 style，而是采用修改<code>class</code>的方式。</li>
<li>使用<code>createDocumentFragment</code>进行批量的 DOM 操作。</li>
<li>对于 resize、scroll 等进行防抖/节流处理。</li>
<li>添加 will-change: tranform ，让渲染引擎为其单独实现一个图层，当这些变换发生时，仅仅只是利用合成线程去处理这些变换，而不牵扯到主线程，大大提高渲染效率。当然这个变化不限于<code>tranform</code>, 任何可以实现合成效果的 CSS 属性都能用<code>will-change</code>来声明。</li>
</ol>
