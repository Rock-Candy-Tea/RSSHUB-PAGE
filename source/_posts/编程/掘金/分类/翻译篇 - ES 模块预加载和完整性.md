<h2 id="user-content-h" data-id="heading-1"><span>生产环境中模块的优化</span></h2>
<p>当在生产环境中使用ES模块时，目前有两种主要的性能优化要应用-代码拆分和预加载。</p>
<p>代码拆分优化可用于打包器（如 esbuild 或 RollupJS）中的原生 ES 模块。代码拆分确保对于任何两个总是被一起加载的模块，它们将始终被内联到同一个模块文件中，作为网络优化的块模块（或者甚至在可能的情况下，会被内联到入口模块本身中）。</p>
<p>然后预加载解决了模块依赖图中模块按引用顺序加载的问题 - 模块仅在静态导入图中的每个模块都加载后才执行，模块仅在其父模块加载后才加载。参考：<a href="https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Fhuangyangquang%2FLatest-technology-tracking%2Fblob%2Fmain%2FECMAScript%2FES%2520Module%25E5%258E%259F%25E7%2590%2586%2F%25E5%25AE%259E%25E6%2593%258D%25E6%258E%25A2%25E7%25A9%25B6%25E4%25B9%258BESM%25E5%25BC%2595%25E5%2585%25A5%25E9%25A1%25B6%25E7%25BA%25A7await%25E5%2589%258D%25E5%2590%258E%25E6%25A8%25A1%25E5%259D%2597%25E7%259A%2584%25E6%2589%25A7%25E8%25A1%258C%25E9%25A1%25BA%25E5%25BA%258F.md" target="_blank" rel="nofollow noopener noreferrer" title="https://github.com/huangyangquang/Latest-technology-tracking/blob/main/ECMAScript/ES%20Module%E5%8E%9F%E7%90%86/%E5%AE%9E%E6%93%8D%E6%8E%A2%E7%A9%B6%E4%B9%8BESM%E5%BC%95%E5%85%A5%E9%A1%B6%E7%BA%A7await%E5%89%8D%E5%90%8E%E6%A8%A1%E5%9D%97%E7%9A%84%E6%89%A7%E8%A1%8C%E9%A1%BA%E5%BA%8F.md" ref="nofollow noopener noreferrer">实操探究之ESM引入顶级await前后模块的执行顺序</a></p>
<h2 id="user-content-h-1" data-id="heading-2"><span>模块预加载</span></h2>
<p>ES 模块的预加载在浏览器中如下写法：</p>
<pre><code class="hljs xml copyable"><span class="hljs-tag"><<span class="hljs-name">link</span> <span class="hljs-attr">rel</span>=<span class="hljs-string">"modulepreload"</span> <span class="hljs-attr">href</span>=<span class="hljs-string">"..."</span>/></span> <br><span class="copy-code-btn">复制代码</span></code></pre>
<p>当它首次在 Chrome 中发布时，<a href="https://link.juejin.cn/?target=https%3A%2F%2Fdevelopers.google.com%2Fweb%2Fupdates%2F2017%2F12%2Fmodulepreload" target="_blank" rel="nofollow noopener noreferrer" title="https://developers.google.com/web/updates/2017/12/modulepreload" ref="nofollow noopener noreferrer">Google Developers 2017</a> 更新中有一篇关于它的精彩文章。</p>
<blockquote>
  <p>建议尽可能为所有深度依赖项注入 modulepreload 标签，以便完全消除模块加载的延迟成本，这同时也是是静态预加载的主要好处。</p>
</blockquote>
<p>【使用 modulepreload 的另一个主要好处是，它是目前使用“integrity”属性支持所有加载模块完全完整性的唯一机制。】（不太明白,等我查找资料）</p>
<p>比如app.js加载dependency.js加载library.js，我们可以这样写：</p>
<pre><code class="hljs xml copyable"><span class="hljs-tag"><<span class="hljs-name">link</span> <span class="hljs-attr">rel</span>=<span class="hljs-string">"modulepreload"</span> <span class="hljs-attr">href</span>=<span class="hljs-string">"/src/app.js"</span> <span class="hljs-attr">integrity</span>=<span class="hljs-string">"sha384-Oe38ELlp8iio2hRyQiz2P4Drqc+ztA7jb7lONj7H3Cq+W88bloPxoZzuk6bHBHZv"</span>/></span><br><br><span class="hljs-tag"><<span class="hljs-name">link</span> <span class="hljs-attr">rel</span>=<span class="hljs-string">"modulepreload"</span> <span class="hljs-attr">href</span>=<span class="hljs-string">"/src/dependency.js"</span> <span class="hljs-attr">integrity</span>=<span class="hljs-string">"sha384-kjKb2aJJUT956WSU7Z0EF9DZyHy9gdvPOvIWbcEGATXKYxJfkEVOcuP1q20GT2LO"</span>/></span><br><br><span class="hljs-tag"><<span class="hljs-name">link</span> <span class="hljs-attr">rel</span>=<span class="hljs-string">"modulepreload"</span> <span class="hljs-attr">href</span>=<span class="hljs-string">"/src/library.js"</span> <span class="hljs-attr">integrity</span>=<span class="hljs-string">"sha384-Fwh0dF5ROSVrdd/vJOvq0pT4R6RLZOOvf6k+ijkAyUtwCP7B0E3qHy8wbot/ivfO"</span>/></span><br><br><span class="hljs-tag"><<span class="hljs-name">script</span> <span class="hljs-attr">type</span>=<span class="hljs-string">"module"</span> <span class="hljs-attr">src</span>=<span class="hljs-string">"/src/app.js"</span>></span><span class="undefined"></span><span class="hljs-tag"></<span class="hljs-name">script</span>></span><br><span class="copy-code-btn">复制代码</span></code></pre>

<h2 id="user-content-h-3" data-id="heading-5"><span>完整性限制</span></h2>
<p>使用 modulepreload 作为主要完整性方法存在一个主要问题是：没有一个简单的方法，可以在不立即执行预加载的情况下，为延迟加载的模块预先提供完整性。对于现在的生产工作方式，最好的方法是构建一个自定义的动态导入包装器，在触发动态导入之前延迟注入预加载。</p>
<p>尽管采用这样办法，这项工作的工作量也不小，而且可能会带来比较大的处突，如果有人采用这样的方法，我会感到惊讶。然而，模块完整性是使用 ES 模块 CDN 的绝对关键特性。</p>
<p>在未来，可能与模块预加载 和 完整性相关的规范包括：</p>
<ul>
<li><p>Import assertions：已建议使用 import 'mod' assert &#123;integrity: '…' &#125; 语法，但尚未为 import assertions 进行规定或实现。</p>
<p>不幸的是，此功能存在以下问题：它取消了导入映射的主要性能优势，即允许图形中的所有模块有更长失效时间可以独立缓存。因此，虽然它对某些特定情况很有用，但作为这个问题的一般解决方案，它会倒退一步。</p></li>
<li><p>Import Map Integrity：我已经为导入地图建议了一个完整性属性规范，允许它们在这里充当编排点。 这里的困难是从浏览器中获得支持，但到目前为止还没有成功。</p></li>
<li><p>Milestones： 这是一个实验性的性能方法，使用当前的 Proposal Doc 和 Chrome CL，旨在允许指定一些关于脚本加载条件的信息，以便进行更细粒度的加载优化。不幸的是，目前没有计划为预加载支持此功能，因此这意味着它无法以当前形式解决延迟模块加载的深度图内容完整性问题。</p></li>
<li><p>Lazy Preloads：另一种设计可能是在预加载标签上有一个属性来指示它不应被预加载，但其完整性值仍应适用。 我在 WebAppSec 上提出了这个建议，但这里的预加载和完整性的组合似乎有些混乱。</p></li>
<li><p>Web Bundles： 从常见问题解答中，似乎 Web Bundle 的当前完整性状态由于哈希验证的高带宽成本而被视为后续提案。</p></li>
</ul>
<p>也许通用模块完整性的概念可以折叠成一个集中的 Web 完整性清单，可能与安全清单中的其他权限/安全功能相结合。相反，验证可以从我们用于优化完整性的广泛方法中受益，包括针对分块边界优化的默克尔树，甚至更奇特的东西。</p>
<h2 id="user-content-h-4" data-id="heading-6"><span>号召一起行动</span></h2>
<p>上网的人应该能够访问互联网上的网站，并且所有执行的代码都经过完整性哈希验证，这一基本原则是一项绝对基本的安全属性。</p>
<p>以最佳方式完成这项工作需要一些新的工作，我们需要积极与浏览器供应商和标准机构合作，以确保可以为未来的 Web 平台完全轻松地启用此安全属性，而不会出现摩擦。</p>
