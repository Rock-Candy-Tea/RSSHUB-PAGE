<h1 data-id="heading-0">背景</h1>
<h2 data-id="heading-1">埋点体系构成</h2>
<p>一般来说，一个完整的埋点体系由以下三个部分构成：</p>
<ul>
<li>应用</li>
<li>数据分析平台</li>
<li>数据平台 SDK</li>
</ul>
<p>埋点上报是将应用层事件上传至上层平台的过程。比方说，在某购物网站上，用户点击了「收藏」按钮，此时，一个点击事件就生成了，这一事件会被上报至一个数据分析平台。这样，相关的数据分析师、产品经理、运营等同学便可以在数据分析平台，通过这些上报的事件数据分析，得出应用中可以优化的方方面面。由此可见，埋点上报是每个产品走向卓越的重要一环。</p>
<p>通过以上描述，我们认识了埋点上报过程的两大主角：应用与数据分析平台。从前端技术的角度来说，我们通常还需要第三个角色的助攻，那就是数据平台 SDK. 这个 SDK 封装了数据分析平台的各种接口，暴露出简单的方法让我们进行调用，实现简易的埋点上传。</p>
<h2 data-id="heading-2">两种埋点事件</h2>
<p>我们可以把应用层事件分为两大类：</p>
<ul>
<li>「页面事件」：一种是通用的「页面事件」，比如说用户在应用某个页面的停留及活跃时长，我们希望这种全局的埋点只用在项目初始化时注入一次，不需要在代码中进行维护。</li>
<li>「触发事件」：另一种则是自定义的「触发事件」，比如点击某个特定的按钮，开启某个特定的流程，这种事件需要前端同学在代码中手动注入埋点。</li>
</ul>
<p>我们为这两种事件分别开发了一套埋点上传 SDK。下面，我们就来详细地讲解一下这两套 SDK 的技术知识。</p>
<h1 data-id="heading-3">处理「页面事件」的 SDK - <code>monitor-tracer</code></h1>
<blockquote>
<p><code>monitor-tracer</code> 是一个用来监控页面及组件可见时长和活跃时长的前端 SDK，同时也是 Monitor 前端埋点体系的一个核心组成部分。</p>
</blockquote>
<h2 data-id="heading-4">背景</h2>
<p>为了更好地理解用户对各个业务功能的使用状况，从而进行相应的产品优化和调整：</p>
<ul>
<li>
<p>对于一般的网页应用，我们需要对用户在应用某个页面的停留及活跃时长进行相应的统计；</p>
</li>
<li>
<p>对于大盘 / 看板 / dashboard 类型的页面（如下图所示），我们希望在页面维度的基础上，更进一步地统计每个组件对用户的可见时长，从而对它们的排列顺序和内容进行优化。</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/232f6967463945869969419a4a5f0a7b~tplv-k3u1fbpfcp-zoom-1.image" alt="一个 dashboard 类页面" loading="lazy" referrerpolicy="no-referrer"></p>
</li>
</ul>
<p>基于以上需求，我们开发了 <code>monitor-tracer</code> SDK, 旨在实现对 <strong>「页面可见、活跃时长」</strong> 及 <strong>「组件可见时长」</strong> 的统计。</p>
<h2 data-id="heading-5">名词解释</h2>
<ul>
<li><strong>页面 (Page)</strong> - 在浏览器中打开的网页，不同页面以路径 <code>location.pathname</code> 来作区分；</li>
<li><strong>页面可见时长</strong> - 一个页面对用户可见的累计时长；</li>
<li><strong>页面活跃时长</strong> - 用户在页面上进行有效的鼠标、键盘及触控活动的累计时长；</li>
<li><strong>组件 (Component)</strong> - DOM 元素的集合，是页面的组成部分。一个页面内可包含多个组件；</li>
<li><strong>组件可见时长</strong> - 一个组件对用户可见的累计时长。</li>
</ul>
<p>其关系为：</p>
<ul>
<li>页面活跃时长 ≤ 页面可见时长；</li>
<li>组件可见时长 ≤ 页面可见时长；</li>
<li>一个页面不可见时，则一定不活跃，且其中的所有组件一定也都不可见。</li>
</ul>
<h2 data-id="heading-6">页面可见及活跃时长统计</h2>
<p>在我们的设计中，衡量一个页面的停留及活跃时长需要两个重要的指标：</p>
<ul>
<li>
<p><strong>可见性 (visibility)</strong></p>
<ul>
<li><code>visible</code> - 页面在当前浏览器的 viewport 中，且浏览器窗口未被最小化；</li>
<li><code>invisible</code> - 页面不在当前浏览器的 viewport 中，或因浏览器最小化导致其无法被看到。</li>
</ul>
</li>
<li>
<p><strong>活跃性 (activity)</strong></p>
<ul>
<li><code>active</code> - 用户在网页中有活动（例如鼠标、键盘活动及页面滚动等）；</li>
<li><code>inactive</code> - 用户在网页中没有任何活动。</li>
</ul>
</li>
</ul>
<p>只要能获取到这四种状态发生的时间戳，就可以按下图所示方法，累加计算出页面从载入到退出的可见和活跃时长：</p>

<h3 data-id="heading-7">获取页面可见性数据</h3>
<h4 data-id="heading-8">Web Lifecycle</h4>
<p>Google 公司在 2018 年 7 月份提出了一套用来描述网页生命周期的 <a href="https://developers.google.com/web/updates/2018/07/page-lifecycle-api" target="_blank" rel="nofollow noopener noreferrer">Page Lifecycle API</a> 规范，本 SDK 便是基于这套规范来监听页面可见性变化的。规范指出，一个网页从载入到销毁的过程中，会通过浏览器的各种事件在以下六种生命周期状态 (Lifecycle State) 之间相互转化。</p>

































<table><thead><tr><th><strong>生命周期状态</strong></th><th><strong>描述</strong></th></tr></thead><tbody><tr><td><code>active</code></td><td>网页可见，且具有焦点</td></tr><tr><td><code>passive</code></td><td>网页可见，但处于失焦状态</td></tr><tr><td><code>hidden</code></td><td>网页不可见，但未被浏览器冻结，一般由用户切换到别的 tab 或最小化浏览器触发</td></tr><tr><td><code>frozen</code></td><td>网页被浏览器冻结（一些后台任务例如定时器、fetch 等被挂起以节约 CPU 资源）</td></tr><tr><td><code>terminated</code></td><td>网页被浏览器卸载并从内存中清理。一般用户主动将网页关闭时触发此状态</td></tr><tr><td><code>discarded</code></td><td>网页被浏览器强制清理。一般由系统资源严重不足引起</td></tr></tbody></table>

<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/84853d397a8045008a9a013cc8db21ba~tplv-k3u1fbpfcp-zoom-1.image" alt="生命周期状态转化" loading="lazy" referrerpolicy="no-referrer"></p>
<p>由以上信息，我们可以得出页面生命周期状态和页面可见状态之间的映射关系：</p>

<p>因此，我们只需要监听页面生命周期的变化并记录其时间，就可以相应获取页面可见性的统计数据。</p>
<h4 data-id="heading-9">监听页面生命周期变化</h4>
<p>在制定 Page Lifecycle 规范的同时，Google Chrome 团队也开发了 <a href="https://github.com/GoogleChromeLabs/page-lifecycle" target="_blank" rel="nofollow noopener noreferrer"><code>PageLifecycle.js</code></a> SDK, 实现了这套规范中描述的生命周期状态的监听，并兼容了 IE 9 以上的所有浏览器。出于易用性及稳定性的考虑，我们决定使用这个 SDK 来进行生命周期的监听。由于 <code>PageLifecycle.js</code> 本身使用 JavaScript 编写，我们为其添加了类型定义并封装为了兼容 TypeScript 的 <a href="https://bnpm.bytedance.net/package/@byted-cg/page-lifecycle-typed" target="_blank" rel="nofollow noopener noreferrer"><code>@byted-cg/page-lifecycle-typed</code></a> SDK. <code>PageLifecycle.js</code> 的使用方法如下：</p>
<pre><code class="hljs language-jsx copyable" lang="jsx"><span class="hljs-keyword">import</span> lifecycleInstance, &#123;
  StateChangeEvent,

<p>通过 <code>PageLifecycle.js</code>, 我们便可以监听 <code>statechange</code> 事件来在页面生命周期发生变化时获得通知，并在生命周期状态为 <code>active</code> 和 <code>passive</code> 时标记页面为 <code>visible</code> 状态，在生命周期状态为其他几个时标记页面为 <code>invisible</code> 状态，更新最后一次可见的时间戳，并累加页面可见时间。</p>
<h4 data-id="heading-10"><code>PageLifecycle.js</code> 的缺陷</h4>
<p>对于我们的需求来说，<code>PageLifecycle.js</code> 本身存在如下两个缺陷。我们也针对这两个缺陷做了一些改进。</p>
<ul>
<li>
<p><strong>无法监控到单页应用 (SPA) 中页面的变化</strong></p>
<p>在单页应用中，页面通常依靠 history 或 hash 路由的变化来切换，页面本身并不会重新载入，因此 <code>PageLifecycle.js</code> 无法感知页面的切换。为了应对这种情况，我们在 monitor-tracer 中手动添加了针对路由变化事件（<code>popstate</code> <code>replacestate</code> 等）的监听。如果发现页面的路由发生了变化，会认为当前页面进入了 <code>terminated</code> 生命周期，从而进行相应的处理。这里对路由变化进行监听的逻辑复用了我们之前开发的 <a href="https://bnpm.bytedance.net/package/@byted-cg/puzzle-router" target="_blank" rel="nofollow noopener noreferrer">@byted-cg/puzzle-router</a> SDK, 有兴趣的同学可以参考。</p>
</li>
<li>
<p><strong>无法捕获 <code>discarded</code> 生命周期</strong></p>
<p><code>discarded</code> 生命周期发生在网页被浏览器强制清除时。此时网页已经被销毁并从内存中清理，无法向外传递任何事件，因此 <code>PageLifecycle.js</code> 也就无法推送 <code>discarded</code> 事件。这种情况一旦发生，就会造成被清除的网页统计数据的丢失。为了应对此场景，<code>monitor-tracer</code> 会在页面进入 <code>invisible</code> 状态时，将现有的页面时长统计数据存储使用 <code>JSON.stringify</code> 序列化并储存在 <code>localStorage</code> 当中。如果页面恢复 <code>visible</code> 状态，则会把 <code>localStorage</code> 中的数据清空；而如果页面被清除，则会在下一次进入页面时先将 <code>localStorage</code> 中存储的上一个页面的数据通过事件推送出去。这样就最大程度地保证了页面即使被强制清除，其数据也能被送出而不至丢失。</p>
</li>
</ul>
<h3 data-id="heading-11">获取页面活跃性数据</h3>
<p>相较于页面可见性，页面活跃性的判断要更加直截了当一些，直接通过以下方法判断页面状态为 <code>active</code> 还是 <code>inactive</code> 即可。</p>
<h4 data-id="heading-12"><code>active</code> 判断标准</h4>
<p>通过监听一系列的浏览器事件，我们就可以判断用户是否在当前页面上有活动。<code>monitor-tracer</code> 会监听以下的六种事件：</p>

































<table><thead><tr><th><strong>事件</strong></th><th><strong>描述</strong></th></tr></thead><tbody><tr><td><code>keydown</code></td><td>用户敲击键盘时触发</td></tr><tr><td><code>mousedown</code></td><td>用户点击鼠标按键时触发</td></tr><tr><td><code>mouseover</code></td><td>用户移动鼠标指针时触发</td></tr><tr><td><code>touchstart</code></td><td>用户手指接触触摸屏时触发（仅限触屏设备）</td></tr><tr><td><code>touchend</code></td><td>用户手指离开触摸屏时触发（仅限触屏设备）</td></tr><tr><td><code>scroll</code></td><td>用户滚动页面时触发</td></tr></tbody></table>
<p>一旦监听到以上事件，<code>monitor-tracer</code> 就会将页面标记为 <code>active</code> 状态，并记录当前时间戳，累加活跃时长。</p>
<h4 data-id="heading-13"><code>inactive</code> 判断标准</h4>
<p>以下两种情况下，页面将会被标记为 <code>inactive</code> 状态：</p>
<ul>
<li>超过一定的时间阈值（默认为 30 秒，可在初始化 SDK 时自定义）没有监测到表示页面活跃的六种事件；</li>
<li>页面状态为 <code>invisible</code>. 因为如果页面对用户不可见，那么它一定是不活跃的。</li>
</ul>
<p>页面被标记为 <code>inactive</code> 后，<code>monitor-tracer</code> 会记录当前时间戳并累加活跃时长。</p>
<h2 data-id="heading-14">组件可见时长统计</h2>
<p>统计组件级别活跃时长需要两个条件，一是拿到所有需要统计的 DOM 元素，二是对这些 DOM 元素按照一定的标准进行监控。</p>
<h3 data-id="heading-15">获取需要统计的 DOM 元素</h3>
<h4 data-id="heading-16">监听 DOM 结构变化</h4>
<p>获取 DOM 元素要求我们对整个 DOM 结构的改变进行监控。<code>monitor-tracer</code> 使用了 <a href="https://developer.mozilla.org/zh-CN/docs/Web/API/MutationObserver" target="_blank" rel="nofollow noopener noreferrer"><code>MutationObserver</code> API</a>, DOM 的任何变动，比如节点的增减、属性的变动、文本内容的变动，都可以通过这个 API 得到通知。</p>
<p>概念上，它很接近事件，可以理解为 DOM 发生变动就会触发 <code>MutationObserver</code> 事件。但是，它与事件有一个本质不同：事件是同步触发，也就是说，DOM 的变动立刻会触发相应的事件，而 <code>MutationObserver</code> 则是异步触发，DOM 的变动并不会马上触发，而是要等到当前所有 DOM 操作都结束才触发。</p>
<p>这样的设计是为了应对 DOM 变动频繁的特点，节省性能。举例来说，如果文档中连续插入 1000 个 <code><p></p></code> 标签，就会连续触发 1000 个插入事件，执行每个事件的回调函数，这很可能造成浏览器的卡顿。而 <code>MutationObserver</code> 完全不同，只在 1000 个标签都插入结束后才会触发，而且只触发一次。</p>
<p><code>MutationObserver</code> API 的用法如下：</p>

<h3 data-id="heading-19">判断 DOM 元素可见性</h3>
<p>对组件可见性的判断可分为三个维度：</p>
<ul>
<li>组件是否在浏览器 viewport 中 - 使用 <a href="https://developer.mozilla.org/zh-CN/docs/Web/API/IntersectionObserver" target="_blank" rel="nofollow noopener noreferrer"><code>IntersectionObserver</code> API</a> 判断；</li>
<li>组件样式是否可见 - 根据元素 CSS 的 <code>display</code> <code>visibility</code> 及 <code>opacity</code> 样式属性判断；</li>
<li>页面是否可见 - 根据页面可见性判断。</li>
</ul>
<h4 data-id="heading-20">判断组件是否在浏览器 viewport 中</h4>
<p>这里我们使用了 <a href="https://developer.mozilla.org/zh-CN/docs/Web/API/IntersectionObserver" target="_blank" rel="nofollow noopener noreferrer"><code>IntersectionObserver</code> API</a>. 该 API 提供了一种异步检测目标元素与祖先元素或 <a href="https://developer.mozilla.org/zh-CN/docs/Glossary/Viewport" target="_blank" rel="nofollow noopener noreferrer">viewport</a> 相交情况变化的方法。</p>
<p>过去，相交检测通常要用到事件监听，并且需要频繁调用 <a href="https://developer.mozilla.org/zh-CN/docs/Web/API/Element/getBoundingClientRect" target="_blank" rel="nofollow noopener noreferrer"><code>Element.getBoundingClientRect</code></a> 方法以获取相关元素的边界信息。事件监听和调用 <code>Element.getBoundingClientRect</code> 都是在主线程上运行，因此频繁触发、调用可能会造成性能问题。这种检测方法极其怪异且不优雅。</p>
<p>而 <code>IntersectionObserver</code> API 会注册一个回调函数，每当被监视的元素进入或者退出另外一个元素时（或者 viewport），或者两个元素的相交部分大小发生变化时，该回调方法会被触发执行。这样，我们网站的主线程不需要再为了监听元素相交而辛苦劳作，浏览器会自行优化元素相交管理。</p>

<h4 data-id="heading-21">判断组件 CSS 样式是否可见</h4>
<p>如果元素的 CSS 样式设为了 <code>visibility: hidden</code> 或 <code>opacity: 0</code>，那么即使其与 viewport 的相交比例为 1，对用户来说也是不可见的。因此我们需要额外判断目标元素的 CSS 属性是否为可见。</p>
<p>如果一个组件的样式被设置为了以下之一，那么它就会被标记为 <code>invisible</code>.</p>
<ul>
<li><code>visibility: hidden</code></li>
<li><code>display: none</code></li>
<li><code>opacity: 0</code></li>
</ul>
<h4 data-id="heading-22">判断页面是否可见</h4>
<p>当页面不可见时，所有组件自然都不可见，因此在页面为 <code>invisible</code> 的状态下，<code>monitor-tracer</code> 会将需监控的所有组件的状态也标记为 <code>invisible</code>。</p>
<h1 data-id="heading-23">处理「触发事件」的 SDK - <code>monitor</code></h1>
<h2 data-id="heading-24"><code>monitor</code> SDK 的定位</h2>
<blockquote>
<p>数据平台 SDK 的单一的埋点上报方式，无法满足我们开发中对 clean code 的极致追求</p>
</blockquote>
<p>数据平台的 SDK 往往只提供了上报埋点的函数式方法，虽然可以满足我们的日常开发需求，但是并不能解决我们在写埋点代码时的两大痛点：</p>
<ul>
<li>只能逐个进行埋点上报</li>
<li>埋点逻辑与业务逻辑的耦合</li>
</ul>
<p>我们希望埋点代码可以轻易地添加、修改与删除，并且对业务代码没有影响。因此，我们基于 TypeScript 开发对框架无感的 <code>monitor</code> SDK. 它支持逐个上传多个埋点，并且接受返回埋点的函数，将其返回值上报；它提供了三种方式注入埋点，覆盖了所有场景，将埋点与业务代码完全分离。</p>
<p>可以看出，<code>monitor</code> 既是一个数据处理器，又是一个方法库。更直观一些，使用 <code>monitor</code> 后，我们的应用在上报埋点时的流程如下：</p>
<p><img src="http://tosv.byted.org/obj/cg-fe/monitor/monitor_doc/%E5%9F%8B%E7%82%B9%E4%B8%8A%E6%8A%A5%E8%B7%AF%E5%BE%84%E7%A4%BA%E6%84%8F%E5%9B%BE.png" alt="埋点上报流程" loading="lazy" referrerpolicy="no-referrer"></p>
<p>埋点由应用层发送给 <code>monitor</code> 后，<code>monitor</code> 首先会对数据进行处理，再调用数据平台 SDK, 将埋点事件上报给数据平台。</p>
<p>在对 <code>monitor</code> 有了初步了解后，这篇文章将主要讲解 <code>monitor</code> 是如何通过以下三种埋点注入的方式，解耦业务逻辑与埋点逻辑的。</p>

<h4 data-id="heading-30">从 <code>@readonly</code> 理解装饰器原理</h4>
<p>装饰器是如何实现将埋点逻辑和业务逻辑相整合的呢？在我们详细解读 <code>@monitorBefore</code> 之前，让我们先从一个常用的装饰器 <code>@readonly</code>讲起吧。</p>
<p>装饰器应用于一个类的单个成员上，包括类的属性、方法、getters 和 setters. 在被调用时，装饰器函数会接收 3 个参数：</p>
<ul>
<li><code>target</code> - 装饰器所在的类</li>
<li><code>name</code> - 被装饰的函数的名字</li>
<li><code>descriptor</code> - 被装饰的函数的属性描述符</li>
</ul>

  <span class="hljs-built_in">console</span>.log(descriptor);
  descriptor.writable = <span class="hljs-literal">false</span>;
  <span class="hljs-keyword">return</span> descriptor;

<ul>

<pre><code class="hljs language-typescript copyable" lang="typescript">f = <span class="hljs-function">(<span class="hljs-params">target: <span class="hljs-built_in">object</span>, name: <span class="hljs-built_in">string</span>, descriptor: <span class="hljs-built_in">object</span></span>) =></span>

<p>而后，编译器又调用函数 <code>f</code>, 把当前的类、被装饰的函数名称与其属性描述符传给<code>f</code>. 函数 <code>f</code> 返回的 <code>this.defineDecorator(event, descriptor, this.before)</code> 会被解析为一个新的 <code>descriptor</code> 对象，它的 <code>value</code> 会在运行时被调用，也就是说会在 <code>onSearch</code> 被触发时所调用。</p>
<p>现在，让我们详细解读 <code>defineDecorator</code> 函数是如何改变生成一个新的 <code>descriptor</code> 的吧。</p>
<pre><code class="hljs language-typescript copyable" lang="typescript"><span class="hljs-comment">// defineDecorator 函数源代码</span>

<p><code>defineDecorator</code> 函数接收三个参数：</p>
<ul>
<li><code>event</code> - 需要上报的埋点；</li>
<li><code>descriptor</code> - 被装饰的函数的属性描述符；</li>
<li><code>decorator</code> - 一个高阶函数。它接收一个埋点事件与一个回调，返回一个函数，用来进行埋点上报，并而后执行回调。</li>
</ul>
<p><code>decorator</code> 首先返回了一个函数 <code>wrapperFn</code>. 在被调用时，<code>wrapperFn</code> 会先上报埋点，然后执行 <code>descriptor.value</code> 的逻辑，即被装饰的函数。</p>
<p>在 <code>defineDecorator</code> 的 <code>composedFn</code> 中, 我们用 <code>wrapperFn.apply(this, arguments)</code> 将调用被装饰的函数时传入的参数透传给 <code>wrapperFn</code>。</p>
<p>最后，我们将 <code>composedFn</code> 设为 <code>descriptor.value</code>, 这样，我们就成功生成了一个融合了埋点逻辑与业务逻辑的新函数。</p>
<p>装饰器埋点注入方式十分整洁，能够清晰地与业务代码区分。不论是增添、修改还是删除埋点，都无需顾虑会对业务代码造成改动。但是其局限性也是显而易见的，装饰器只能用于类组件，现在我们常用的函数式组件是无法使用装饰器的。</p>

<h2 data-id="heading-33">小结</h2>
<p>上述三种埋点方式，覆盖了所有使用场景。不论你是用 React, Vue, 还是原生 JavaScript, 不论你是使用类组件，还是函数式组件，不论你的埋点是否需要复杂的前置逻辑，<code>monitor</code> SDK 都提供了适合你的场景的使用方式。</p>
