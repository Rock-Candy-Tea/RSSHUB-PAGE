h2&#123;font-size:20px&#125;.markdown-body h3&#123;font-size:18px&#125;&#125;</style><h1 data-id="heading-0">内存泄漏</h1>
<p>系统进程不再用到的内存<strong>没有释放</strong>，就叫做内存泄漏（memory leak）。当内存占用越来越高，轻则影响系统性能，重则导致进程崩溃。</p>
<blockquote>
<p>目前没有找到能够完美检测内存泄漏的工具和排查方法（尤其是大项目），因此本文主要提供一些内存泄漏的可能原因，以及常用的一些排查方法。</p>
</blockquote>
<h3 data-id="heading-1">垃圾回收机制</h3>
<p>浏览器的垃圾回收机制主要有标记清理和引用计数两种，目前绝大多数都是标记清除。</p>
<p>要注意大多数情况下垃圾回收都不是实时的，浏览器更倾向于在程序比较"清闲"的时候进行这个过程。</p>
<p><strong>标记清除</strong><br>
如下图所示，触发garbage collect 时，会遍历数据，然后将无法访问的数据当作垃圾回收</p>
<p><img alt="垃圾回收.png" class="lazyload" src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/d29fd5528e2c499b886326a003f90288~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p><strong>引用计数</strong></p>
<blockquote>
<p>另一种不常用的垃圾回收策略是引用计数（reference counting）。其思路是对每个值都记录它被引用的次数。</p>
</blockquote>
<blockquote>
<p>声明变量并给它赋一个引用值时，这个值的引用数为1。如果同一个值又被赋给另一个变量，那么引用数加1。类似地，如果保存对该值引用的变量被其他值给覆盖了，那么引用数减1。当一个值的引用数为0 时，就说明没办法再访问到这个值了，因此可以安全地收回其内存了。垃圾回收程序下次运行的时候就会释放引用数为0 的值的内存。</p>
</blockquote>
<p>不过引用计数有非常大的问题：如果是互相引用，次数也会叠加，如下面的函数，变量a会被记成2次引用，即使函数执行完了，a也不会被当成垃圾回收。</p>
<pre><code class="hljs language-js copyable" lang="js"><span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">fn</span>(<span class="hljs-params"></span>)</span>&#123;
  <span class="hljs-keyword">var</span> a=<span class="hljs-number">1</span>
  <span class="hljs-keyword">var</span> b=a
&#125;
fn()

<h1 data-id="heading-2">主要原因</h1>
<h3 data-id="heading-3">全局变量</h3>
<p>在非严格模式下当引用未声明的变量时，会在全局对象中创建一个新变量。在浏览器中，全局对象将是window。</p>
<pre><code class="hljs language-js copyable" lang="js"><span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">foo</span> (<span class="hljs-params">arg</span>) </span>&#123; 
    bar = <span class="hljs-string">"some text"</span>; <span class="hljs-comment">// bar将泄漏到全局.</span>
&#125;
<span class="copy-code-btn">复制代码</span></code></pre>
<p>一般可以通过严格模式或者代码校验来解决（如eslint）</p>
<blockquote>
<p>console与之类似，console打印的信息可以在控制台查看，其输出的对象不会被释放。可以利用webpack，code-review，eslint等方式避免无用的console语句。</p>
</blockquote>
<h3 data-id="heading-4">闭包</h3>
<p>闭包可能是内存泄漏最大的嫌疑犯，其定义如下：</p>
<blockquote>
<p>声明在函数内部，可以访问函数的局部变量的函数<br>
1.可以读取函数内部的变量。<br>
2.让这些变量的值始终保持在内存中。</p>
</blockquote>
<pre><code class="hljs language-js copyable" lang="js"><span class="hljs-keyword">var</span> count=<span class="hljs-number">10</span>;
<span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">add</span>(<span class="hljs-params"></span>)</span>&#123;
    <span class="hljs-keyword">var</span> count=<span class="hljs-number">0</span>;
    <span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">inner</span> (<span class="hljs-params"></span>)</span>&#123;
        count+=<span class="hljs-number">1</span>;
        alert(count);
    &#125;
    <span class="hljs-keyword">return</span> inner
&#125;
<span class="hljs-keyword">var</span> s=add();
s();<span class="hljs-comment">//输出1</span>
s();<span class="hljs-comment">//输出2</span>
<span class="copy-code-btn">复制代码</span></code></pre>
<p>原因：闭包可以读取函数内部的变量，然后让这些变量始终保存在内存中。如果在<strong>使用结束后</strong>没有将局部变量清除，就可能导致内存泄露。</p>
<p><strong>注意: 闭包本身没有错,不会引起内存泄漏.而是使用错误导致.</strong></p>
<h3 data-id="heading-5">没有清理的 DOM 元素引用</h3>
<p>原因：虽然别的地方删除了，但是对象中还存在对 dom 的引用。</p>
<pre><code class="hljs language-js copyable" lang="js"><span class="hljs-comment">// 在对象中引用DOM</span>
<span class="hljs-keyword">var</span> elements = &#123;
  <span class="hljs-attr">btn</span>: <span class="hljs-built_in">document</span>.getElementById(<span class="hljs-string">'btn'</span>),
&#125;
<span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">doSomeThing</span>(<span class="hljs-params"></span>) </span>&#123;
  elements.btn.click()
&#125;

<span class="hljs-function"><span class="hljs-keyword">function</span> <span class="hljs-title">removeBtn</span>(<span class="hljs-params"></span>) </span>&#123;
  <span class="hljs-comment">// 将body中的btn移除, 也就是移除 DOM树中的btn</span>
  <span class="hljs-built_in">document</span>.body.removeChild(<span class="hljs-built_in">document</span>.getElementById(<span class="hljs-string">'button'</span>))
  <span class="hljs-comment">// 但是此时全局变量elements还是保留了对btn的引用, btn还是存在于内存中,不能被GC回收</span>
&#125;
<span class="copy-code-btn">复制代码</span></code></pre>
<p>除上述代码外，也要考虑如果在清除dom前有子元素被引用，需要先将子元素的引用释放，否则该dom还是会在内存中无法释放。</p>
<p>有一个虚拟列表的组件在1.x版本就有该问题，感兴趣的可以看一下：<a href="https://github.com/tangbc/vue-virtual-scroll-list/issues/148" target="_blank" rel="nofollow noopener noreferrer">github.com/tangbc/vue-…</a></p>
<h3 data-id="heading-6">被遗忘的定时器或者回调</h3>
<p>定时器中有 dom 的引用，即使 dom 删除了，但是定时器还在，所以内存中还是有这个 dom。</p>
<p>以及组件已经销毁，但是定时器还在导致代码报错堵塞，也有可能会造成内存泄漏（不过这个就很明显了，毕竟控制台都报错了）</p>
<h3 data-id="heading-7">监听事件没有解绑</h3>
<p>window.addEventListener 之类的时间监听，绑在 EventBus 的事件.</p>
<p>例如在组件初始化时绑定监听，但是销毁时没有解绑，这样再次初始化时就会重复绑定监听，导致内存不断泄漏。</p>
<h3 data-id="heading-8">不当的代码操作</h3>
<p>例如（我见过的）：</p>
<ul>
<li>对于一个数组只有push而没有清理，导致该数组占用不断增大，但保存的基本都是废弃信息</li>
<li>（Vue）destroy拼写错误导致节点无法销毁</li>
</ul>
<h1 data-id="heading-9">排查方法</h1>
<h3 data-id="heading-10">工具</h3>
<p>chrome Memory</p>
<p><img alt="memory.png" class="lazyload" src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/5659a884346344ca9394daa5377aa370~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>主要用到两种模式：</p>
<ul>
<li>Heap snapshot - 用以打印堆快照，堆快照文件显示页面的 javascript 对象和相关 DOM 节点之间的内存分配</li>
<li>Allocation instrumentation on timeline - 在时间轴上记录内存信息，随着时间变化记录内存信息。</li>
</ul>
<p><strong>Heap snapshot</strong></p>
<p><img alt="企业微信截图_20210223181415.png" class="lazyload" src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/9ffbea80753e4880b591c069b36f3dee~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>一般用于在循环操作的不同次数后录制快照，然后进行比对。点击快照会显示当前快照的信息，点击左上角模式可以切换为对比模式，与其他的快照进行比较。</p>
<p><img alt="retaubers.png" class="lazyload" src="https://p9-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/f92a0c89637941beabd3e8b7471d3df0~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer">
点开每个变量，提供引用链retainers，会说明该变量被哪里引用。</p>
<p>需要注意的一点是Total JS heap size显示数值和Heap snapshot生成快照数值可能会有所不同，原因是（chrome）在调用快照时会自动触发一次gc，所以生成快照后数值和之前显示的统计值有差异。</p>
<p><strong>Allocation instrumentation on timeline</strong></p>
<p><img alt="timeline2.png" class="lazyload" src="https://p6-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/015f46492a9b418d88f5c9cad0404ce9~tplv-k3u1fbpfcp-watermark.image" data-width="800" data-height="600" referrerpolicy="no-referrer">
图中最为明显的信息为时间轴，记录了随着时间推移系统对内存的申请情况。每一根柱条都代表内存空间，灰色的代表已经释放，蓝色的代表仍在占用。</p>
<p>可以细化时间轴的范围，只显示某一次循环的内存情况。</p>
<p>由图显而易见，在两个页面状态之间循环切换，每次切换都有没有释放的内存，说明有内存泄漏的情况。</p>
<p>注意：由于该工具会占用较大性能，导致页面变卡。</p>
<h3 data-id="heading-11">判断是否存在内存泄漏</h3>
<ul>
<li>观察相邻的snapshot快照（项目较大时不好判断，且干扰很多）</li>
<li>观察内存时间轴是否在<strong>循环操作</strong>的过程中<strong>每次</strong>都有未释放的内存</li>
<li>使用脚本模拟长时间操作，观察内存占用数值（需要手动触发GC防止误判）</li>
</ul>
<p>注意：这个过程需要在线上环境进行，因为开发环境由于一些未知原因（可能是webpack热更新引用导致）在某些情况会出现内存泄漏，但实际上发布后并没有这个问题。（例如打包后使用http-server模拟线上环境进行验证）</p>
<h3 data-id="heading-12">如何解决内存泄漏</h3>
<p>内存泄漏的原因五花八门，没有万能的解决办法，但只要定位了原因，就能够针对性的进行修复（一般来说不会有结构性的问题），这个优化的主要难点也就是在定位上，往往是花数小时定位，然后花数分钟解决。</p>
<ul>
<li>观察内存记录看是否为已知的数据结构，是否经常出现某个组件或函数</li>
<li>观察控制台看是否有报错或警告信息</li>
<li>观察排查模块是否有定时或防抖函数设置，是否有绑定监听事件等</li>
<li>暴力定位（注释二分法）</li>
</ul>
<p>一般来说对项目代码越熟悉，就能越快的从海量的信息中找到问题所在的位置，但有时候也可能被一些无用的信息误导导致走一堆弯路，所以对代码完全不熟悉或者光是看引用链找不到问题的时候，暴力定位也不是不能接受————最起码好过钻牛角尖。</p>

<h1 data-id="heading-14">总结</h1>
<blockquote>
<p>在 Web 应用中查找和修复内存泄漏的状态仍然很初级。在本文中介绍了一些可能有用的技术和实例，但是这仍然是一个困难且耗时的过程。<br>
与大多数性能问题一样，少量预防胜过大量的治疗。进行综合测试是值得的，而不是在事实发生后尝试调试内存泄漏。尤其是如果页面上存在多个泄漏，则可能会变成洋葱剥皮练习——你先修复一个泄漏，然后查找另一个泄漏，然后重复（整个过程都在哭泣！）。<br>
当然，如果能在代码编写阶段就进行预防，那就更好了。<br>
