h2&#123;font-size:20px&#125;.markdown-body h3&#123;font-size:18px&#125;&#125;</style><p>今天来讲解统一语言，什么是统一语言呢？</p>
<p>统一语言就是大家都有共识的语言 —— 这不是废话么 —— 其实有些时候还真不是</p>
<p>事情远没有我们想的简单</p>
<p>比如：</p>
<ul>
<li>同一个数据结构在不同的编程语言中可能不同</li>
<li>同一种技术实现在不同编程语言甚至不同技术栈下也不同</li>
<li>即使语言相通，但是大家都不清楚双方的语言相通，可能会出现怀疑和疑惑，最后造成重复实现或者业务耦合</li>
</ul>
<p>举个简单的例子 —— IOC</p>
<p>react 下 叫状态逻辑复用 hooks，vue 下叫组合函数 composition，ng 下叫做 service</p>
<p>谁说得对？</p>
<p>其实大家说的都是一个东西，并没有区别，但是如果语言不统一，团队间就无法形成合作</p>
<p>不只是大家用的技术栈不一样，即使技术栈相同，如果开发经历不同，也可能造成语言不一致的情况</p>
<p>比如，大家都用 IOC 实现某个需求</p>
<p>但是 React 程序员喜欢遵从函数式开发的思想，用 reducer 传递 dispatch，而 ng 程序员喜欢组合服务</p>
<p>造成的后果是什么？</p>
<p>react 程序员可能会直接在服务中写下 ：</p>
<pre><code class="hljs language-js copyable" lang="js">Rx.reduce((state,reducer))  
action$.next(<span class="hljs-string">'new action'</span>) 
<span class="copy-code-btn">复制代码</span></code></pre>
<p>的代码，然后让 ng 程序员看得一头雾水（即便不是一头雾水，也会降低阅读效率）</p>
<p>再比如，React Vue 程序员都习惯组件级别注入，依托组件构建逻辑结构，但是这对 ng 程序员来说完全无法理解 —— 不是说优先通过模块组织功能结构么？</p>
<p>这又有了新的问题，对于 React Vue 程序员来说 —— 什么是模块？</p>
<p>这又会反过来影响 React Vue 程序员，比如 —— 组件中频繁写 Context.Provider / provide()好麻烦</p>
<hr>
<p>说了一下问题，但是这个问题怎么解决呢？</p>
<p>答案就是，<strong>统一语言</strong></p>
<h2 data-id="heading-0">我们用统一的语言，描述应用的核心功能，抛弃一切具体实现</h2>
<p>我们用提供者（提供商），代替 provider value 代替 provide() 代替 providers:[...]</p>
<p>我们用服务，代替 hooks 代替 composition 代替 service</p>
<p>我们用依赖注入，代替 useContext 代替 inject 代替 constructor(injection)</p>
<p>我们用模块，代替 服务+注入点组件构成的文件夹</p>
<p>我们用分形聚合 aggregation，代替子服务</p>
<p>用值对象，代替全局数据，领域静态数据</p>
<p>用领域事件，代替 dispatch，代替 injected ref 赋值，代替流</p>
<p>这样，我们就能取缔具体技术实现对我们的干扰，并且在团队中能够顺畅交流</p>
<p>这样的语言 ——</p>
<h2 data-id="heading-1">不分前后端终端和算法</h2>
<p>任何人，只要是程序员，都可以用这套语言进行交流</p>
<hr>
<p>而这还只是开始</p>
<p>下一步，我们要对业务语言进行约束，提供统一的 ——</p>
<h2 data-id="heading-2">限界上下文</h2>
<p>领域内为统一语言，而领域外又将命名权限给与其他系统</p>
<p>每个系统内部，语言一致</p>
<p>保证单机内，文件划分符合 LIFT 原则</p>
<p>保证多节点，节点命名空间也符合 LIFT 原则</p>
<p>其实业务开发工作的核心逻辑，并不复杂，不是么？</p></div> <div class="image-viewer-box" data-v-78c9b824><!----></div>
