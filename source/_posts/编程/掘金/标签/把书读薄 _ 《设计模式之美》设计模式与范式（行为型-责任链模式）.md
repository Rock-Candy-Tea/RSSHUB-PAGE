<h2 data-id="heading-0">0x0、引言</h2>
<p>😆 时间飞快，又到周五，继续啃设计模式之美，本文对应设计模式与范式：行为型(62-63)，<strong>责任链模式</strong> (Chain of Responsibility Pattern)，常用于框架开发中，为框架提供扩展点，让框架使用者在不修改框架源码的情况下，基于扩展点添加新的功能，具体点说，最常用来开发框架的 <strong>拦截器</strong> 和 <strong>过滤器</strong>。</p>

<h2 data-id="heading-1">0x1、定义</h2>
<p><strong>原始定义</strong></p>
<blockquote>
<p>将请求的发送和接收解耦，让多个接收对象都有机会处理这个请求。将这些接收对象串成一条链，并沿着这条链传递这个请求，知道链上某个接收对象能够处理它为止。</p>
</blockquote>
<p>定义看起来有点抽象，简单点就是：<strong>构建一个处理流水线来对一次请求进行多次处理</strong>。</p>
<p>还是不懂？没关系，写个简单例子帮助理解~</p>
<hr>

<blockquote>

<p>如果再来个疼爱孙子的爷爷，超过1000的可以找他报销，又得嵌套一个if-else，在一些复杂的实际业务场景，这样的写法可能要套十几层，对于这种流水线式加工的场景，其实可以使用责任链模式解耦~</p>
<blockquote>
<p>把伸手要钱这个行为看成一个 <strong>请求</strong>，你是 <strong>请求者</strong>，家人是 <strong>接收者</strong>，他们会按照特定的顺序：哥哥 → 粑粑 → 麻麻 → 爷爷 对你的请求进行处理，在自己额度范围内，就不往下走，不在就往下走，直到最后一个接收者为止。这个特定顺序可以看成一条 <strong>链</strong>，请求就是沿着这样的链往后传递~</p>
</blockquote>

<p>责任链模式解耦后，此时我们要再添加一个爷爷就很简单了，四步：</p>
<blockquote>
<p>继承AbstractHandler → 重写handleRequest() → 初始化爷爷实例 → motherHandler.setNextHandler()</p>
</blockquote>
<p>如果要再来个曾祖父或高曾祖父，也是这样操作，你可能会吐槽是不是有点过度设计，简单的if-else嵌套就能解决，这样会写多很多类，有点复杂化了。</p>
<blockquote>
<p>应用设计模式主要是为了：应付代码复杂性，让其满足开闭原则，提高代码扩展性。而将大块代码逻辑拆解成函数、大类拆分成小类，是应对代码复杂性的常用方法。</p>
</blockquote>
<p>此处使用责任链模式，把处理请求的函数拆分出来，设计成独立的类，简化了ChainTest类，使其不至于代码过多，过复杂。</p>
<p>客户端代码新增一个接收者，不需要去修改框架的代码，只需基于框架提供的扩展点进行扩展，可以说是：<strong>在框架代码范围内实现了开闭原则</strong>。</p>
<p>老规矩，带出UML类图、组成角色、使用场景及优缺点的总结</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/8f3c6ff514c344ea97f69f5e63ae07f1~tplv-k3u1fbpfcp-zoom-in-crop-mark:1956:0:0:0.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<ul>
<li><strong>Handler</strong> (抽象处理者) → 定义了处理请求的接口或抽象类，提供处理请求的方法和设置下一个处理者的方法；</li>
<li><strong>ConcreteHandler</strong> (具体处理者) → 抽象处理者的具体实现，按照链条顺序对请求进行具体处理；</li>
</ul>
<p><strong>使用场景</strong></p>
<ul>
<li>运行时需要动态使用多个关联对象来处理同一次请求，如编译打包发布流程；</li>
<li>不想让使用者知道具体的处理逻辑，如权限校验的登录拦截器；</li>
<li>需动态替换流程处理中的流程对象；</li>
<li>if-else多层嵌套或冗长的switch结构解耦；</li>
</ul>
<p><strong>优点</strong></p>
<ul>
<li>降低客户端(请求者)与处理链条上对象(接收者)间的耦合度；</li>
<li>动态组合，简化了对象前后关联处理的复杂性，只需存储一个指向后继者的引用；</li>
<li>扩展灵活，新增具体请求者时无需修改原有系统的代码，满足开闭原则；</li>
</ul>
<p><strong>缺点</strong></p>
<ul>
<li>产生许多细颗粒对象；</li>
<li>比较长的责任链，可能涉及多个处理对象，可能存在性能问题，及调试不方便；</li>
<li>建链不当，可能造成循环调用，导致死循环，进而导致堆栈溢出错误；</li>
</ul>
<p>另外，除了使用上述 <strong>递归的方式形成链条</strong> 实现责任链模式外，还可以使用数据结构(数组、列表、链表等) <strong>按顺序保存具体处理者实例</strong>，然后遍历的方式实现~</p>
<hr>
<h2 data-id="heading-3">0x3、责任链的 "纯与不纯"</h2>
<ul>
<li><strong>纯责任链</strong> → 要么承担全部责任，要么责任推给下家，不允许在某处承担了部分或全部责任，然后又把责任推给下家；</li>
<li><strong>不纯责任链</strong> → 责任在某处部分或全部处理后，还往下传递；</li>
</ul>
<hr>

<h3 data-id="heading-6">② 怎么组成一条链</h3>

<li>根据游标获取 <strong>当前的拦截器</strong>，调用拦截器的 <strong>intercept()</strong> 方法把新的Chain作为参数传入；</li>
</ul>
<p>这里其实就是 <strong><code>递归</code></strong>，先回到一开始的接口：</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/18b5a9f8ce6d41a3b14662c3bfab0113~tplv-k3u1fbpfcp-zoom-in-crop-mark:1956:0:0:0.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>责任链模式有两个要点：<strong>如何往后传递处理结果</strong> 和 <strong>何时结束传递</strong></p>
<p>第一点传递很好理解：</p>
<blockquote>
<p>Interceptor实现类调用 <strong>intercept</strong>(Chain) 往下传递Chain实例，实例中包含了<strong>拦截器列表</strong>、<strong>下一个拦截器的下标</strong>、<strong>处理后的request实例</strong> 等。</p>
</blockquote>
<p>第二点何时结束：</p>
<blockquote>
<p>最后一个拦截器调用 <strong><code>chain.proceed()</code></strong> 返回Response实例为止，而这个实例会往前传递，前面调用了 <strong><code>chain.proceed()</code></strong> 的拦截器可以获得Response实例并进行加工(如打日志)，第一个调用了 <strong><code>chain.proceed()</code></strong> 的拦截器得到的就是处理完后的Response。</p>
</blockquote>
<p>所以OkHttp的拦截器/过滤器是 <strong><code>双向</code></strong> 的，而Chain接口就是链接其中的链条：</p>
<blockquote>
<ul>
<li><strong>前 → 后</strong>：request()获得当前request实例，拦截器可对请求进行加工，可也调用intercept往下传递Chain；</li>
<li><strong>后 → 前</strong>：proceed()获得当前response实例，拦截器可对响应进行加工，作为返回值往上传；</li>
</ul>
</blockquote>
