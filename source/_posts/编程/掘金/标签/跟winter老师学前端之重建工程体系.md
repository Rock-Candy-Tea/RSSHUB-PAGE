<h3 data-id="heading-5">5. 成就</h3>
<p>怎么让你的 领域知识，能力，潜力 体现出来？ 那就是 "成就"！即你做的东西！</p>
<p>大家越容易了解的东西(如:美团APP,淘宝网站,京东网站...)！你在简历上就可以写得更少！</p>
<p>简历: </p>
<p><strong>做了什么项目，什么阶段做了什么具体内容，做到了什么水平</strong>！</p>
<p>简历显示的就是之前的职业经历获得的一个个成就。多写客观的东西！少写主观的东西，什么学习能力强！之类的就是废话！</p>
<h2 data-id="heading-6">二. 谁是你职业发展的负责人</h2>

<p>公司全是机械的做需求， 没有技术含量怎么办? </p>
<p>不一定在工作中才能成长！自己还可以找其他途径进行成长！如参加开源项目，参加付费课程！</p>
<h2 data-id="heading-7">三. 职业发展</h2>
<p>职业发展总纲！</p>
<pre><code class="hljs language-mermaid" lang="mermaid">
graph LR;
    晋升-->成长 --> 成就-->晋升
    
</code></pre>
<p>当获得一定成长后， 就能拿到一个更好成就的东西(技术变好了就能做更厉害的东西)，获得一定的成就会拿到更好的晋升(公司升级，拿到更好的offer)！这自然就形成了一个良性循环</p>
<p>分析自己在这个循环中 缺少哪一角! 然后在来考虑如何获得进步！</p>
<h3 data-id="heading-8">3.1 成就</h3>
<ul>
<li>业务型成就</li>

</ul>
<h4 data-id="heading-9">业务型成就是什么</h4>
<ol>
<li>业务目标
<ul>
<li>理解公司业务的核心目标</li>
<li>将目标转化为指标</li>
</ul>
</li>
<li>技术方案
<ul>
<li>业务指标到技术指标的转化</li>
<li>形成纸面方案，完成小规模实验</li>
</ul>
</li>
<li>实施方案
<ul>
<li>确定实施目标，参与人</li>
<li>管理实施进度</li>
</ul>
</li>
<li>结果评估
<ul>
<li>数据采集，数据报表</li>
<li>向上级汇报</li>
</ul>
</li>
</ol>
<p>业务型人才: 业务的KPI是什么？</p>
<p>首先要搞清楚业务目标，即公司的核心目标(公司是怎么挣钱的)，</p>
<p>有时候技术上是小事， 但是对于业务(如:用户交互) 的影响很大！因此做技术时也要站在业务的角度思考问题!</p>
<h4 data-id="heading-10">技术难题是什么</h4>
<p>技术难题应该是一个公认的技术难点！</p>
<ol>
<li>目标
<ul>
<li>公认的技术难题</li>
</ul>
</li>
<li>方案与实施
<ul>
<li>依靠扎实的编程能力，架构能力，形成解决方案</li>
</ul>
</li>
<li>结果
<ul>
<li>问题解决</li>
</ul>
</li>
</ol>
<p>就比如，一个前端实现问题！公司其他人都实现不了！ 但是你却实现了！</p>
<p>如：app 根据不同的环境进行热更新，实现app消息推送！</p>
<h4 data-id="heading-11">工程型成就是什么</h4>
<ol>
<li>目标
<ul>
<li>质量，效率</li>
</ul>
</li>
<li>方案与实施
<ul>
<li>规章制度</li>
<li>库</li>
<li>工具</li>
<li>系统</li>
</ul>
</li>
<li>结果
<ul>
<li>线上监控</li>
</ul>
</li>
</ol>
<p>程序员多通过 工程型成就来得到晋升和成长</p>
<p>比如:</p>
<p><code>XSS</code>攻击:分为两个方面， 一是来源问题，而是去向问题。</p>
<ul>
<li>来源:从哪能取到从别的网站发过来的信息？  <code>URL参数</code>， <code>postMessage</code> .....</li>
<li>去向: <code>innerHTML</code> ,<code>.appendChild</code>, <code>document.write</code>....</li>
</ul>
<p>将所有可能的来源和去向整理为一个安全手册， 重新<code>review</code>历史代码。照着安全手册开发一个代码扫描工具，杜绝无意识的造成<code>XSS</code>漏洞！</p>
<p><code>XSS</code>攻击预防， 前端怎么处理</p>
<ul>
<li>对全体前端宣讲</li>
<li>整体<code>review</code>代码</li>
<li>更改代码发布流程</li>
</ul>
<p>数据结构中，对前端而言， <code>顺序表</code>与<code>二叉树</code>可以优先学习！~</p>

<h3 data-id="heading-13">目标</h3>
<h4 data-id="heading-14">举例1</h4>
<p>公司必须保证用户使用软件活跃！</p>
<p>则可以将用户活跃度(日活/月活)  制定为具体的数据指标</p>
<p>产品上如何提升， 在软件中使用 发红包， 签到，每日任务机制 .....机制</p>
<h4 data-id="heading-15">举例2</h4>
<p>性能优化(页面加载快) 跟业务有什么强相关</p>
<p>如：将页面加载平均时长 跟用户点击率 联系到一起！</p>
<p><strong>一定要理解公司共同认可的指标是什么</strong>！否则很难得到真正意义的认可，只能做螺丝钉！
如果不了解公司到底想要的是什么！是绝对不可能得到晋升的！如果公司都认为你踏实肯干的话，那你的方向就走错了！应该走 技术好！业务敏感度高！ 老板赏识！的路线！ </p>
<p>公司为什么想要你经常加班， 是因为老板真正的核心指标没有提升，老板在巨大不安的驱动下才会要求你加班，  当公司核心指标高速增长的时候， 老板是不会有 加班，节约用纸之类的奇怪要求的！只有公司的业务长期停滞或下滑的时候才会出现！ </p>
<h3 data-id="heading-16">现状</h3>
<p>目标有了，找准了方向了! 然后开始出方案了！但<strong>应该与之前的情况进行对比。</strong></p>
<p>前端建立数据展示系统 如: <code>echarts</code>, <code>antv</code> ...</p>
<p>如果一开始就搞优化，但是最终只出现结果是不行的！ 应该采集好当前的数据与 优化有的情况进行对比!</p>
<p>采集数据:</p>
<ul>
<li>采集线上数据</li>
<li>采集技术数据
<ul>
<li>采集加载时间(window.performance)</li>
<li>采集错误数据(window.error)</li>
</ul>
</li>
</ul>
<h3 data-id="heading-17">方案</h3>
<p>设计技术方案前，必须有一个预估数据(预期结果达到的数据是什么样的，数据不用非常准确，数据集正确即可)
不能绝对的说 实施完这个方案后，性能一定变好! 研发效率一定提高</p>
<h3 data-id="heading-18">实施</h3>
<p>出了方案以后就该实施了！ 当你不确定这个东西造成的影响有多大的时候，可以搞一个小块的业务进行试验，实验后，拿到一定的数据后。 然后再去要求扩大实验范围</p>
<h3 data-id="heading-19">结果</h3>
<p>一些人总是做老黄牛，就是因为你默默地做了很多事情，但是你不汇报！ 跟老板汇报应该是一个日常的操作！
如果你不汇报， 很多事就相当于没做</p>
<p>老板不想听你的汇报，是什么原因呢?  是因为你的目标定的不对</p>
<p>有了结果后！就可以制定更高的目标！</p>

<h4 data-id="heading-22">init</h4>
<p>创建一个新项目，创建后出现 基本的目录结构和文件</p>

<p>在本地让项目跑起来，开始调试</p>

<p>单元测试， 测试覆盖率是很重要的</p>

<p>发布上线</p>
<hr>
<p>工具链体系除了 做好每个工具， 规定好每个工具之间的配合 。还有就是 </p>
<ul>
<li>版本问题(保证同一个团队里用同一个版本去做)--版本策略</li>
<li>数据统计</li>
</ul>
<p>通过工具链 对 **研发阶段的数据(<strong>平时怎么敲命令(什么时候run一下)，代码安了哪些组件，单元测试执行了多少次，发布了多少了次， 实际有多少项目，又发布了多少次</strong>)**进行统计，比如</p>
<ul>
<li>可以轻易的知道每个组件使用的比例是怎么样的，</li>
<li>老板也可以通过研发阶段数据统计成的图表发现一些研发问题</li>
<li>可以通过研发数据知道哪个工具经常报错</li>
</ul>
<h2 data-id="heading-26">六. 持续集成</h2>
<p>每个人开发各自的模块， 至少每天都将多人的代码集成，然后构建一次！</p>
<p>前端持续集成</p>
<ul>
<li>Check-in build(前端每次提交代码都集成构建一次，产生一个版本)</li>
<li>Lint(一些简单的规则检查) + Rule Check(比如:通过一些无头浏览器(Selenium, Phantomjs)检测及代码匹配情况，检查图片是够合规格)</li>
</ul>
<p>单张图片不得大于50K， 图片总包不得大于200K。原因: 用一张覆盖全屏的图片，经过正常的压缩后，大概占50K，将一个屏幕覆盖最多4次，则是200K
 </p>

<h2 data-id="heading-32">参考名词：</h2>

<li>PV:（Page View）访问量, 即页面浏览量或点击量，在一定统计周期内用户每打开或刷新一个页面就记录 1 次。</li>
<li>ctr:点击率（click-through rate）</li>
<li>判断用户活跃度：日活除以月活</li>
