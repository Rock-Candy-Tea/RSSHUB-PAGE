<p>延迟队列：一种带有 <em>延迟功能</em> 的消息队列</p>
<ol>
<li>延时 → 未来一个不确定的时间</li>
<li>mq → 消费行为具有顺序性</li>
</ol>
<p>这样解释，整个设计就清楚了。<strong>你的目的是 延时，承载容器是 mq。</strong></p>
<h2 data-id="heading-0">背景</h2>
<p>列举一下我日常业务中可能存在的场景：</p>
<ol>
<li>建立延时日程，需要提醒老师上课</li>
<li>延时推送 → 推送老师需要的公告以及作业</li>
</ol>
<p>为了解决以上问题，最简单直接的办法就是定时去扫表：</p>
<blockquote>
<p>服务启动时，开启一个异步协程 → 定时扫描 msg table，到了事件触发事件，调用对应的 handler</p>
</blockquote>
<p>几个缺点：</p>
<ol>
<li>每一个需要定时/延时任务的服务，都需要一个 msg table 做额外存储 → 存储与业务耦合</li>
<li>定时扫描 → 时间不好控制，可能会错过触发时间</li>
<li>对 msg table instance 是一个负担。反复有一个服务不断对数据库产生持续不断的压力</li>
</ol>
<p>最大问题其实是什么？</p>
<p><strong>调度模型基本统一，不要做重复的业务逻辑</strong></p>
<p>我们可以考虑将逻辑从具体的业务逻辑里面抽出来，变成一个公共的部分。</p>
<p>而这个调度模型，就是 <strong>延时队列</strong> 。</p>
<p>其实说白了：</p>
<p><em><strong>延时队列模型，就是将未来执行的事件提前存储好，然后不断扫描这个存储，触发执行时间则执行对应的任务逻辑。</strong></em></p>
<p>那么开源界是否已有现成的方案呢？答案是肯定的。Beanstalk (<a href="https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Fbeanstalkd%2Fbeanstalkd" target="_blank" rel="nofollow noopener noreferrer" title="https://github.com/beanstalkd/beanstalkd" ref="nofollow noopener noreferrer"></a><a href="https://link.juejin.cn/?target=https%3A%2F%2Fgithub.com%2Fbeanstalkd%2Fbeanstalkd" target="_blank" rel="nofollow noopener noreferrer" title="https://github.com/beanstalkd/beanstalkd" ref="nofollow noopener noreferrer">github.com/beanstalkd/…</a>) 它基本上已经满足以上需求</p>
<h2 data-id="heading-1">设计目的</h2>
<ol>
<li>消费行为 at least</li>
<li>高可用</li>
<li>实时性</li>
<li>支持消息删除</li>
</ol>
<p>一次说说上述这些目的的设计方向：</p>
<h3 data-id="heading-2">消费行为</h3>
<p>这个概念取自 mq 。mq 中提供了消费投递的几个方向：</p>
<ul>
<li><code>at most once</code> → 至多一次，消息可能会丢，但不会重复</li>
<li><code>at least once</code> → 至少一次，消息肯定不会丢失，但可能重复</li>
<li><code>exactly once</code> → 有且只有一次，消息不丢失不重复，且只消费一次。</li>
</ul>
<p><code>exactly once</code> 尽可能是 producer + consumer 两端都保证。当 producer 没办法保证是，那 consumer 需要在消费前做一个去重，达到消费过一次不会重复消费，这个在延迟队列内部直接保证。</p>
<p>最简单：<strong>使用 redis 的 setNX 达到 job id 的唯一消费</strong></p>
<h3 data-id="heading-3">高可用</h3>
<p>支持多实例部署。挂掉一个实例后，还有后备实例继续提供服务。</p>
<p>这个对外提供的 API 使用 cluster 模型，内部将多个 node 封装起来，多个 node 之间冗余存储。</p>
<h2 data-id="heading-4"><strong>为什么不使用 kafka？</strong></h2>
<p>考虑过类似基于 kafka/rocketmq 等消息队列作为存储的方案，最后从存储设计模型放弃了这类选择。</p>
<p>举个例子，假设以 Kafka 这种消息队列存储来实现延时功能，每个队列的时间都需要创建一个单独的 topic(如: Q1-1s, Q1-2s..)。这种设计在延时时间比较固定的场景下问题不太大，但如果是延时时间变化比较大会导致 topic 数目过多，会把磁盘从顺序读写会变成随机读写从导致性能衰减，同时也会带来其他类似重启或者恢复时间过长的问题。</p>
<ol>
<li>topic 过多 → 存储压力</li>
<li>topic 存储的是现实时间，在调度时对不同时间 (topic) 的读取，顺序读 → 随机读</li>
<li>同理，写入的时候顺序写 → 随机写</li>
</ol>

<h2 data-id="heading-6">API 设计</h2>
<p>整体上，只需要提供三个 api：</p>
<p>producer</p>
<ol>
<li><code>producer.At(msg []byte, at time.Time)</code></li>
<li><code>producer.Delay(body []byte, delay time.Duration)</code></li>
<li><code>producer.Revoke(ids string)</code></li>
</ol>
<p>consumer</p>
<ol>
<li><code>consumer.Consume(consume handler)</code></li>
</ol>
<blockquote>
<p>handler → 执行函数类型</p>
</blockquote></div>
