<p>🤡提测了等改BUG，摸鱼时啃下《设计模式之美》，本文对应设计模式与范式：结构型(54-55)，<strong><code>享元模式</code></strong> (Flyweight Pattern)，结构型设计模式最后一个~</p>
<ul>
<li>很好理解 → <strong><code>享元，被共享的单元</code></strong></li>
<li>意图 → <strong><code>节省内存，复用不可变对象</code></strong></li>
<li>换句话说 → <strong><code>找出相似对象间的共有特征，然后复用这些特征</code></strong>。</li>
</ul>
<p>一个形象的例子：游戏开发中，一个森林场景，成千上万种树，每棵树都实例化不同的模型，内存直接爆炸，可以抽取出所有树对象的共有属性，转移到一个单独的类中，然后只需要一个类实例，然后森林里的每一颗树对这个实例做一次引用(图摘自：<a href="http://gameprogrammingpatterns.com/flyweight.html" target="_blank" rel="nofollow noopener noreferrer">Flyweight</a>)：</p>

<p>使用享元模式只创建了4个卡牌对象，提提内部状态和外部状态的概念：</p>
<ul>
<li><strong>内部状态</strong>：<strong>固定不变可共享</strong> 的部分，存储在享元对象内部，比如这里的花色；</li>
<li><strong>外部状态</strong>：<strong>可变不可共享</strong> 的部分，一般由客户端传入享元对象内部，比如这里的大小；</li>
</ul>
<p>当然，状态的区分也不是绝对的，要看场景，比如扩展到斗地主的对局，内部状态就变成了54张牌(怎么发都不会超过54张)，外部状态变成了牌的持有人。扩展到象棋棋局，内部状态(颜色、文字)，外部状态(位置信息等)。</p>
<p>顺带带出UML类图和组成角色：</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/af30a7e668dc4d5486d0ac06a71d19cb~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<ul>
<li><strong>Flyweight</strong> (享元类) → 抽象类或接口，定义享元对象要实现的公共操作方法，方法会传入外部状态参数；</li>
<li><strong>ConcreteFlyweight</strong> (具体享元类) → 实现享元接口，并为内部状态添加存储空间；</li>
<li><strong>FlyweightFactory</strong> (享元工厂) → 创建并管理共享的享元对象，对外提供访问接口；</li>
</ul>
<p><strong>享元模式的本质</strong>：</p>
<blockquote>
<p>通过创建更多的可复用对象的共有特征，来尽可能减少创建重复对象的内存消耗。</p>
</blockquote>
<p><strong>缺点</strong>：</p>
<blockquote>
<p>时间换空间，对于需要快速响应的系统并不合适，需分离出内部和外部状态，难统一，增加了系统设计实现的复杂度；</p>
</blockquote>
<hr>
<h2 data-id="heading-2">0x2、享元模式 VS 多例、缓存、对象池</h2>
<p>从代码实现上看，享元模式和前面学的 <strong>单例中的多例</strong> 很相似，但从设计意图上看，是完全不同的。</p>
<blockquote>
<p>多例是为了限制对象的个数，享元模式则是为了对象复用，节省内存。</p>
</blockquote>
<p>在来看看与缓存的区别：</p>
<blockquote>
<p><strong>享元模式</strong>强调的是<strong>空间效率</strong>(大数据模型对象复用)，而缓存模式强调的是 <strong>时间效率</strong>(如缓存秒杀的活动数据和库存数据等，数据可能会占用大量空间，目的是为了及时响应)。</p>
</blockquote>
<p>还有对象池：</p>
<blockquote>
<p>为了避免对象频繁创建和释放导致内存碎片，预先申请一块连续的内存空间，每次创建对象时，直接从对象池里取出一个空闲对象来使用，使用完再重新放回对象池中以后后续使用，而非直接释放。</p>
<p>池化技术里的复用，可以理解为<strong>重复使用</strong>，主要目的是<strong>节省时间</strong>，任意时刻，每个对象都是被一个使用者<strong>独占。</strong>
而享元模式的复用，可以理解为<strong>共享使用</strong>，主要目的是<strong>节省空间</strong>，在整个生命周期中，都是被所有使用者<strong>共享</strong>。</p>
</blockquote>
<hr>
<h2 data-id="heading-3">0x3、享元模式在Java Integer、String中的应用</h2>
<p>先说下Integer的，下面的代码：</p>
<pre><code class="hljs language-java copyable" lang="java">Integer i1 = <span class="hljs-number">12</span>;
Integer i2 = <span class="hljs-number">12</span>;
Integer i3 = <span class="hljs-number">128</span>;
Integer i4 = <span class="hljs-number">128</span>;
System.out.println(i1 == i2);   <span class="hljs-comment">// 输出：true</span>
System.out.println(i3 == i4);   <span class="hljs-comment">// 输出: false</span>
<span class="copy-code-btn">复制代码</span></code></pre>
<p>学过Java的都知道：**双等号(==)**区别对待两类数据类型：</p>
<ul>
<li><strong>基本数据类型</strong>(byte、char、int等) → 比较值是否相同；</li>
<li><strong>引用数据类型</strong>(对象实例) → 比较变量指向的对象内存地址是否相同；</li>
</ul>
<p>既然i1和i2相等，说明指向同一对象内存地址，存在对象重用，按理来说，i3和i4应该也相等啊？且听我娓娓道来：</p>
<p>执行到这样的语句：Integer i1 = 12，12是基本数据类型int，赋值给包装器类型Integer时，会触发自动装箱机制，底层相当于执行了这样的语句：</p>
<pre><code class="hljs language-java copyable" lang="java"><span class="hljs-comment">// 创建Integer类型的实例，并赋值</span>
Integer i1 = Integer.valueOf(<span class="hljs-number">12</span>);

<span class="hljs-comment">// 附：自动拆箱 int j = i1 底层执行的是：</span>
<span class="hljs-keyword">int</span> j = i1.intValue();
<span class="copy-code-btn">复制代码</span></code></pre>
<p>行，那就来跟下 <strong><code>valueOf()</code></strong> 方法都做了啥：</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/7975e7423e414e2c8ce192e533e691bb~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>判断是否大于low小于high，是返回缓存中的Integer实例，否则返回新的Integer实例，看下 <strong>IntegerCache</strong>：</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/9b91120699514195a131ca4400c69344~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>呕吼，当IntegerCache类被加载是，初始化了值在 <strong>-128-127</strong> 间的Integer实例，并加入到缓存数组中。而只缓存这个范围的原因：</p>
<blockquote>
<p>不可能预先缓存好所有的整型值，只能选择对于大部分应用来说或最常用的整形值，即一个字节的大小。</p>
</blockquote>
<p>最大值也可以通过修改JVM配置进行修改 (二选一)：</p>
<pre><code class="hljs language-java copyable" lang="java">-Djava.lang.Integer.IntegerCache.high=<span class="hljs-number">255</span>
-XX:AutoBoxCacheMax=<span class="hljs-number">255</span>
<span class="copy-code-btn">复制代码</span></code></pre>
<p>这就是上面的代码输出true和false的原因了(-128-127范围内复用，不在范围外的直接new)，这里的IntegerCache就是享元模式中的享元工厂。</p>
<p>再接着看下String，下面这样的代码：</p>
<pre><code class="hljs language-java copyable" lang="java">String s1 = <span class="hljs-string">"杰哥"</span>;
String s2 = <span class="hljs-string">"杰哥"</span>;
String s3 = <span class="hljs-string">"杰"</span> + <span class="hljs-string">"哥"</span>;
String s4 = <span class="hljs-keyword">new</span> String(<span class="hljs-string">"杰哥"</span>);
System.out.println(s1 == s2);   <span class="hljs-comment">// 输出: true</span>
System.out.println(s1 == s3);   <span class="hljs-comment">// 输出: true</span>
System.out.println(s1 == s4);   <span class="hljs-comment">// 输出: false</span>
<span class="copy-code-btn">复制代码</span></code></pre>
<p>和Integer类的设计思想相似，JVM会专门开辟一块存储区来存储字符串常量(字符串常量池)，跟Integer不同，无法事先知道要共享哪些字符串常量，只能第一次用到的时候存储到常量池中，后续用到直接引用，就无需重新创建了。解释下上述代码：</p>
<ul>
<li>s1 == s2 → 编译时，"杰哥"被存储在常量池中，s1和s2的引用都指向常量池里的"杰哥"；</li>
<li>s1 == s3 → 编译器优化，先把字符串拼接，再去常量池里查找字符串是否存在，存在让s3直接指向该字符串；</li>
<li>s1 != s4 → 不是显式赋值，编译器会在堆中重新分配一个区域来存储它的对象数据；</li>
</ul>
