<h2 data-id="heading-0">WHY：为什么要学习复盘方法论？</h2>
<h3 data-id="heading-1">没有复盘方法论的领导简直就是公司的灾难！</h3>
<ul>
<li>不重视复盘可能导致只分析了表因而<code>没挖掘根因</code>，下次爆发出更严重的问题。</li>
<li>不重视复盘会导致员工觉得出现问题也没关系，<code>降低对自身的要求</code>。</li>
<li>没有定责标准会陷入无限的来回扯皮中，难以确定主要责任和次要责任，<code>不能形成统一意见</code>。</li>
</ul>
<h3 data-id="heading-2">有方法论的可以员工在复盘的过程中占据主动，展示自己的能力，防止别人甩锅给你。</h3>
<ul>
<li>很多人都非常讨厌扯皮，但是<code>越怕麻烦麻烦就越缠着你！</code></li>
<li>因为定责影响绩效，而且又有很多可以扯皮的地方，所以<code>定责不亚于一场宫斗戏！</code>。掌握主动可以给大家先入为主的印象和暗示，<code>进攻永远是最好的防守</code>。</li>
<li>提前复盘主动思考问题，就可以提前准备补救的方案、他人的质疑，避免手忙脚乱的说错话。</li>
</ul>
<h3 data-id="heading-3">微医讲究主人翁意识、阿里讲究要性、拼多多讲究本分</h3>
<ul>
<li>一言以蔽之，我们要有<code>主观能动性</code>。</li>
<li>主观能动性并不是资本家剥削压榨我们剩余价值的毒苹果，在工作中时刻保持主观能动性可以<code>提高你的执行力、沟通能力、影响力、思考问题、解决问题的能力</code>。而且义务和权利是互相的，你在<code>主动承担无人认领义务的时候，也悄悄的拥有了该业务的话语权</code>，积少成多，你的话语权就会越来越大。韩信点兵多多益善，强者是不怕撑死的。</li>
</ul>
<h3 data-id="heading-4">知耻而后勇、失败是成功之母</h3>
<ul>
<li>承认失败是一件很没有面子的事情，但是只有正视错误才能分析出根本原因并改正，促进自己的成长。</li>
<li><code>强者更在意自身的成长，只要有了实力面子迟早能挣回来</code>。</li>
</ul>
<h2 data-id="heading-5">四线复盘方法论</h2>
<h3 data-id="heading-6">复盘的目标</h3>
<ul>
<li><code>搞清楚事实</code>。</li>
<li><code>全面且深入的分析出根因</code>：丰田五问法。</li>
<li><code>得出让各方心服口服的定责结论</code>。</li>
<li><code>制定可以落地的改进方案</code>。</li>
</ul>
<h3 data-id="heading-7">一、时间线</h3>
<blockquote>
<p>时间线信息能反映出问题发现速度、各团队反应速度、方案执行速度、各团队处理效率等</p>
</blockquote>
<ul>
<li>明确问题的<code>发现过程</code>。</li>
<li>明确问题的<code>处理过程</code>中采取的各种关键措施。</li>
<li>明确问题<code>恢复的时间</code>。</li>
<li>明确问题<code>影响的结果</code>。</li>
</ul>
<h3 data-id="heading-8">二、问题链</h3>
<ul>
<li>分析业务流程：理清问题在关联的所有<code>业务系统中的传导顺序</code>，找到问题的根源点(某个项目)和放大点。</li>
<li>分析项目流程：理清问题在项目<code>开发流程中的根源点</code>，具体是产品、前端、后端、测试、运维哪一个的原因。</li>
</ul>
<h3 data-id="heading-9">三、责任链</h3>
<blockquote>
<p>因为责任影响绩效，所以明确各团队各人员的责任，并让各方心服口服是件非常复杂困难的事情，一定要<code>公平公正公开</code>才能够解决。</p>
</blockquote>
<ul>
<li>根据时间线中造成的影响，问题链中产生的原因来确定主要责任和次要责任
<ul>
<li><code>违反公司规章制度流程</code>的承担主责。</li>
<li>出现<code>重大疏忽</code>的承担主责。</li>
<li>问题的<code>源头</code>承担主责。</li>
<li>问题的<code>放大</code>者承担主责。</li>
</ul>
</li>
</ul>
<h3 data-id="heading-10">四、改进线</h3>
<blockquote>
<p>问题复盘的重点不是追究责任，而是为了提高个人和团队的能力，不再犯同样的错误，即使犯了错也能降低影响。所以我们要制定可以落地的<code>改进方案、改进负责人、时间节点、验收人</code></p>
</blockquote>
<ul>
<li>改进方案
<ul>
<li>通过时间线对问题的<code>发现、定位、处理流程进行改进</code>，减小日后出现问题造成的影响和范围
<ul>
<li>发现：<code>缺陷检测、日志收集上报、自动报警</code>。</li>
<li>定位：<code>日志分析、全链路分析、错误影像回放</code>。</li>
<li>流程：<code>紧急回滚恢复、问题处理预案、工单规范模板、各系统负责人自动拉群讨论</code>。</li>
</ul>
</li>
<li>通过问题链对问题的关联系统间的连接点做改进，<code>提高系统间的容错性、健壮性、问题发现的及时性</code>。</li>
<li>通过问题链对<code>项目开发流程、规范、健壮性进行改进，提高交付标准、测试标准、验证标准</code>。</li>
</ul>
</li>
<li>改进负责人：一般是产生问题的人或者是项目管理人员。</li>
<li>时间节点：没有时间节点的改进都是低优先级任务，迟早不了了之。</li>
<li>验收人：验收人和改进人要<code>编写改进报告并发送所有关联人员，经验教训应该分享</code>。</li>
</ul>
<h3 data-id="heading-11">备注</h3>
<ul>
<li>每一个线上错误的代价都是及其昂贵的，规范开发流程和做好问题复盘同样重要。</li>
<li>只会低头写业务代码而拒绝思考的人是没有前途的。</li>
<li>我们可以通过问题复盘学习到团队大佬分析、定位、处理、改进问题的思路和方法论，不是天天都有大佬现场教学的机会的，要珍惜。</li>
</ul>
