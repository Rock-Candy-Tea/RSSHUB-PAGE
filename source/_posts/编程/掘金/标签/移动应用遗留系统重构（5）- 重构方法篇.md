h2&#123;font-size:20px&#125;.markdown-body h3&#123;font-size:18px&#125;&#125;</style><h1 data-id="heading-0">前言</h1>
<p>上一篇<a href="https://juejin.cn/post/6950077521790500894" target="_blank">移动应用遗留系统重构（4）-分析篇</a>我们根据CloudDisk未来的架构，借助ArchUnit进行架构测试守护以及Intellij的Dependendencies分析出了按照未来的架构设计需要解决的异常依赖。</p>
<p>这一篇开始我们将分享进行依赖解除的重构流程、方法以及常用的工具使用。</p>
<h1 data-id="heading-1">重构流程</h1>
<pre><code class="hljs language-mermaid" lang="mermaid">graph TD
1.识别一个内聚的包-->2.解除该包的异常依赖

2.解除该包的异常依赖-->3.移动该包对应的代码及资源到新的模块

3.移动该包对应的代码及资源到新的模块-->4.包解耦验收
</code></pre>
<h2 data-id="heading-2">1.识别内聚的包</h2>
<p>对于移动应用通常我们可以通过产品的业务划分进行领域的识别划分。例如CloudDisk这个产品的相对还是比较清晰，业务上主要分为文件、动态及个人中心。</p>
<p>对于部分遗留系统来说，旧代码可能散落在不同的包下，或者原先的代码组织方式是以功能划分，而非业务划分。就像CloudDisk的代码一样，第一步我们得先把相关的业务代码组织到同一包下，这个阶段我们可以先不管是否存在异常依赖，因为只有先组织到一个内聚的包下才方便我们进行依赖分析及代码重构。</p>
<h2 data-id="heading-3">2.解除异常依赖</h2>
<p>这里我们将介绍几种通用的依赖解除手法。包含下沉、接口提取、路由跳转。</p>
<blockquote>
<p>后续的演示篇会通过视频进行具体的操作演示</p>
</blockquote>





















<table><thead><tr><th>依赖解除手法</th><th>使用场景</th></tr></thead><tbody><tr><td>下沉</td><td>原本类功能属于Library或者Platform的，直接下沉。例如LogUtil 或 DateUtil等</td></tr><tr><td>接口提取</td><td>适用于Bundle间有数据或者行为依赖。例如某个BundleA中的classA需要触发BundleB的某个业务行为</td></tr><tr><td>路由跳转</td><td>适用于UI页面间跳转。例如某个BundleA中的Activity1，需要跳转到BundleB的Activity2</td></tr></tbody></table>
<p><strong>重构手法：</strong></p>
<ol>
<li><strong>类下沉</strong></li>
</ol>
<ul>
<li>
<p>将<code>具体类</code>移动到适当的 Lib 模块中</p>
</li>
<li>
<p>在调用模块增加对该 Lib 的依赖</p>
</li>
</ul>
<ol start="2">
<li><strong>接口提取</strong></li>
</ol>
<ul>
<li>在适当的公用模块中创建空的<code>接口</code></li>
<li>将调用具体页面类的跳转代码块所在的包中建立新的<code>实现类</code>实现该<code>接口</code></li>
<li>(自动)将<code>调用代码块</code>通过 Extract method 提取成<code>新方法</code></li>
</ul>
<blockquote>
<p>如已经是独立方法跳过此步</p>
</blockquote>
<ul>
<li>(自动)在原调用逻辑所属的类中增加<code>实现类</code>的<code>成员变量</code>作为<code>delegate</code></li>
</ul>
<blockquote>
<p>需要预留 Inject 接口，建议采用 Constructor Inject，静态成员提供setter</p>
</blockquote>
<ul>
<li>(自动)将<code>新方法</code>调用转移到<code>delegate</code>中</li>
</ul>
<blockquote>
<p>如果是静态方法先通过 Change Method Signature 将 delegate 作为参数传给该方法</p>
</blockquote>
<ul>
<li>(自动)将新方法 Pull up 到<code>接口</code>中</li>
<li>(自动)将<code>实现类</code>移动到壳程序中</li>
<li>在壳程序中实现<code>实现类</code>的Inject</li>
</ul>
<ol start="3">
<li><strong>路由跳转</strong></li>
</ol>
<ul>
<li>在跳转类定义对应的映射Path</li>
<li>在调用处使用对应的path进行跳转</li>
</ul>
<h2 data-id="heading-4">3.移动代码及资源</h2>
<p>当包的异常依赖全部解耦完后，就可以直接进行移动了。这里我们分享2中常用的代码移动方式。</p>
<ol>
<li><strong>Move</strong></li>
</ol>
<p>这种方式大家应该比较常用，选择一个File或者Directory,按下F6选择希望移动后的目录则可。</p>
<p><img alt class="lazyload" src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/439fce24e104478ebfd13fbf1edb1560~tplv-k3u1fbpfcp-zoom-1.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>但是这种方式会存在一个问题，就是被移动的类如果依赖了其他的类或者资源，移动后会出现依赖异常。</p>
<p><strong>适用场景：移动的File或Directory没有其他的依赖</strong></p>
<ol start="2">
<li><strong>Modularize</strong></li>
</ol>
<p>Modularize能够分析出移动的File存在的相关依赖，并一起关联移动，很好解决Move的痛点，非常适用于跨Module的移动。</p>
<p><img alt class="lazyload" src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/e45b77efd1ef41ffbb4d7c1b7b4c62d4~tplv-k3u1fbpfcp-zoom-1.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>选择移动的Module后点击Preview。</p>
<p><img alt class="lazyload" src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/22fa5a7c42f048808cd08744fa7b143b~tplv-k3u1fbpfcp-zoom-1.image" data-width="800" data-height="600" referrerpolicy="no-referrer"></p>
<p>这里注意，有一些划线的文件，那是因为这个文件同时被多处引用，如果跟随一起移动，那么其他的地方会报错。所以我们需要将划线的文件先移动至公用的合适位置。待Preview没有任何的文件划线时，就可以进行移动。</p>
<h2 data-id="heading-5">4.包解耦验收</h2>
<ul>
<li>所有模块编译通过</li>
<li>所有新增模块符合模块依赖规则</li>
<li>通过架构守护测试</li>
</ul>
<h1 data-id="heading-6">总结</h1>
<p>这一篇我们分享了进行依赖解除的重构流程，主要为4个操作步骤，识别内聚包、解除依赖、移动、验收。同时也介绍了Intellij中非常好用的Modularize功能。接下来我们就可以开始动手进行代码重构，但此时我们又面临着另外一个问题，也是很多同学在做重构时经常担心的一个问题。<strong>重构时如何保证功能的正确性，不会修改出新问题。</strong></p>
