<h1 data-id="heading-0">以 “深度思考” 为立足之本</h1>
<p>在过去两年里，我们以 “架构组件” 为话题，不遗余力地在每篇文章中贯彻落实 <strong>基于 “第一性原理” 和 “实事求是” 精神的深度思考方式</strong>，</p>
<p>在 “深度思考” 的帮助下，以及基于对大量样本的追踪和反思，我们独树一帜地从软件工程的视角出发抓住本质，并开源了一系列高频使用的仓库，</p>
<p>包括 腾讯音乐、BMW、TCL 等知名厂商的软件，都在使用我们正在维护的<a href="https://github.com/KunMinX/UnPeek-LiveData" target="_blank" rel="nofollow noopener noreferrer">《UnPeek-LiveData》</a>等仓库。</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/3ba4705f546d4a6996bd0f4979635604~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<h1 data-id="heading-1">不小心当了回 “万恶之源”</h1>
<p>不知何时起，你是否经常在 培训机构软文 或网文中看到 “一致性” 等令人迷惑的说辞？</p>
<p>这些术语在 Android 领域是从未有过，网上关于 “一致性问题” 等说辞，它们都有一个共同的来源，</p>
<p>事实上，这些术语都是本人经由长期的深度思考、实践和交流后，为现象本质匹配的高度概括，《重学安卓》与之相对应的每一篇文章都 <strong>提供了 背景缘由、职责边界 等完整的解析过程</strong>，</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/f34c9fd1306a4a9890fe3fcfe221ba46~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>但十分令人遗憾的是，本是从 “实事求是” 出发概括的术语，却频繁被人拿去 “点缀文章” 和 “挂羊头卖狗肉”，这些滥用行为，无形中扭曲和破坏了术语 “实事求是” 的形象，</p>
<p>所以过去两年里，我们积极地举报 “培训机构的洗稿软文”，以及在相关文章的评论区补充参考文献来源，</p>
<p>非常感谢小伙伴们的主动反馈。</p>
<h1 data-id="heading-2">本质概括一览</h1>
<p>基于深度思考，我们确立下来并广泛传播的 “本质概括” 包括但不限于：</p>
<h3 data-id="heading-3">Jetpack 架构组件本质：</h3>
<p>Lifecycle 的本质是解决 “生命周期管理” 的一致性问题</p>
<p>LiveData 的本质是解决 “跨域消息同步” 的一致性问题</p>
<p>ViewModel 的本质是解决 “状态保存恢复” 的一致性问题</p>
<p>DataBinding 的本质是解决 “视图实例的 null 安全” 的一致性问题</p>
<p>Navigation 的本质是解决 “路由初始参数恢复” 的一致性问题</p>
<p>…</p>
<p>若要说它们有什么共性的话，就是透过各种方式 <strong>实现样板逻辑的 “内聚”</strong>，从而达到规避一致性问题的目的。</p>

<p>文章来源</p>

<h3 data-id="heading-4">声明式 UI 本质：</h3>
<p>声明式 UI 的本质是函数式编程，</p>
<p>函数式编程的基石是纯函数，</p>
<p>纯函数的特性是 <strong>只有一个入口、只有一个出口</strong>，且无副作用，</p>
<p>声明式 UI 正是通过对视图实例的屏蔽，来规避 “视图实例的 null 安全” 的一致性问题，</p>
<p><img src="https://p3-juejin.byteimg.com/tos-cn-i-k3u1fbpfcp/00fa7778e58748b0a3d47d6250b3c4ca~tplv-k3u1fbpfcp-zoom-1.image" alt loading="lazy" referrerpolicy="no-referrer"></p>
<p>也即声明式 UI 可用于替代 DataBinding 等框架，</p>
<p>如果公司项目执意使用 Java，为了规避 null 安全问题，务必使用 DataBinding 等框架，</p>
<p><strong>如果允许使用 kotlin，那么当下 kotlin + ViewBinding 的组合是更优解</strong>。</p>
<blockquote>
<p>文章来源</p>
<p><a href="https://xiaozhuanlan.com/topic/2356748910" target="_blank" rel="nofollow noopener noreferrer">《是 “一通百通” 的 声明式 UI 扫盲干货》</a></p>
<p><a href="https://xiaozhuanlan.com/topic/9816742350" target="_blank" rel="nofollow noopener noreferrer">《从 被误解 到 真香 的 Jetpack DataBinding》</a></p>
</blockquote>
<h3 data-id="heading-5">架构模式本质：</h3>
<p>MVP 的本质是基于 “依赖倒置原则” 实现组件的可替换，适合非页面开发场景的编写（具体可参见我开源的 Linkage-RecyclerView 中万用的适配器），</p>
<p>MVVM 的本质是基于 “数据绑定” 来解决视图实例 null 安全一致性问题，也即它是专用于页面开发的模式，</p>
<p>当我们剔除了 DataBinding 框架而使用 Compose 或 kotlin + ViewBinding 等方式来规避一致性问题，虽然效果是等同的，但已不能称作是 MVVM。</p>
<blockquote>
<p>文章来源</p>
<p><a href="https://xiaozhuanlan.com/topic/8204519736" target="_blank" rel="nofollow noopener noreferrer">《如何让同事爱上架构模式、少写 bug 多注释》</a></p>
<p><a href="https://xiaozhuanlan.com/topic/6105792348" target="_blank" rel="nofollow noopener noreferrer">《是让人提神醒脑的 MVP、MVVM 关系精讲》</a></p>
</blockquote>
<h3 data-id="heading-6">LiveData 的那些事：</h3>
<p>LiveData 的设计存在缺陷。</p>
<p>一方面它提供了面向 “事件” 的设计，</p>
<p>这使得我们萌生了通过 <strong>“决策权收紧” 的结构</strong>（也即所谓 “唯一可信源”）来确保 “消息分发可靠一致” 的目的成为可能，</p>
<p>另一方面它权当自己是 “状态”，而仅提供粘性的设计，</p>
<p>正是这种令人迷惑的设计，导致了所谓 “数据倒灌” 现象的发生。</p>
<p>要想弄清楚 “唯一可信源” 和 “数据倒灌”，得先正确理解和区分 “状态” 和 “事件”。</p>
<blockquote>
<p>文章来源</p>
<p><a href="https://xiaozhuanlan.com/topic/2049857631" target="_blank" rel="nofollow noopener noreferrer">《LiveData 唯一可信源 读写分离设计 独家解析》</a></p>
<p><a href="https://xiaozhuanlan.com/topic/6719328450" target="_blank" rel="nofollow noopener noreferrer">《LiveData 数据倒灌 背景缘由全貌 独家解析》</a></p>
</blockquote>
