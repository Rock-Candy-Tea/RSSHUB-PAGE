                                                                                            <p>距离 Go 1.17 发布七个月后，Go 1.18 正式发布了！ Go 1.18 是一个包含大量新功能的版本，包括对语言本身做了有史以来最大的改变（泛型）、工具链的实现、运行时和库的更改，还改善了性能。</p> 
<p>与往常一样，该版本保持了 Go 1<a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Ftip.golang.org%2Fdoc%2Fgo1compat" target="_blank">的兼容性承诺</a>：<span style="background-color:#ffffff; color:#202224">几乎所有 Go 程序都能像以前一样继续编译和运行。下面来看一下新版本的一些重大特性：</span></p> 
<h3 style="text-align:start">泛型</h3> 
<p>以下是关于 Go 1.18 泛型的最明显变化的列表，如需更全面的概述请参阅泛型<a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fgo.googlesource.com%2Fproposal%2F%2B%2Frefs%2Fheads%2Fmaster%2Fdesign%2F43651-type-parameters.md" target="_blank">提案</a>，更详细信息请参阅<a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Ftip.golang.org%2Fref%2Fspec" target="_blank">语言规范</a>。</p> 
<ul> 
 <li>函数和类型声明的语法，现在接受<a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Ftip.golang.org%2Fref%2Fspec%23Type_parameter_declarations" target="_blank">类型参数</a>。</li> 
 <li>参数化函数和类型可以通过<strong>在方括号中</strong>列出类型参数来实例化。</li> 
 <li>新标记<code>~</code>已添加到<a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Ftip.golang.org%2Fref%2Fspec%23Operators_and_punctuation" target="_blank">操作符和标点符号</a>中。</li> 
 <li>接口类型的语法现在允许嵌入任意类型（不仅仅是接口的类型名称）以及 union 和 ~T 类型元素。，这样的接口只能用作类型约束。</li> 
 <li>新的 <a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Ftip.golang.org%2Fref%2Fspec%23Predeclared_identifiers" target="_blank">预声明标识符</a> <code>any</code>是空接口的别名，可以用来代替 <code>interface&#123;&#125;</code>.</li> 
 <li>新的 <a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Ftip.golang.org%2Fref%2Fspec%23Predeclared_identifiers" target="_blank">预声明标识符</a> <code>comparable</code>是一个接口，表示可以使用<code>==</code>或者 <code>!=</code> 比较的所有类型的集合，它只能用作（或嵌入）类型约束。</li> 
</ul> 
<p style="text-align:start">有三个使用泛型的实验包可能有用，这些包在 x/exp 存储库中；但它们的 API 不在 Go 1 兼容性承诺的保证范围内：</p> 
<ul> 
 <li style="text-align:start"><a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fpkg.go.dev%2Fgolang.org%2Fx%2Fexp%2Fconstraints" target="_blank"><code>golang.org/x/exp/constraints</code></a>：对通用代码有用的约束，例如 <a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fpkg.go.dev%2Fgolang.org%2Fx%2Fexp%2Fconstraints%23Ordered" target="_blank"><code>constraints.Ordered</code></a>.</li> 
 <li style="text-align:start"><a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fpkg.go.dev%2Fgolang.org%2Fx%2Fexp%2Fslices" target="_blank"><code>golang.org/x/exp/slices</code></a><span style="background-color:#ffffff; color:#202224">：对任何元素类型的切片进行操作的通用函数集合。</span></li> 
 <li style="text-align:start"><a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fpkg.go.dev%2Fgolang.org%2Fx%2Fexp%2Fmaps" target="_blank"><code>golang.org/x/exp/maps</code></a><span style="background-color:#ffffff; color:#202224">：对任何键或元素类型的映射进行操作的通用函数集合。</span></li> 
</ul> 
<p style="text-align:start">当前的泛型实现具有以下已知限制：</p> 
<ul> 
 <li>Go 编译器无法处理泛型函数或方法中的类型声明，计划在 Go 1.19 中取消这个限制。</li> 
 <li>Go 编译器不接受具有预声明函数 real、imag 和 complex 的参数类型的参数，计划在 Go 1.19 中取消这个限制。</li> 
 <li>如果 m 由 P 的约束接口显式声明，Go 编译器仅支持在类型参数类型 P 的值 x 上调用方法 m。 类似地，方法值 x.m 和方法表达式 P.m 也仅在 m 由 P 显式声明时才受支持，即使 m 可能在 P 的方法集中，因为 P 中的所有类型都实现了 m，计划在 Go 1.19 中取消这个限制。</li> 
 <li>Go 编译器不支持访问结构字段 x.f，其中 x 是类型参数类型，即使类型参数的类型集中的所有类型都具有字段 f，计划在 Go 1.19 中取消这个限制。</li> 
 <li>不允许将类型参数或指向类型参数的指针作为结构类型中的未命名字段嵌入，同样地，也不允许在接口类型中嵌入类型参数。</li> 
 <li>具有多个 <span style="color:#222222">term </span>的 union 元素可能不包含具有非空方法集的接口类型。</li> 
</ul> 
<p>泛型代表 Go 生态系统的巨大变化，虽然官方更新了几个支持泛型的核心工具，但还有很多工作要做。剩余的工具、文档和库需要一些时间才能赶上这些语言变化。此外，官方公告中还有这么一段话：</p> 
<blockquote> 
 <ul> 
  <li>可能会有一些使用泛型的代码可以在 1.18 版本中使用，但在以后的版本中会中断。</li> 
  <li>我们不计划或期望做出任何此类更改，但是，由于我们今天无法预见的原因，可能需要<strong>在未来版本中破坏 1.18 的程序</strong>。</li> 
  <li>我们鼓励在有意义的地方使用泛型，但在生产环境中部署泛型代码时，请谨慎行事。</li> 
 </ul> 
</blockquote> 
<p>（虽然泛型是搞出来了，但很可能有 Bug，不建议在生产中使用）</p> 
<h2>模糊测试</h2> 
<ul> 
 <li>Go 1.18 包括 fuzzing（模糊测试） 的实现，如 <a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fgolang.org%2Fissue%2F44551" target="_blank">fuzzing 提案</a>所述，详情请参阅 <a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fgo.dev%2Fdoc%2Ftutorial%2Ffuzz" target="_blank">fuzzing 教程</a>以开始使用。</li> 
 <li><strong>注意，模糊测试会消耗大量内存</strong>，并且可能会影响机器运行时的性能。</li> 
 <li>另请注意，模糊引擎在运行时会将扩展测试覆盖率的值写入模糊缓存目录 <code>$GOCACHE/fuzz</code>。目前对可以写入模糊缓存的文件数量或总字节数<strong>没有限制</strong>，因此可能会占用大量存储空间（可能为 GB 级别）。</li> 
</ul> 
<h2 style="text-align:start">编译器</h2> 
<ul> 
 <li><span style="background-color:#ffffff; color:#202224">现在编译器可以内联包含范围循环或标记为循环的函数。</span></li> 
 <li><span style="background-color:#ffffff; color:#202224">编译器的类型检查器被完全替换以支持泛型，一些错误消息可能使用与以前不同的措辞（提供更多详细信息，或以更有用的方式表述）。</span></li> 
 <li>由于与支持泛型相关的编译器的更改，Go 1.18 的<strong>编译速度</strong>可能比 Go 1.17 的编译速度<strong>慢大约 15%</strong>，代码的执行时间不受影响，目前计划在 Go 1.19 中提高编译器的速度。</li> 
</ul> 
<h3 style="text-align:start">Bug fixes</h3> 
<ul> 
 <li>Go 1.18 编译器可以正确地报告在函数文本中设置但从未使用过的变量的错误（已声明但未使用），解决了一个老问题 <span style="background-color:#ffffff; color:#202224">issue<span> </span></span><a href="https://www.oschina.net/action/GoToLink?url=https%3A%2F%2Fgolang.org%2Fissue%2F8560" target="_blank">#8560</a><span style="color:#202224"> 。</span></li> 
 <li><span style="color:#202224">Go 1.18 编译器现在在将如 '1' << 32 之类的符文常量表达式作为参数传递给预声明函数 print 和 println 时会报告溢出。</span></li> 
</ul> 
<h2 style="text-align:start">Ports</h2> 
<h3 style="text-align:start"><span><span><span style="color:#202224"><span><span><span><span><span><span><span><span><span><span><span style="background-color:#ffffff"><span><span><span>AMD64</span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></h3> 
<p>Go 1.18 引入了新的<code>GOAMD64</code>环境变量，它在编译时选择 AMD64 架构的最低目标版本，允许的值为<code>v1</code>、 <code>v2</code>、<code>v3</code>或<code>v4</code>，默认是<code>v1</code>。</p> 
<h3 style="text-align:start"><span><span><span style="color:#202224"><span><span><span><span><span><span><span><span><span><span><span style="background-color:#ffffff"><span><span><span>RISC-V</span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></h3> 
<p><span><span><span style="color:#202224"><span><span><span><span><span><span><span><span><span><span><span style="background-color:#ffffff"><span><span><span>Linux 上的 64 位 RISC-V 架构（linux/riscv64 端口）现在支持 c-archive 和 c-shared 构建模式。</span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></span></p> 
<h3 style="text-align:start">Linux</h3> 
<p><span style="background-color:#ffffff; color:#202224">Go 1.18 需要 Linux 内核版本 2.6.32 或更高版本。</span></p> 
<h3><span style="background-color:#ffffff; color:#202224">Windows</span></h3> 
<p><span style="background-color:#ffffff; color:#202224">windows/arm 和 windows/arm64 端在支持非合作抢占，有希望解决在调用 Win32 函数时遇到的一些细微的 bug，这些bug在很长一段时间内会阻塞。</span></p> 
<h3 style="text-align:start">iOS</h3> 
<p>在 iOS（ios/arm64 端口）和在基于 AMD64 的 macOS（ios/amd64 端口）上运行的 iOS 模拟器上，Go 1.18 现在需要 iOS 12 或更高版本； 已停止支持以前的版本。</p> 
<h3 style="text-align:start">FreeBSD</h3> 
<p>Go 1.18 是支持 FreeBSD 11.x 的最后一个版本，Go 1.19 需要 FreeBSD 12.2+ 或 FreeBSD 13.0+。</p> 
<h2>性能提升</h2> 
<p>由于 Go1.17 中寄存器 ABI 调用约定扩展到了 <span style="background-color:#ffffff; color:#121212">RM64 / Apple M1 / PowerPC 64<span> </span></span>架构，因此 Go1.18 对这几个架构包含了高达 20% 的 CPU 性能提升。</p> 
<p> </p>
