<div>   
<br>技术传播的价值，不仅仅体现在通过商业化产品和开源项目来缩短我们构建应用的路径。加速业务的上线速率，也体现在优秀工程师的工作效率提升、产品性能优化和用户体验改善等经验方面的分享，以提高我们的专业能力。<br>
<br>接下来，阿里巴巴技术专家三画，将分享自己和团队在画好架构图方面的理念和经验，希望对你有所帮助。<br>
<br>当我们想用一张或几张图来描述我们的系统时，是不是经常遇到以下情况：<br>
<ul><li>对着画布无从下手、删了又来</li><li>如何用一张图描述我的系统，并且让产品、运营、开发都能看明白</li><li>画了一半的图还不清楚受众是谁</li><li>画出来的图到底是产品图功能图还是技术图又或是大杂烩</li><li>图上的框框有点少是不是要找点儿框框加进来</li><li>布局怎么画都不满意……</li></ul><br>
<br>如果有同样的困惑，本文将介绍一种画图的方法论，来让架构图更清晰。<br>
<h3>先厘清一些基础概念</h3><h4>什么是架构？</h4>架构就是对系统中的实体以及实体之间的关系所进行的抽象描述，是一系列的决策。<br>
<br>架构是结构和愿景。系统架构是概念的体现，是对物/信息的功能与形式元素之间的对应情况所做的分配，是对元素之间的关系以及元素同周边环境之间的关系所做的定义。<br>
<br>做好架构是个复杂的任务，也是个很大的话题，本篇就不做深入了。有了架构之后，就需要让干系人理解、遵循相关决策。<br>
<h4>什么是架构图？</h4>系统架构图是为了抽象地表示软件系统的整体轮廓和各个组件之间的相互关系和约束边界，以及软件系统的物理部署和软件系统的演进方向的整体视图。<br>
<h4>架构图的作用</h4>一图胜千言。要让干系人理解、遵循架构决策，就需要把架构信息传递出去。架构图就是一个很好的载体。<br>
<br>那么，画架构图是为了：<br>
<ul><li>解决沟通障碍</li><li>达成共识</li><li>减少歧义</li></ul><br>
<br><h4>架构图分类</h4>搜集了很多资料，分类有很多，有一种比较流行的是 4+1 视图，分别为场景视图、逻辑视图、物理视图、处理流程视图和开发视图。<br>
<br><strong>1.、场景视图</strong><br>
<br>场景视图用于描述系统的参与者与功能用例间的关系，反映系统的最终需求和交互设计，通常由用例图表示。<br>

<br>
<br><strong>2、逻辑视图</strong><br>
<br>逻辑视图用于描述系统软件功能拆解后的组件关系，组件约束和边界，反映系统整体组成与系统如何构建的过程，通常由 UML 的组件图和类图来表示。<br>

<br><strong>3、物理视图</strong><br>
<br>物理视图用于描述系统软件到物理硬件的映射关系，反映出系统的组件是如何部署到一组可计算机器节点上，用于指导软件系统的部署实施过程。<br>

<br><strong>4、处理流程视图</strong><br>
<br>处理流程视图用于描述系统软件组件之间的通信时序，数据的输入输出，反映系统的功能流程与数据流程，通常由时序图和流程图表示。<br>

<br>
<br><strong>5、开发视图</strong><br>
<br>开发视图用于描述系统的模块划分和组成，以及细化到内部包的组成设计，服务于开发人员，反映系统开发实施过程。<br>

</div>
<br>
以上 5 种架构视图从不同角度表示一个软件系统的不同特征，组合到一起作为架构蓝图描述系统架构。<br>
<h3>怎样的架构图是好的架构图</h3>上面的分类是前人的经验总结，图也是从网上摘来的，那么这些图画的好不好呢？是不是我们要依葫芦画瓢去画这样一些图？<br>
<br>先不去管这些图好不好，我们通过对这些图的分类以及作用，思考了一下，总结下来。<br>
<br>我们认为，在画出一个好的架构图之前， 首先应该要明确其受众，再想清楚要给他们传递什么信息。<br>
<br>所以，不要为了画一个物理视图去画物理视图，为了画一个逻辑视图去画逻辑视图，而应该根据受众的不同，传递的信息的不同，用图准确地表达出来，最后的图可能就是在这样一些分类里。<br>
<br>那么，画出的图好不好的一个直接标准就是：受众有没有准确接收到想传递的信息。<br>
<br>明确这两点之后，从受众角度来说，一个好的架构图是不需要解释的，它应该是自描述的，并且要具备一致性和足够的准确性，能够与代码相呼应。<br>

<h3>本文推荐的画图方法</h3><div class="aw-upload-img-list active">
<a href="http://dockone.io/uploads/article/20210925/4bfd7014981cdd26a7fef3cb0dc1842c.jpeg" target="_blank" data-fancybox-group="thumb" rel="lightbox"><img src="https://cors.zfour.workers.dev/?http://dockone.io/uploads/article/20210925/4bfd7014981cdd26a7fef3cb0dc1842c.jpeg" class="img-polaroid" title="9.jpeg" alt="9.jpeg" referrerpolicy="no-referrer"></a>
</div>
<br>
C4 模型使用容器（应用程序、数据存储、微服务等）、组件和代码来描述一个软件系统的静态结构。<br>
<br>这几种图比较容易画，也给出了画图要点，但最关键的是，我们认为，它明确指出了每种图可能的受众以及意义。<br>
<br>下面的案例来自 C4 官网，然后加上了一些我们的理解，来看看如何更好的表达软件架构。<br>
<h4>语境图（System Context Diagram）</h4><div class="aw-upload-img-list active">
<a href="http://dockone.io/uploads/article/20210925/677e2aa76b6dde52853e77cbe0edafe5.jpeg" target="_blank" data-fancybox-group="thumb" rel="lightbox"><img src="https://cors.zfour.workers.dev/?http://dockone.io/uploads/article/20210925/677e2aa76b6dde52853e77cbe0edafe5.jpeg" class="img-polaroid" title="10.jpeg" alt="10.jpeg" referrerpolicy="no-referrer"></a>
</div>
<br>
这是一个想象的待建设的互联网银行系统，它使用外部的大型机银行系统存取客户账户、交易信息，通过外部电邮系统给客户发邮件。<br>
<br>可以看到，非常简单、清晰，相信不需要解释，都看的明白，里面包含了需要建设的系统本身，系统的客户，和这个系统有交互的周边系统。<br>
<br>这样一个简单的图，可以告诉我们，要构建的系统是什么;它的用户是谁，谁会用它，它要如何融入已有的 IT 环境。<br>
<br>这个图的受众可以是开发团队的内部人员、外部的技术或非技术人员。即：<br>
<ul><li>构建的系统是什么</li><li>谁会用它</li><li>如何融入已有的 IT 环境</li></ul><br>
<br>怎么画？中间是自己的系统，周围是用户和其他与之相互作用的系统。这个图的关键就是梳理清楚待建设系统的用户和高层次的依赖，梳理清楚了画下来只需要几分钟时间。<br>
<h4>容器图（Container Diagram）</h4>容器图是把语境图里待建设的系统做了一个展开。<br>
<div class="aw-upload-img-list active">
<a href="http://dockone.io/uploads/article/20210925/c93c8e83aa24d28b7ef7d59bb0d49fa7.jpeg" target="_blank" data-fancybox-group="thumb" rel="lightbox"><img src="https://cors.zfour.workers.dev/?http://dockone.io/uploads/article/20210925/c93c8e83aa24d28b7ef7d59bb0d49fa7.jpeg" class="img-polaroid" title="11.jpeg" alt="11.jpeg" referrerpolicy="no-referrer"></a>
</div>
<br>
上图中，除了用户和外围系统，要建设的系统包括一个基于 Java\Spring MVC的 Web 应用提供系统的功能入口，基于 Xamarin 架构的手机 App 提供手机端的功能入口，一个基于 Java 的 API 应用提供服务，一个 MySQL 数据库用于存储，各个应用之间的交互都在箭头线上写明了。<br>
<br>看这张图的时候，不会去关注到图中是直角方框还是圆角方框，不会关注是实线箭头还是虚线箭头，甚至箭头的指向也没有引起太多注意。<br>
<br>我们有许多的画图方式，都对框、线的含义做了定义，这就需要画图的人和看图的人都清晰的理解这些定义，才能读全图里的信息。<br>
<br>而现实是，这往往是非常高的一个要求，所以，很多图只能看个大概的含义。<br>
<br>这个图的受众可以是团队内部或外部的开发人员，也可以是运维人员。用途可以罗列为：<br>
<ul><li>展现了软件系统的整体形态。</li><li>体现了高层次的技术决策。</li><li>系统中的职责是如何分布的，容器间是如何交互的。</li><li>告诉开发者在哪里写代码。</li></ul><br>
<br>怎么画？用一个框图来表示，内部可能包括名称、技术选择、职责，以及这些框图之间的交互，如果涉及外部系统，最好明确边界。<br>
<h4>组件图（Component Diagram）</h4><div class="aw-upload-img-list active">
<a href="http://dockone.io/uploads/article/20210925/133dffabd4da4336a60256636cb7e44d.jpeg" target="_blank" data-fancybox-group="thumb" rel="lightbox"><img src="https://cors.zfour.workers.dev/?http://dockone.io/uploads/article/20210925/133dffabd4da4336a60256636cb7e44d.jpeg" class="img-polaroid" title="12.jpeg" alt="12.jpeg" referrerpolicy="no-referrer"></a>
</div>
<br>
组件图是把某个容器进行展开，描述其内部的模块。<br>
<br>这个图主要是给内部开发人员看的，怎么去做代码的组织和构建。其用途有：<br>
<ul><li>描述了系统由哪些组件/服务组成</li><li>厘清了组件之间的关系和依赖</li><li>为软件开发如何分解交付提供了框架</li></ul><br>
<br><h4>类图（Code/Class Diagram）</h4><div class="aw-upload-img-list active">
<a href="http://dockone.io/uploads/article/20210925/fee85b02afa725c87f27a7148aeb6661.jpeg" target="_blank" data-fancybox-group="thumb" rel="lightbox"><img src="https://cors.zfour.workers.dev/?http://dockone.io/uploads/article/20210925/fee85b02afa725c87f27a7148aeb6661.jpeg" class="img-polaroid" title="13.jpeg" alt="13.jpeg" referrerpolicy="no-referrer"></a>

<br>
这个图很显然是给技术人员看的，比较常见，就不详细介绍了。<br>
