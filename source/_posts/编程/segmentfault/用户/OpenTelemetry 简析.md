It supports a variety of popular open-source projects including Jaeger and Prometheus.</p><p>OpenTelemetry 是一组标准和工具的集合，旨在管理观测类数据，如 trace、metrics、logs 等 (未来可能有新的观测类数据类型出现)。</p><p>OpenTelemetry 提供与 vendor 无关的实现，根据用户的需要将观测类数据导出到不同的后端，如开源的 Prometheus、Jaeger 或云厂商的服务中。</p><p>那么 OpenTelemetry 不是什么？从官方描述可以看出：</p><blockquote>OpenTelemetry is not an observability back-end like Jaeger or Prometheus. Instead, it supports exporting data to a variety of open-source and commercial back-ends. It provides a pluggable architecture so additional technology protocols and formats can be easily added.</blockquote><p>即 OpenTelemetry 不提供与可观测性相关的后端服务，这类后端服务通常提供的是存储、查询、可视化等服务。</p><p>通过下述抽象图可以简单理解 OpenTelemetry 的工作范围：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762550" alt="1.png" title="1.png" referrerpolicy="no-referrer"></span></p><h2>OpenTelemetry 面对的问题域是什么？</h2><p>从 <a href="https://en.wikipedia.org/wiki/Observability" rel="nofollow">wikipedia: Observability</a> 可理解到 <strong>可观测性</strong> 的定义：</p><blockquote>In control theory, observability is a measure of how well internal states of a system can be inferred from knowledge of its external outputs.</blockquote><p>Consider a physical system modeled in state-space representation. A system is said to be observable if, for any possible evolution of state and control vectors, the current state can be estimated using only the information from outputs (physically, this generally corresponds to information obtained by sensors). In other words, one can determine the behavior of the entire system from the system's outputs. On the other hand, if the system is not observable, there are state trajectories that are not distinguishable by only measuring the outputs.</p><p>简单表述为，可观测性是一种方法，通过系统的外部输出推导出系统内部的状态。</p><p>下图简化了系统的组成和系统间的交互：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762552" alt="2.png" title="2.png" referrerpolicy="no-referrer"></span></p><p>从上述交互图可了解到，系统的交互行为有如下几种形态：</p><ul><li><p>系统内部</p><ul><li>组件功能闭环，不与其他组件或系统交互</li><li>组件之间交互</li></ul></li><li><p>系统之间</p><ul><li>系统和系统之间进行交互</li></ul></li></ul><p>这样，若想通过系统的外部输出了解系统的状态，就需要两种形态的信息：</p><ul><li>组件闭环的信息</li><li>组件间或系统间流动的信息</li></ul><p>第一种形态通常可通过 logs 或 metrics 表征，第二种形态就需要 trace 表征，在流动的信息中增加标记。</p><p>对于 logs 和 metrics 的区别，可通过它们的操作方法进行理解。</p><p>再进一步抽象，可观测性涉及到如下问题：</p><ul><li>观测数据的数据模型</li><li>观测数据的采集</li><li>观测数据的处理</li><li>观测数据的导出</li><li>观测数据的使用</li><li>etc.</li></ul><p>上述即是 OpenTelemetry 面对的问题域及具体的问题，且将具体的问题限定在：</p><ul><li>观测数据的数据模型</li><li>观测数据的采集</li><li>观测数据的处理</li><li>观测数据的导出</li></ul><h2>OpenTelemetry 的解决方案是什么？</h2><p>OpenTelemetry 通过 Spec 规范了观测数据的数据模型以及采集、处理、导出方法，包括 trace、metrics、logs (未来不排除会有新的类型)，参见 <a href="https://github.com/open-telemetry/opentelemetry-specification" rel="nofollow">opentelemetry-specification</a>。</p><p>同时为了方便使用，通过 protobuf 来描述，参见<a href="https://github.com/open-telemetry/opentelemetry-proto" rel="nofollow">opentelemetry-proto</a>。</p><p>基于 Spec，OpenTelemetry 面向观测数据的生成和处理，做了如下的努力：</p><ul><li>为了方便开发者使用，提供了语言相关的 SDK，如 <a href="https://github.com/open-telemetry/opentelemetry-go" rel="nofollow">opentelemetry-go</a>、<a href="https://github.com/open-telemetry/opentelemetry-java" rel="nofollow">opentelemetry-java</a>、<a href="https://github.com/open-telemetry/opentelemetry-js" rel="nofollow">opentelemetry-js</a> 等，所有支持的开发语言可参见 <a href="https://opentelemetry.io/docs/" rel="nofollow">官方文档</a></li><li>为了方便可观测数据的采集、处理、导出，提供了通过配置管理的 Collector 服务，如对接开源项目的 <a href="https://github.com/open-telemetry/opentelemetry-collector" rel="nofollow">opentelemetry-collector</a>、对接第三方 vendor 的 <a href="https://github.com/open-telemetry/opentelemetry-collector-contrib" rel="nofollow">opentelemetry-collector-contrib</a></li></ul><p>通过下图可直观理解 OpenTelemetry 的组件和工作流：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762551" alt="3.png" title="3.png" referrerpolicy="no-referrer"></span></p><h2>OpenTelemetry 的历史是什么？</h2><p>从 <a href="https://www.cncf.io/blog/2019/05/21/a-brief-history-of-opentelemetry-so-far/" rel="nofollow">A brief history of OpenTelemetry (So Far)</a> 可简单了解到，OpenTelemetry 是由两个开源项目合并组成的：</p><ul><li><p><a href="https://opencensus.io/" rel="nofollow">OpenCensus</a></p><ul><li>面向 trace 和 metrics 进行数据模型标准化，并提供采集、处理、导出的工具</li></ul></li><li><p><a href="https://opentracing.io/" rel="nofollow">OpenTracing</a></p><ul><li>面向 trace 进行数据模型标准化，并提供采集、处理、导出的工具</li></ul></li></ul><p>2019 年 5 月，两个开源项目合并，官方宣布开源 OpenTelemetry 项目。</p><p>2021.02，trace spec 达到 1.0 版本，根据官方的成熟度模型 (<a href="https://github.com/open-telemetry/opentelemetry-proto#maturity-level" rel="nofollow">link</a>)，目前 trace 的 spec 已经达到 stable 级别，metrics 达到了 beta 级别，logs 当前还处在 alpha 级别：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762553" alt="4.png" title="4.png" referrerpolicy="no-referrer"></span></p><h2>OpenTelemetry 的前景如何？</h2><p>自 OpenTelemetry 推出以来，有越来越多的厂商开始关注和贡献。</p><p>从 <a href="https://github.com/open-telemetry/opentelemetry-collector-contrib" rel="nofollow">opentelemetry-collector-contrib</a> 可看出来，厂商的关注重点在于 exporter 部分，将观测数据便利导入到自身的服务中，其中已经包含阿里云自身的 <a href="https://www.aliyun.com/product/sls" rel="nofollow">SLS 产品</a>：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762556" alt="5.png" title="5.png" referrerpolicy="no-referrer"></span></p><p>对于 receiver 和 processor 环节，相信厂商也会逐步投入更多的精力，如：</p><ul><li>通过 receiver 和 exporter 的配合，形成观测数据的处理 workflow</li><li>通过 processor，在观测数据存储前进行规范化处理</li></ul><p>对于多云场景，OpenTelemetry 定义的观测数据模型、采集/处理/导出 标准，将有利于用户通过一套可观测性标准对接多种云厂商，避免 vendor 锁定。</p><p>即使是面向单一的云 (如云厂商内部的服务)，也不可避免会考虑未来进行开源、与外部共建等，使用社区的可观测性标准可以降低开源成本。同时，可观测性的理念、标准、技术在不断迭代，通过跟进社区，可以更好使用到社区带来的技术红利和影响力。</p><p>因此，无论是面对多云场景还是单一的云厂商，采用业界的可观测性标准都是很有必要。</p><h2>OpenTelemetry 如何使用？</h2><h3>核心概念</h3><p>OpenTelemetry 中的概念比较多，这里列举些常见的概念，方便进行理解：</p><ul><li><p>观测数据相关</p><ul><li><p>Signal</p><ul><li>观测数据类型，如 trace、metrics、logs</li></ul></li><li><p>Instrument</p><ul><li>可认为是某种 Signal 的实例</li></ul></li></ul></li><li><p>OpenTelemetry 自身项目相关</p><ul><li><p>API</p><ul><li>OpenTelemetry Spec 的形式化描述，如 <a href="https://github.com/open-telemetry/opentelemetry-proto" rel="nofollow">opentelemetry-proto</a></li></ul></li><li><p>SDK</p><ul><li>面向不同开发语言的 API 实现</li></ul></li><li><p>Contrib Packages</p><ul><li>与具体的开源项目或 vendor 产品相关的实现</li></ul></li></ul></li><li><p>使用的组件相关</p><ul><li><p>Components</p><ul><li><p>Receivers</p><ul><li>接收观测数据的组件</li></ul></li><li><p>Processors</p><ul><li>处理接收到的观测数据的组件</li></ul></li><li><p>Exporters</p><ul><li>将观测数据导出的组件，如导出到开源项目 Prometheus 或云厂商服务中</li></ul></li><li><p>Extensions</p><ul><li>不参与观测数据的处理，辅助相关处理组件的运行，如健康检测、服务发现等</li></ul></li><li><p>Services</p><ul><li>表征配置的哪些组件需要运行，如 receivers / processors / exporters / extentions</li></ul></li><li><p>Collector</p><ul><li>可认为是 receivers / processors / exporters / extentions / services 组成的整体</li></ul></li><li><p>Controller</p><ul><li>用于开发者开发的应用中，作用可等同于 receivers / processors / exporters 组成的整体</li></ul></li></ul></li></ul></li></ul><h3>golang demo</h3><p>笔者写了一个 golang demo，用来演示：</p><ul><li>APP 中如何生成 trace / metrics 数据</li><li>APP 中使用 stdout controller 来采集、处理、打印 trace / metrics 数据</li><li>APP 中通过 otlp controller 采集 trace / metrics 数据，并导出到外部运行的 collector 中</li><li>本地独立运行一个 collector 服务，接收 otlp controller 推送的 trace / metrics 数据，并将其导出到本地文件和阿里云 SLS 中</li></ul><p>demo 参见：<a href="https://github.com/flyer103/otel-demo" rel="nofollow">https://github.com/flyer103/otel-demo</a></p><p>具体的使用方法参见 demo 的 README.md，下述简单描述下思路。</p><p>cmd/app/server.go 文件描述了 OpenTelemetry 的使用逻辑，分成两部分：</p><ol><li>初始化并运行全局的 controller，用来在 APP 内部 receive / process / export 观测数据，或将 APP 内的观测数据推送到外部</li><li>APP 内按照业务需求生成 metrics 和 trace</li></ol><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762554" alt="6.png" title="6.png" referrerpolicy="no-referrer"></span></p><p>pkg/ 目录下分别封装了 controller 和 signal (trace / metrics)，具体的实现不再赘述：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762555" alt="7.png" title="7.png" referrerpolicy="no-referrer"></span></p><p>yaml/ 下提供了一个将观测数据导出到 SLS 的示例，包括了用于接收观测数据的 receiver (client 端可通过 grpc client 将数据推送到该 receiver)、用于观测数据转换处理的 processors、用于数据导出的 exporters、用于开启组件的 services：</p><p><span class="img-wrap"><img class="lazy" src="https://segmentfault.com/img/remote/1460000039762557" alt="8.png" title="8.png" referrerpolicy="no-referrer"></span></p><h2>畅想</h2><p>通过上述分析，大家对 OpenTelemetry 的概念、问题域、解决方案和使用方法会有一个直观的体会，通过上述 golang demo 可以快速上手。</p><p>对于开发者，基于 OpenTelemetry 可通过一套标准的方案进行 trace / metrics / logs 的生成和导出，降低开发过程中对不同类型观测数据的使用成本，也降低对接不同后端服务的成本，如开源项目 Prometheus 或第三方云厂商的服务。</p><p>对于 SRE，基于 OpenTelemetry 可为观测数据提供一套标准的采集、处理、导出流程，并在处理环节根据团队需求规范化观测数据，便于后续采用标准化的方案使用观测数据，如监控、告警服务。</p><p>同时，不论对于开发者还是 SRE，均可以通过社区的力量持续迭代对可观测性问题域的理解，吸收社区的技术红利，并将生产中产生的最佳实践回馈社区，更好推动可观测性领域的发展。</p><p><a
